Seite "HTTP-Transportkonfiguration"

Beim HTTP-Transport werden sowohl HTTP- als auch HTTPS-Endpunkte unterstützt.

Die Seite Business-Service erstellen/bearbeiten - HTTP-Transportkonfiguration ist eine von mehreren Seiten für das Erstellen und Bearbeiten von Business-Services. Weitere Informationen hierzu finden Sie unter Business-Services erstellen und konfigurieren und unter Business-Service-Konfigurationen bearbeiten.

Auf dieser Seite können Sie Transporteinstellungen für einen Business-Service mit dem HTTP-Transportprotokoll konfigurieren. In Tabelle 18-10 wird die Verwendung der Seite beschrieben.

Lese-Timeout
Geben Sie das Intervall für das Lese-Timeout in Sekunden ein.
Der Wert null (0) steht für kein Timeout.
Verbindungs-Timeout
Geben Sie das Intervall für das Verbindungs-Timeout in Sekunden ein. Wenn das Timeout abläuft, bevor die Verbindung hergestellt werden kann, wird in Oracle Service Bus ein Verbindungsfehler ausgegeben.
Der Wert null (0) steht für kein Timeout.
HTTP-Anforderungsmethode
Mit diesem Parameter können Sie eine der folgenden HTTP-Methoden in einer Anforderung verwenden:
  • POST - Überträgt sämtliche Daten von unbegrenzter Länge direkt über die Socket-Verbindung als Teil des HTTP-Anforderungstextes. Der Austausch ist für den Client unsichtbar, und der URL ändert sich nicht. Bei REST-basierten Anforderungen gibt POST dem Transport die Anweisung, einen Erstell-/Ersetzungsvorgang oder bei der Anforderung eine Aktion auszuführen.
  • GET - Innerhalb der Anforderung können Sie eigene Informationen angeben, die besser beschreiben, was abgerufen werden soll. Diese Informationen werden als Zeichenfolge übergeben, die in einer Abfragezeichenfolge an den Anforderungs-URL angehängt wird. Sie können die GET-Methode in einem Business-Service mit dem Service-Typ "Beliebiger XML-Service" oder mit dem Service-Typ "Messaging Service" verwenden, wenn der Anforderungsnachrichtentyp auf "Keine" gesetzt ist. Bei REST-basierten Anforderungen wird mit der GET-Methode eine Darstellung einer Remote-Ressource abgerufen.
  • PUT - Sie können die PUT-Methode bei einem Business-Service mit dem Service-Typ "Beliebiger XML-Service" oder "Messaging Service" verwenden. PUT gibt dem Transport die Anweisung, einen Erstell-/Ersetzungsvorgang bei einer REST-basierten Anforderung auszuführen, beispielsweise eine Datei an einen bekannten Speicherort hochzuladen.
  • HEAD - Sie können die HEAD-Methode bei einem Business-Service mit dem Service-Typ "Beliebiger XML-Service" oder mit dem Service-Typ "Messaging Service" verwenden, wenn der Antwortnachrichtentyp auf "Keine" gesetzt ist. HEAD gibt dem Transport die Anweisung, Header-Informationen für eine Remote-Ressource abzurufen, statt eine vollständige Darstellung der Ressource in einer REST-basierten Anforderung abzurufen.
  • DELETE - Sie können die PUT-Methode bei einem Business-Service mit dem Service-Typ "Beliebiger XML-Service" oder "Messaging Service" verwenden. DELETE gibt dem Transport die Anweisung, einen Löschvorgang bei einer REST-basierten Anforderung auszuführen.
Hinweis: Wenn in der Variablen "$outbound/transport/request/http:http-method" bereits eine Methode festgelegt ist, hat dieser Wert Vorrang vor den Methoden, die Sie als HTTP-Anforderungsmethode wählen.
Authentifizierung
Wählen Sie eine der folgenden Optionen:
  • Keine - Gibt an, dass für den Zugriff auf diesen Service keine Authentifizierung erforderlich ist.
  • Basis - Gibt an, dass für den Zugriff auf diesen Service die Basisauthentifizierung erforderlich ist.

    • Bei der Basisauthentifizierung erhält WebLogic Server die Anweisung, den Client mittels eines Benutzernamens und Kennwortes anhand der Authentication Provider zu authentifizieren, die in der Sicherheits-Realm konfiguriert sind, beispielsweise ein LDAP-Directory Service (Lightweight Directory Access Protocol) und Windows Active Directory. Der Client muss den zugehörigen Benutzernamen und das zugehörige Kennwort im HTTP-Anforderungs-Header senden.

    Von der Verwendung der Basisauthentifizierung wird bei HTTP dringend abgeraten, weil das Kennwort im Klartext übertragen wird. Das Übertragen von Kennwörtern über HTTPS ist jedoch sicher, weil HTTPS einen verschlüsselten Kanal bereitstellt.

    Warnung: Standardmäßig können alle Benutzer (autorisierte und anonyme Benutzer) auf einen Business-Service zugreifen. Wenn Sie die Benutzer eingrenzen möchten, die auf einen Business-Service zugreifen können, erstellen Sie eine Autorisierungs-Policy der Transportebene. Weitere Informationen finden Sie unter Zugriffs-Policys der Transportebene bearbeiten.

  • Clientzertifikat - Gibt verschlüsselte Kommunikation und starke Clientauthentifizierung (wechselseitige SSL) an. Weitere Informationen finden Sie unter "Configuring Transport-Level Security" im Oracle Fusion Middleware Developer's Guide for Oracle Service Bus unter http://www.oracle.com/pls/as111170/lookup?id=OSBDV1557.
Service-Account
Geben Sie einen Service-Account ein. Bei einem Service-Account handelt es sich um eine Aliasressource für einen Benutzernamen und ein Kennwort. Dieses Feld ist obligatorisch, wenn Sie das Feld Basisauthentifizierung erforderlich aktiviert haben.

Dispatch Policy
Wählen Sie die Instanz von WebLogic Server Work Manager, die Sie für die Dispatch Policy für diesen Endpunkt verwenden möchten. Wenn kein anderer Work Manager vorhanden ist, wird der Standard-Work Manager verwendet. Weitere Informationen zu Work Managern finden Sie unter "Using Work Managers to Optimize Scheduled Work" in Oracle Fusion Middleware Configuring Server Environments for Oracle WebLogic Server unter http://www.oracle.com/pls/as111170/lookup?id=CNFGD112.
Anforderungscodierung
Übernehmen Sie den Standardwert iso-8859-1 als Zeichensatzcodierung für Anforderungen in HTTP-Transporten, oder geben Sie eine andere Zeichensatzcodierung ein.
Antwortcodierung
Übernehmen Sie den Standardwert iso-8859-1 als Zeichensatzcodierung für Antworten in HTTP-Transporten, oder geben Sie eine andere Zeichensatzcodierung ein.
Proxyserver
Geben Sie eine Proxyserverressource ein, oder klicken Sie auf "Durchsuchen", um einen Eintrag aus der Liste der konfigurierten Proxyserverressourcen auszuwählen.
HTTP-Umleitungen folgen
Aktivieren Sie dieses Kontrollkästchen, um anzugeben, dass HTTP-Umleitungen (das heißt Anforderungen mit dem Antwortcode 3xx) automatisch gefolgt werden soll. Eine Umleitung tritt auf, wenn Sie eine ausgehende Anforderung an den URL eines Business-Service senden und dieser Service einen Antwortcode (zum Beispiel 302) mit der Bedeutung zurückgibt, dass der URL nicht mehr gültig ist, und diese Anforderung an einen anderen URL gesendet werden muss.
Wenn Sie das Kontrollkästchen HTTP-Umleitungen folgen aktivieren, wird die Anforderung von Oracle Service Bus ohne Ihr Dazutun automatisch erneut an den neuen URL gesendet. Deaktivieren Sie dieses Kontrollkästchen, wenn Sie den HTTP-Umleitungen nicht automatisch folgen möchten.
Chunk-Streaming-Modus verwenden
Wählen Sie diese Option, wenn Sie zum Senden von Nachrichten die HTTP-Chunk-Übertragungscodierung verwenden möchten.
Hinweis: Verwenden Sie kein Chunk Streaming, wenn Sie die Option "HTTP-Umleitungen folgen" aktiviert haben. Im Chunk-Modus können Umleitung und Authentifizierung nicht automatisch verarbeitet werden.
Sessionbindung
Wählen Sie diese Option, wenn Sie Sessionbindung (auch Sessionaffinität genannt) bei HTTP-Anforderungen für den Business-Service verwenden möchten.
Sticky SessionID-Name
Falls Session-Stickiness aktiviert ist, geben Sie einen eindeutigen Namen zur Identifizierung der Session ein.
