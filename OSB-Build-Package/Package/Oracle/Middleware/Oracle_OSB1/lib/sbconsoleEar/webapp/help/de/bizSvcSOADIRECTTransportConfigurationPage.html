Seite "SOA-DIRECT-Transportkonfiguration"

In Tabelle 18-17 sind die transportspezifischen Optionen für den SOA-DIRECT-Transport beschrieben.

Weitere Informationen zum SOA-DIRECT-Transport finden Sie unter "SOA-DIRECT Transport" im "Oracle Fusion Middleware Developer's Guide for Oracle Service Bus" unter http://www.oracle.com/pls/as111170/lookup?id=OSBDV909.

Tabelle 18-17 Business-Services erstellen/bearbeiten – SOA-DIRECT-Transportkonfiguration
Eigenschaft
Beschreibung
JNDI-Service-Account
Optional. Diese Option gibt die Sicherheitszugangsdaten für die JNDI-Suche im Ziel-SOA-Service an. Der Service-Account muss statisch sein. Klicken Sie auf Durchsuchen, und wählen Sie einen Service-Account aus. Wenn Sie keinen Service-Account angeben, wird ein anonymes Subject verwendet.
Rolle
Erforderlich. Diese Option gibt das vom Service verwendete Kommunikationsmuster an. Wählen Sie eine der folgenden Optionen aus:
  • Synchroner Client (Standard): In dieser Rolle ist die Option "Callback Proxy" deaktiviert, da es keinen asynchronen Callback gibt. Die Option "WS-Addressing-Version" ist ebenfalls aktiviert.
  • Asynchroner Client: In dieser Rolle ist in der Regel ein asynchroner Callback erforderlich, und Sie können einen Callback Proxy angeben und müssen eine WS-Addressing-Version wählen. Diese Option ist nur dann aktiviert, wenn der WSDL-Service vom Typ SOAP ist.
  • Service Callback: Diese Rolle dient der Rückgabe des asynchronen Callbacks an einen SOA-Service nach einem externen Service-Aufruf.

    • Lastverteilung oder Failover sind für diese Services in der Rolle "Service Callback" nicht verfügbar.

Callback Proxy
Optional. Diese Option ist nur für die Rolle "Asynchroner Client" aktiviert.
Mit dieser Option können Sie den Proxy-Service angeben, der Callbacks empfängt. Wenn Sie einen Callback Proxy wählen und von der Anforderung oder der Proxy-Service Pipeline kein WS-Addressing angegeben wird, wird die ReplyTo-Eigenschaft im SOAP-Header von Oracle Service Bus automatisch ausgefüllt. Sie müssen einen WSDL-Proxy-Service wählen, der den SB-Transport (für RMI) verwendet, und der Callback Proxy-Service muss WS-Addressing verstehen.
WS-Addressing-Eigenschaften, die in der Anforderung gesendet werden oder in der Proxy-Service Pipeline festgelegt sind, werden anstelle des in dieser Option festgelegten Callback Proxys verwendet.
Wenn Sie kein Callback Proxy angeben und die Anforderung keine ReplyTo-Eigenschaften enthält, müssen Sie ReplyTo-Eigenschaften im SOAP-Header über die Proxy-Service Pipeline angeben.
Fehler-Proxy
Diese Option wird momentan nicht unterstützt. Sie müssen die Callback Services konfigurieren, um Fehler in einem asynchronen Schema zu verarbeiten.
WS-Addressing-Version
Erforderlich. Diese Option ist nur für die Rolle "Asynchroner Client" aktiviert.
Geben Sie die WS-Addressing-Version an, die standardmäßig verwendet werden soll, wenn kein WS-Addressing in der Anforderung oder der Proxy-Service Pipeline angegeben wird. WS-Addressing-Eigenschaften, die in der Anforderung gesendet werden oder in der Proxy-Service Pipeline festgelegt sind, werden anstelle der in dieser Option festgelegten WS-Addressing-Version verwendet.
Wenn Konflikte hinsichtlich der WS-Addressing-Version zwischen Umgebungen bestehen, führen Sie alle notwendigen Transformationen in der Proxy-Service Pipeline durch. Weitere Informationen finden Sie unter "Transformation Examples" im Oracle Fusion Middleware Developer's Guide for Oracle Service Bus unter http://www.oracle.com/pls/as111170/lookup?id=OSBDV1827.
Dispatch Policy
Wählen Sie die Instanz des WebLogic Server Work Managers, die Sie für die Dispatch Policy dieses Endpunkts verwenden möchten. Der Standard-Work Manager wird verwendet, wenn kein anderer Work Manager vorhanden ist.
Weitere Informationen zu Work Manager finden Sie unter "Using Work Managers to Optimize Scheduled Work" in Oracle Fusion Middleware Configuring Server Environments for Oracle WebLogic Server unter http://www.oracle.com/pls/as111170/lookup?id=CNFGD112.
Subject des Callers übergeben
Optional. Aktivieren Sie diese Option, damit Oracle Service Bus das authentifizierte Subject vom Proxy-Service beim Aufruf des SOA-Service übergibt. Die Option "Aufruf-Service-Account" ist eine Alternative zur Option "Subject des Callers übergeben" und wird deaktiviert, wenn Sie diese Option aktivieren.
Hinweis: Stellen Sie sicher, dass zwischen Client- und Zielserver das Domainvertrauen aktiviert ist, wenn sie sich in verschiedenen Domains befinden. Weitere Informationen finden Sie unter "Important Information Regarding Cross-Domain Security Support" in "Oracle Fusion Middleware Securing Oracle WebLogic Server" unter http://www.oracle.com/pls/as111170/lookup?id=SECMG403.
Aufruf-Service-Account
Optional. Mit dieser Alternative zur Option "Subject des Callers übergeben" können Sie benutzerdefinierte Sicherheitszugangsdaten angeben, indem Sie einen Service-Account für den RMI-Aufruf wählen. Sie können einen beliebigen Service-Account-Typ angeben (Passthrough, Statisch, Zuordnung).
Klicken Sie auf Durchsuchen, und wählen Sie einen Service-Account. Wenn Sie keinen Service-Account angeben, wird ein anonymes Subject verwendet.
