Seite "Transportkonfiguration"

Die Seite Business-Service erstellen/bearbeiten – Transportkonfiguration ist eine von diversen Seiten, auf denen Sie Business-Services erstellen und bearbeiten können. Diese Funktionen sind unter Business-Services erstellen und konfigurieren und Business-Service-Konfigurationen bearbeiten genauer beschrieben.

Auf dieser Seite können Sie ein Transportprotokoll für den Business-Service wählen und andere allgemeine Einstellungen für die Transportkonfiguration festlegen.

Sicherheit auf ausgehender Transportebene gilt für die Verbindungen zwischen Proxy-Services und Business-Services in Oracle Service Bus. Weitere Informationen zu Sicherheit auf Transportebene finden Sie unter "Configuring Transport-Level Security" im Oracle Fusion Middleware Developer's Guide for Oracle Service Bus unter http://www.oracle.com/pls/as111170/lookup?id=OSBDV1557.

Protokoll
Wählen Sie aus der Liste ein Transportprotokoll. Die verfügbaren Protokolle hängen vom erstellten Service-Typ ab:
  • WSDL-Web Service: soa-direct, bpel-10g, dsp, http, jca, jms, sb, ws
  • Transporttyp-Service: ejb, flow, jejb
  • Messaging Service: email, file, ftp, http, jms, mq (sofern verfügbar), sftp, tuxedo
  • Beliebiger SOAP-Service: dsp, http, jms, sb
  • Beliebiger XML-Service: dsp, email, file, ftp, http, jms, mq (sofern verfügbar), sb, sftp, tuxedo
Load Balancing-Algorithmus
Wählen Sie einen der folgenden Load Balancing-Algorithmen:
  • Round-Robin: Mit diesem Algorithmus werden die URLs, die Sie in das Feld Endpunkt-URI für diesen Business-Service eingeben, dynamisch sortiert. Wenn der erste URL einen Fehler meldet, wird der nächste verwendet usw., bis die maximale Anzahl der Wiederholungen erreicht wurde.

    • Für jede neue Nachricht gibt es eine neue URL-Reihenfolge.

  • Zufällig: Mit diesem Algorithmus wird die Liste mit den URLs, die Sie in das Feld Endpunkt-URI für diesen Business-Service eingeben, nach dem Zufallsprinzip sortiert. Wenn der erste URL einen Fehler meldet, wird der nächste verwendet usw., bis die maximale Anzahl der Wiederholungen erreicht wurde.
  • Zufallsgewichtet: Mit diesem Algorithmus wird die Liste mit URLs, die Sie in das Feld Endpunkt-URI für diesen Business-Service eingeben, nach dem Zufallsprinzip sortiert. Bei einigen URLs wird jedoch häufiger ein Wiederholungsversuch gestartet als bei anderen. Die Anzahl der Wiederholungsversuche hängt von dem Wert ab, den Sie in das Feld Gewichtung eingeben.
  • Keine: Mit diesem Algorithmus wird die Liste mit URLs, die Sie für diesen Business-Service in das Feld Endpunkt-URI eingeben, von oben nach unten sortiert.
Endpunkt-URI
Geben Sie einen Endpunkt-URL in dem Format ein, das dem im Feld Protokoll gewählten Transportprotokoll entspricht. Im Folgenden finden Sie Beschreibungen der URI-Formate für alle Transporte. Optionale URI-Elemente stehen in eckigen Klammern. Weitere Informationen zu Transport-URIs finden Sie in den entsprechenden Kapiteln zum jeweiligen Transport im Oracle Fusion Middleware Developer's Guide for Oracle Service Bus unter http://www.oracle.com/pls/as111170/lookup?id=OSBDV.
  • bpel-10g - protocol://host[:port] [/protocol-path]/domain/process[/version[/partnerlink/role]]
  • dsp - t3://dsp-ip-address:port/dsp-app-name
  • ejb - ejb:jndi_provider_name:ejb_jndi_name

    • Im URI steht jndi_provider_name für den Namen der JNDI-Providerressource und ejb_jndi_name für den JNDI-Namen im JNDI-Server für das EJB.

    Wenn sich der JNDI-Provider auf demselben Server befindet, muss er nicht angegeben werden. Der URI lautet dann ejb::ejb_jndi_name.

    Hinweis: Wenn die EJBs auf IBM WebSphere ausgeführt werden, muss ejb_jndi_name folgendes Format haben:

    cell/nodes/node_name/servers/server_name/ejb_jndi_name

    or

    cell/clusters/cluster_name/ejb_jndi_name

    Beschreibungen dazu finden Sie in der IBM WebSphere-Dokumentation unter http://publib.boulder.ibm.com/infocenter/wasinfo/v6r1/index.jsp?topic=/com.ibm.websphere.nd.iseries.doc/info/iseriesnd/ae/rnam_example_prop3.html.

  • E-Mail: Sie können mehrere Endpunkt-URIs in beliebigen Kombinationen der folgenden Formate eingeben:
    • mailto:name@domain_name.com
    • mailto:name@domain_name.com?smtp=smtp_server_resource
    • mailto:name@domain_name.com?mailsession=jndi_mail_session
  • file - file:///drivename:/somename
  • flow - flow:<reference path to split-join resource>

    • Der reference path to split-join resource ist der Pfad (Projekt/Ordner) und Name des Split Joins, z.B. batchorderProcessing/batchorder.

  • ftp - ftp://host:port/directory
  • http - http://host:port/someService

    • Beim HTTP-Transport werden sowohl HTTP- als auch HTTPS-Endpunkte unterstützt.

  • jca - jca://<resource_adapter_jndi>
  • jejb - jejb:jndi_provider_name:ejb_jndi_name

    • Beispiel: EJB 2.1: jejb:myProvider:osb.jejb.myJejbBiz21

    Beispiel: EJB 3.0: jejb:myProvider:myBiz31#osb.jejb.myJejbBiz

    Hinweis: Wenn die EJBs auf IBM WebSphere ausgeführt werden, muss ejb_jndi_name folgendes Format haben:

    cell/nodes/node_name/servers/server_name/ejb_jndi_name

    or

    cell/clusters/cluster_name/ejb_jndi_name

    Beschreibungen dazu finden Sie in der IBM WebSphere-Dokumentation unter http://publib.boulder.ibm.com/infocenter/wasinfo/v6r1/index.jsp?topic=/com.ibm.websphere.nd.iseries.doc/info/iseriesnd/ae/rnam_example_prop3.html.

  • jms - jms://host:port/factoryJndiName/destJndiName

    • Wenn Sie ein JMS-Ziel für mehrere Server definieren möchten, verwenden Sie das folgende URI-Format:
    jms://host1:port,host2:port/QueueConnectionFactory/DestName

    Hinweis: Während bei Oracle WebLogic Server Vorwärtsschrägstriche in JNDI-Namen (Beispiel: "myqueues/myqueue") zulässig sind, stören JNDI-Namen mit Vorwärtsschrägstrichen das URI-Format, das für Oracle Service Bus erforderlich ist, und Sie können diese Namen nicht verwenden. Um dieses Problem zu umgehen, definieren Sie einen JMS-Fremdserver, und referenzieren Sie diesen Fremdserver im URI. Weitere Informationen finden Sie unter "Configure foreign servers" in der Oracle Fusion Middleware Oracle WebLogic Server Administration Console Online Help unter http://www.oracle.com/pls/as111170/lookup?id=WLACH.

    In einem Cluster: Die Hostnamen im JMS-URI müssen genau mit den Hostnamen der Cluster Server übereinstimmen, da sie in WebLogic Server konfiguriert werden.

Endpunkt-URI (Fortsetzung)
  • mq - mq://<local-queue-name>?conn=<mq-connection-resource-ref>

    • local-queue-name ist der Name der MQ Queue, von der der Business-Service Nachrichten abruft.

    mq-connection-resource-ref ist der Pfad (Projekt/Ordner) und Name der MQ-Verbindungsressource, z.B. default/my_MQconnection.

    Hinweis: Der Endpunkt-URI darf keine Leerzeichen enthalten, Sie dürfen also keine MQ-Verbindungsressourcen oder Projekte/Ordner mit Leerstellen im Namen erstellen.

    Informationen dazu, wie Sie einen MQ-Transport in Oracle Service Bus verfügbar machen, finden Sie unter MQ-Verbindungen.

  • sb - sb://[<jndi_provider_name/>]service_name

    • jndi_provider_name (optional) ist der Name der Oracle Service Bus-JNDI-Providerressource. Wenn Sie diesen Wert weglassen, wird der Standardkontext verwendet.

    service_name ist der Ziel-Service und entspricht der Remote Proxy-Service-URI.

  • sftp - sftp://hostname:port/directory
  • soa-direct - [protocol://authority]/default/compositeName[!versionNumber[*label]]/serviceName

    • "Protocol" und "authority" sind optional, wenn sich die SOA-Services auf demselben Server befinden wie Oracle Service Bus.

  • tuxedo - tuxedo:resourcename[/remotename]

    • Im URI ist resourcename ein WTC-Importnamen und remotename der Service-Name, der von der Remote-Tuxedo-Domain exportiert wurde. Der resourcename des URI ist erforderlich, der remotename ist optional.

    Wenn mehr als ein URI angegeben wird, müssen die Endpunkte eindeutige Ressourcennamen besitzen. Wenn kein Remote-Name angegeben wird, entspricht dessen Wert dem Wert des Ressourcennamens. Wenn kein Remote-Name eingegeben wird oder Remote-Name und Ressourcenname identisch sind, ist nur ein URI zulässig. In diesem Fall haben Ressourcenname und Remote-Name denselben Wert. Dadurch können bereits definierte WTC-Importe WTC-Lastverteilung und -Failover nutzen.

  • ws - http://host:port/somename oder https://host:port/somename
Klicken Sie auf Hinzufügen, um einen oder mehrere zusätzliche URIs hinzuzufügen. Zur Laufzeit werden die URLs basierend auf dem Load Balancing-Algorithmus gewählt, den Sie im Feld Load Balancing-Algorithmus festgelegt haben.
Wenn Sie im Feld Load Balancing-Algorithmus den Wert Zufallsgewichtet gewählt haben, können Sie in das Feld Endpunkt-URI auch eine Gewichtung eingeben. Der Standardwert ist 1.
Wenn Sie mehrere Endpunkte definiert und im Feld Load Balancing-Algorithmus den Wert Keine gewählt haben, ist die Reihenfolge der Endpunkte wichtig. Sie können die Endpunkte in der Spalte Optionen mit den Pfeilen Nach unten und Nach oben neu anordnen.
Oracle Service Bus unterstützt keine doppelten Endpunkt-URIs innerhalb desselben Business-Service.
Anzahl Wiederholungen
Geben Sie für den Fall eines Zustellungsfehlers beim Senden von ausgehenden Anforderungen die Anzahl der Wiederholungsversuche für einzelne URL-Endpunkte an (d.h. die Anzahl der Failover-Versuche).
Beispiel 1: Ein Business-Service besitzt einen konfigurierten URI (U1), und die Anzahl der Wiederholungsversuche wurde auf 3 gesetzt. Wenn U1 beim ersten Versuch einen Fehler meldet, verwendet das System den Endpunkt U1 drei weitere Male.
Ein Business-Service besitzt zwei konfigurierte URIs (U1 und U2), und die Anzahl der Wiederholungsversuche wurde auf 3 gesetzt. Wenn der erste Versuch (z.B. mit U1) nicht erfolgreich ist, verwendet das System den nächsten URI (U2). Ist dieser Versuch ebenfalls nicht erfolgreich, startet das System zwei weitere Versuche, einmal mit U1 und einmal mit U2.
Zeitintervall für Wiederholungen
Geben Sie in Sekunden an, wie lange das System anhält, bevor es alle Endpunkt-URIs in der Liste wieder verwendet.
Beispiel: Ein Business-Service besitzt zwei konfigurierte URIs (U1 und U2), die Anzahl der Wiederholungen wurde auf 2 gesetzt, und für das Zeitintervall wurden 5 Sekunden festgelegt. Wenn der erste Versuch (mit U1) nicht erfolgreich ist, verwendet das System sofort U2. Wenn U2 ebenfalls nicht erfolgreich ist, wartet das System 5 Sekunden lang, bis es erneut U1 verwendet.
Anwendungsfehler wiederholen
Wählen Sie Ja oder Nein.
Legen Sie für den Fall eines Zustellungsfehlers beim Senden von ausgehenden Anforderungen fest, ob mit einem Endpunkt-URI bei Anwendungsfehlern (z.B. einem SOAP-Fehler) einer neuer Versuch gestartet werden soll oder nicht.
Dieses Feld ist nur für folgende Transporte verfügbar: HTTP, EJB, JMS, DSP, Tuxedo, SB und WS.
