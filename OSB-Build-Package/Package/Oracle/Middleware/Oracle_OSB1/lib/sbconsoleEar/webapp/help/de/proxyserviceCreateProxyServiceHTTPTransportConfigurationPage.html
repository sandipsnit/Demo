Seite "HTTP-Transportkonfiguration"

Der HTTP-Transport unterstützt jetzt sowohl HTTP als auch HTTPS-Endpunkte.

Die Seite Proxy-Service erstellen/bearbeiten - HTTP-Transportkonfiguration gehört zu einer Reihe von Seiten zum Erstellen und Bearbeiten von Proxy-Services. Diese Aktionen sind in Proxy-Services erstellen und konfigurieren und in Proxy-Service-Konfigurationen bearbeiten beschrieben.

Sie verwenden diese Seite, um Transporteinstellungen für einen Proxy-Service mithilfe des HTTP-Transportprotokolls zu konfigurieren. Tabelle 19-7 liefert eine Beschreibung zur Verwendung dieser Seite.

Tabelle 19-7 Seite "Proxy-Service erstellen und bearbeiten - HTTP-Transportkonfiguration" 
Option
Verwendung
HTTPS erforderlich
Aktivieren Sie dieses Kontrollkästchen für eingehende HTTPS-Endpunkte.
Weitere Informationen finden Sie unter "Configuring Transport-Level Security" im Oracle Fusion Middleware Developer's Guide for Oracle Service Bus unter http://www.oracle.com/pls/as111170/lookup?id=OSBDV1557.
Authentifizierung
Wählen Sie eine der folgenden Optionen:
  • Keine - Gibt an, dass keine Authentifizierung erforderlich ist.
  • Basis - Gibt an, dass für den Zugriff auf diesen Service die Basisauthentifizierung erforderlich ist.

    • Bei der Basisauthentifizierung erhält WebLogic Server die Anweisung, den Client mittels eines Benutzernamens und Kennwortes anhand der Authentication Provider zu authentifizieren, die in der Sicherheits-Realm konfiguriert sind, beispielsweise ein LDAP-Directory Service (Lightweight Directory Access Protocol) und Windows Active Directory. Der Client muss den zugehörigen Benutzernamen und das zugehörige Kennwort im HTTP-Anforderungs-Header senden.

    Von der Verwendung der Basisauthentifizierung wird bei HTTP dringend abgeraten, weil das Kennwort im Klartext übertragen wird. Das Übertragen von Kennwörtern über HTTPS ist jedoch sicher, weil HTTPS einen verschlüsselten Kanal bereitstellt.

    Warnung: Standardmäßig können alle Benutzer (gleich ob autorisiert oder anonym) auf einen Proxy-Service zugreifen. Um die Zahl derjenigen Benutzer einzuschränken, die Zugriff auf einen Proxy-Service haben, erstellen Sie eine Autorisierungs-Policy auf Transportebene. Siehe Zugriffs-Policys auf Transportebene bearbeiten.

  • Clientzertifikat - Gibt verschlüsselte Kommunikation und starke Clientauthentifizierung (wechselseitige SSL) an. Weitere Informationen finden Sie unter "Configuring Transport-Level Security" im Oracle Fusion Middleware Developer's Guide for Oracle Service Bus unter http://www.oracle.com/pls/as111170/lookup?id=OSBDV1557.
  • Benutzerdefinierte Authentifizierung - Legt fest, dass ein Authentifizierungstoken in einem HTTP-Header enthalten ist. Die Identität des Clients wird durch Verwendung dieses vom Client gelieferten Tokens gesichert. Sie müssen einen Identity Assertion Provider erstellen, der das Token einem Oracle Service Bus-Benutzer zuordnet.

    • Das benutzerdefinierte Authentifizierungstoken kann jeder aktive Tokentyp sein, der von einem konfigurierten WebLogic Server Identity Assertion Provider unterstützt wird.

Dispatch Policy
Wählen Sie eine Dispatch Policy für diesen Endpunkt. Treffen Sie keine Wahl, wird die standardmäßige Dispatch Policy verwendet.
Die Dispatch Policy bezieht sich auf die WLS Work Manager-Instanz, die Sie für den Service-Endpunkt verwenden möchten. Weitere Informationen zu Work Manager finden Sie unter "Using Work Managers to Optimize Scheduled Work" im Oracle Fusion Middleware Configuring Server Environments for Oracle WebLogic Server auf der Website http://www.oracle.com/pls/as111170/lookup?id=CNFGD112.
Anforderungscodierung
  • Für eingehende HTTP-Transporte:

    • Wenn der Parameter für die Zeichensatzcodierung des Headers Content-Type nicht in der Clientanforderung angegeben wurde, geben Sie einen entsprechenden Parameter ein. Falls Sie keinen Wert eingeben, übernimmt das Feld den Standardwert ISO-8859-1.

  • Für ausgehende HTTP-Transporte:

    • Falls Sie keine Anforderungscodierung konfiguriert haben, entscheidet die Oracle Service Bus-Laufzeit während ihrer Anforderung an den Business-Service, welche die geeignetste Codierung ist. In einem Nicht-Passthrough-Szenario heißt die standardmäßige Zeichencodierung UTF-8 zur Laufzeit. Bei einem Passthrough-Szenario hingegen reicht die Laufzeit die mit der ausgehenden Antwort erhaltene Codierung weiter.

Antwortcodierung
Akzeptieren Sie den Standard ISO-8859-1 als Zeichensatzcodierung von Antworten in HTTP-Transporten, oder geben Sie eine andere Zeichensatzcodierung ein.
Authentifizierungs-Header
Geben Sie den HTTP-Header ein (jeden beliebigen mit Ausnahme von Autorisierung), aus dem Oracle Service Bus das Token extrahieren soll. Das Feld ist nur verfügbar, wenn Sie das Kontrollkästchen Benutzerdefinierte Authentifizierung aktiviert haben.
Beispiel: client-xyz-token.
Typ des Authentifizierungstokens
Wählen Sie ein Authentifizierungstokentyp. Es sind nur die aktiven Tokentypen verfügbar, die für einen Identity Assertion Provider konfiguriert wurden. (Weitere Informationen finden Sie unter Identity Assertion Provider konfigurieren für benutzerdefinierte Tokens.) Dieses Feld ist nur verfügbar, wenn Sie das Kontrollkästchen Benutzerdefinierte Authentifizierung aktiviert haben.

Wenn Sie fertig sind

Klicken Sie auf Weiter, um mit der Konfiguration dieses Service auf der nächsten Seite fortzufahren. Oder klicken Sie auf Letzte, um diese Konfiguration auf der Seite Zusammenfassung unter Proxy-Service erstellen/bearbeiten zu prüfen und zu speichern.
