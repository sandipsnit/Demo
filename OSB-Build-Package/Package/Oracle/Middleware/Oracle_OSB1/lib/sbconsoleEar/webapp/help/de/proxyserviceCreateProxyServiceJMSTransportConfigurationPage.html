Seite "JMS-Transportkonfiguration"

Die Seite Proxy-Service erstellen/bearbeiten - JMS-Transportkonfiguration ist eine von einer Serie zum Erstellen und Bearbeiten von Proxy-Services, wie in Proxy-Services erstellen und konfigurieren und Proxy-Service-Konfigurationen bearbeiten beschrieben.

Weitere Informationen finden Sie im Oracle Fusion Middleware Developer's Guide for Oracle Service Bus (http://www.oracle.com/pls/as111170/lookup?id=OSBDV1037) unter "JMS Transport".

Verwenden Sie diese Seite, um Transporteinstellungen für einen Proxy-Service unter Verwendung des JMS-Transportprotokolls zu konfigurieren. Tabelle 19-10 beschreibt, wie die Seite zu verwenden ist.

Tabelle 19-10 Einen Proxy-Service erstellen/bearbeiten - Seite "JMS-Transportkonfiguration"
Option
Verwendung
Zieltyp
Wählen Sie eine der folgenden Optionen:
  • Queue (für einen Punkt-zu-Punkt Zieltyp)
  • Topic (für ein Publishing-/Abonnement-Ziel)
Ist Antwort erforderlich
Die Option ist nur verfügbar, wenn Queue als Zieltyp gewählt wurde.
Wählen Sie diese Option, um anzugeben, dass eine Antwort erwartet wird, wenn eine ausgehende Nachricht gesendet wurde.
Antwortmuster
Diese Option ist nur verfügbar, wenn das Kontrollkästchen "Ist Antwort erforderlich" aktiviert wurde.
Wählen Sie eine der folgenden Optionen:
  • Wählen Sie JMSMessageID für JAX-RPC-Services, die auf Oracle WebLogic Server ausgeführt werden.
  • Wählen Sie JMSCorrelationID für alle anderen Services. Wenn Sie diese Option wählen, müssen Sie auch einen Antwort-URI eingeben.
Antwortnachrichtentyp
Diese Option ist nur verfügbar, wenn das Kontrollkästchen Ist Antwort erforderlich aktiviert wurde.
Wählen Sie eine der folgenden Optionen:
  • Byte (für einen Stream nicht interpretierter Byte)
  • Text (für Textnachrichten)
Diese Option ist deaktiviert, wenn Sie einen Java-Nachrichtentyp für die Antwort auswählen.
Client-JAR-Archiv
Diese Option ist verfügbar, wenn der Service ein Messaging-Service mit Java-Anforderungstyp ist. Wählen Sie die Client-JAR-Datei, die für das Entfernen von Nachrichten mit Java-Objekten aus der Queue verwendet werden soll. Durch die Auswahl der Client-JAR-Datei wird sichergestellt, dass diese sich im Classpath befindet.

Weitere Informationen finden Sie unter "Sending and Receiving Java Objects in Messages" im Oracle Fusion Middleware Developer's Guide for Oracle Service Bus unter http://www.oracle.com/pls/as111170/lookup?id=OSBDV1874.
Dispatch Policy
Wählen Sie diejenige Instanz von Oracle WebLogic Server Work Manager, die Sie für die Dispatch Policy für diesen Endpunkt verwenden möchten. Falls kein anderer Work Manager vorhanden ist, wird der standardmäßige Work Manager verwendet.
Beispiel: Wenn der Business-Service über ein JMS-Transportprotokoll verfügt, ist der Business-Service-Endpunkt eine MDB (Message-driven BeanRAWJAR-Datei , die Sie mit der Dispatch Policy verknüpfen können.
Informationen über Work Manager finden Sie unter "Work Manager für das Optimieren geplanter Arbeit verwenden" in Oracle Fusion Middleware Configuring Server Environments for Oracle WebLogic Server unter http://www.oracle.com/pls/as111170/lookup?id=CNFGD112.
Anforderungscodierung
Geben Sie den Zeichensatz für die Codierung von Anforderungen ein. Standard ist UTF-8.
Antwortcodierung
Diese Option ist nur verfügbar, wenn das Kontrollkästchen "Ist Antwort erforderlich" aktiviert wurde.
Geben Sie den Zeichensatz für die Codierung von Antworten ein. Standard ist UTF-8.
Timeout für Clientantwort
Diese Option ist nur verfügbar, wenn das Kontrollkästchen "Ist Antwort erforderlich" aktiviert wurde.
Geben Sie die Sekunden an, wie lange ein Server warten soll, bevor die Verbindung unterbrochen wird. Dies trifft nur zu, wenn der Client ein anderer Proxy-Service in der gleichen Domain ist.
Antwort-URI
Diese Option ist nur verfügbar, wenn JMSCorrelationID für das Antwortkorrelationsmuster gewählt wurde.
Geben Sie einen Antwort-URI im folgenden Format ein:
jms://host:port/connection_factory/jndi_destination
Verwenden Sie das folgende Format, wenn Sie mehrere Server als Ziel definieren möchten:
jms://host1:port,host2:port/connection_factory/jndi_destination
Sie können den Host und den Port im Antwort-URI auch auslassen. Beispiel:
jms:///connection_factory/jndi_destination

Wenn Sie Host und Port auslassen, wird die Connection Factory/das Ziel-Lookup auf dem lokalen Server angezeigt. Dies ist nützlich, falls der Anforderungs-URI zu einer fremden Connection Factory bzw. einem fremden Ziel-Lookup führt, Sie aber die Antwort an den lokalen Server senden möchten.

Hinweis: Während bei Oracle WebLogic Server Vorwärtsschrägstriche in JNDI-Namen (Beispiel: "myqueues/myqueue") zulässig sind, stören JNDI-Namen mit Vorwärtsschrägstrichen das URI-Format, das für Oracle Service Bus erforderlich ist, und Sie können diese Namen nicht verwenden. Um dieses Problem zu umgehen, definieren Sie einen JMS-Fremdserver, und referenzieren Sie diesen Fremdserver im URI. Weitere Informationen finden Sie unter "Configure foreign servers" in der Oracle Fusion Middleware Oracle WebLogic Server Administration Console Online Help unter http://www.oracle.com/pls/as111170/lookup?id=WLACH.

Antwort-Connection Factory
Die Option ist nur verfügbar, wenn JMSMessageID für das Antwortkorrelationsmuster gewählt wurde.
Geben Sie einen Antwort-Connection-Factory-URI ein.
Wurde keine Connection Factory angegeben, wird die Connection Factory für die Anforderung für die Antwort verwendet.
JMS-Service-Account
Wählen Sie einen Service-Account, der für die vom JMS-Server verwaltete JMS-Ressource verwendet werden soll. Ein Service-Account ist eine Aliasressource für Benutzer-ID und Kennwort, welche für Anforderung und Antwort benötigt werden. Derselbe Service-Account wird für JMS- und JNDI-Zwecke verwendet.
Weitere Informationen finden Sie unter Service-Accounts.
SSL verwenden
Wählen Sie diese Option nur, wenn die Anforderungen über eine TLS-/SSL-Verbindung gestellt werden.
SSL (Secure Sockets Layer) stellt sichere Verbindungen bereit, indem es zwei Anwendungen gestattet, eine Verbindung über ein Netzwerk herzustellen, um die gegenseitige Identität zu authentifizieren, und indem es die zwischen den Anwendungen ausgetauschten Daten verschlüsselt. Die Authentifizierung ermöglicht einem Server und wahlweise einem Client, die Identität der Anwendung am anderen Ende der Netzwerkverbindung zu verifizieren. Zusätzlich, wenn der Administrator den Zugriff auf individuelle JMS-Ziele beschränkt hat (Queues oder Topics), indem er eine Zugriffskontrolle auf den JNDI-Eintrag für das Ziel eingerichtet hat, muss der Service authentifizieren, wenn der Eintrag im JNDI-Baum gesucht wird.
Nachrichten-Selector
Geben Sie einen Nachrichten-Selector-Ausdruck ein.
Nur Nachrichten mit Eigenschaften, die dem Ausdruck des Nachrichten-Selectors entsprechen, werden verarbeitet.
JMS-Client-ID

Geben Sie nur bei Themen die Client-ID ein, die der Abonnent verwenden soll. Geben Sie keinen Wert ein, wird eine Client-ID automatisch generiert. Das Zuweisen einer benutzerdefinierten ID hilft Ihnen, diese Komponente während der Serviceüberwachung zu identifizieren. Beachten Sie, dass dies nur für dauerhafte Abonnenten gilt.

Die JMS-Client-ID ist einer von mehreren MDB-Deskriptoren, die zur Generierung der Abonnentennamen und des ClientID-Wertes für Abonnenten des Topics verwendet werden. Die Definition einer JMS-Client-ID vereinfacht die Identifizierung und Anzeige von Abonnenten für ein bestimmtes Topic, indem der Abonnentenname oder die Client-ID geprüft wird.

Weitere Informationen finden Sie im Oracle Fusion Middleware Developer's Guide for Oracle Service Bus (http://www.oracle.com/pls/as111170/lookup?id=OSBDV1037) unter "JMS Transport".
Dauerhaftes Abonnement
Die Option ist nur verfügbar, wenn Topic als Zieltyp gewählt wurde.
Aktivieren Sie dieses Kontrollkästchen, wenn das Abonnement dauerhaft ist, oder lassen Sie es leer, wenn das Abonnement nicht dauerhaft ist
Anzahl Wiederholungen
Geben Sie die Anzahl Wiederholungsversuche für die Zustellung einer Nachricht ein, bevor diese in ein Fehlerziel verschoben wird. Dieses Feld gilt nur für Oracle WebLogic Server JMS-Ziele.
Wiederholungsintervall
Geben Sie die Verzögerungszeit in Millisekunden an, wann zurückgerollte oder wiederhergestellte Nachrichten erneut zugestellt werden. Dieses Feld gilt nur für Oracle WebLogic Server JMS-Ziele.
Fehlerziel
Geben Sie den Namen des Ziels ein, für Nachrichten, die den Grenzwert der Neuzustellung erreicht haben. Dieses Feld gilt nur für Oracle WebLogic Server JMS-Ziele.
Ablauf-Policy
Wählen Sie die zu verwendende Ablauf-Policy, wenn auf einem Ziel eine abgelaufene Nachricht angetroffen wird. Dieses Feld gilt nur für Oracle WebLogic Server JMS-Ziele.
Ist XA erforderlich
Aktivieren Sie dieses Kontrollkästchen, wenn die Connection Factory XA ist.
Dieser Wert wird berücksichtigt, wenn die Remote Connection Factory nicht verfügbar ist. Wenn die Connection Factory verfügbar ist, und der Wert "true" ist, stellen Sie sicher, dass die Connection Factory als transaktional definiert wurde.
Transaktions-Timeout
Geben Sie die Zeit in Sekunden an, die der JMS-Proxy-Service auf die Verarbeitung einer Transaktion soll. Diese Option gilt nur für Services, die eine XA-Connection Factory verwenden. Der Standardwert beträgt 600 Sekunden.
Verteilung der Themenmeldungen
Diese Option ist verfügbar, wenn Sie als Zieltyp "Thema" wählen. Die Verteilung der Themenmeldungen stellt mehrere Optionen für den Umgang mit eingehenden JMS-Nachrichten und für hohe Verfügbarkeit und Failover bereit.

Wählen Sie eine der folgenden Eigenschaften, um festzulegen, wie Message-driven Beans eingehende JMS-Nachrichten behandeln sollen.
  • Eine Ausgabe pro Anwendung (Standard) – Wenn der JMS-Proxyservice in einem Cluster bereitgestellt wird, bietet diese Option hohe Verfügbarkeit und Skalierbarkeit, indem sie sicherstellt, dass eine eingehende JMS-Nachricht nur auf einem verfügbaren Server im Cluster verarbeitet wird.
  • Eine Ausgabe pro Server – Wählen Sie diese Option, wenn Sie eingehende JMS-Nachrichten zu einem Thema publizieren möchten, das der Proxy-Service der einzelnen Cluster-Mitglieder erhalten soll.
  • Kompatibilität – Wählen Sie diese Option, wenn Sie eingehende JMS-Nachrichten auf einem bestimmten Managed Server oder auf allen Managed Servern eines Clusters verarbeiten möchten. Bei Auswahl dieser Eigenschaft müssen Sie einen Zielserver auswählen.
Hinweis: Die einzig gültige Option für mit einer Oracle WebLogic Server-Version vor 10.3.4 bereitgestellte Themenziele ist "Kompatibilität".
Die "Eine Ausgabe"-Option, die Sie für die Verteilung der Themenmeldungen wählen, setzt die Abonnementfreigabe-Policy und/oder Client-ID-Policy außer Kraft, die Sie in JMS-Connection Factory konfiguriert haben.
Ziel
Diese Option ist nur in einem Oracle Service Bus-Cluster verfügbar, wenn Sie bei Verteilung der Themenmeldungen "Kompatibilität" wählen. Wählen Sie den Zielserver, der die eingehenden JMS-Nachrichten verarbeitet. Wenn Sie für diese Verteilung eine der "Eine Ausgabe"-Optionen wählen, wird in diesem Feld der Cluster-Name angezeigt.

Falls Sie kein Ziel angeben, erhält jede JMS-Proxy-Service-Instanz (die Nachrichten zum Thema abliest) auf jedem Managed Server im Cluster eine Ausgabe der Nachricht.
JNDI-Timeout
Das JNDI-Verbindungs-Timeout (in Sekunden) wird beim Lookup des Ziels oder der Connection Factory in der JNDI-Baumstruktur verwendet.
