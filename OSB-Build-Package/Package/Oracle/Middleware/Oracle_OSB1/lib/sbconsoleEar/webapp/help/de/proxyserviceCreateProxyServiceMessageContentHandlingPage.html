Seite "Nachrichtenbehandlungs-Policy"

Die Seite Proxy-Service erstellen/bearbeiten – Nachrichtenübermittlung ist eine von diversen Seiten, auf denen Sie Proxy-Services erstellen und bearbeiten können. Diese Funktionen sind unter Proxy-Services erstellen und konfigurieren und Proxy-Service-Konfigurationen bearbeiten genauer beschrieben.

Auf dieser Seite können Sie festlegen, ob der Proxy-Service Nachrichtencontent einlesen soll, wie der Service empfangene Anforderungsnachrichten decodieren und gesendete Antwortnachrichten codieren soll und ob der Proxy-Service MIME-Anhänge einlesen soll, anstatt Anhangscontent im Speicher zu puffern.

Auf dieser Seite können Sie festlegen, dass der Proxy-Service Nachrichtencontent einlesen und nicht im Speicher cachen soll. Sie können außerdem festlegen, dass der Proxy-Service eingehende Nachrichten im MTOM/XOP-Format decodieren und parsen und Antworten gegebenenfalls im MTOM/XOP-Format senden soll. Die SOAP-Nachrichtenübertragungsoptimierung (MTOM; Message Transmission Optimization Mechanism) ist eine Methode zum Senden von Binärdaten an und von Web Services. MTOM verwendet XOP (XML-binary Optimized Packaging) zum Übertragen der Binärdaten.

Auf dieser Seite können Sie auch festlegen, dass der Proxy-Service MIME-Anhangscontent als Datenträgerdatei speichern und die Daten durch Streaming verarbeiten soll, ohne den Anhangscontent zwischenzuspeichern. Dadurch können Proxy-Services große Anhänge zuverlässig und effizient verarbeiten.

In Tabelle 19-17 ist die Verwendung der Seite Proxy-Service erstellen/bearbeiten – Nachrichtenübermittlung beschrieben.

Option
Verwendung
Transaktion erforderlich
Aktivieren Sie diese Option, um sicherzustellen, dass Oracle Service Bus den Proxy-Service-Nachrichtenfluss im Kontext einer Transaktion ausführt. Wenn bereits eine globale Transaktion vorhanden ist, wird sie vom Transportprovider in der Anforderung eingegeben (auch, wenn Sie diese Option nicht aktivieren). Wenn keine globale Transaktion vorhanden ist, startet die Laufzeit des Nachrichtenflusses eine Transaktion. Wenn die Laufzeit des Nachrichtenflusses eine Transaktion startet, beginnt der Transaktionskontext vor der Validierung oder Ausführung der Service-Konfiguration (z.B. Sicherheitsprüfung oder WS-I-Validierung) und vor der Ausführung des Nachrichtenflusses. Dadurch wird sichergestellt, dass die gesamte Verarbeitung und Ausführung im Transaktionskontext enthalten ist.
Wenn die Laufzeit des Nachrichtenflusses eine Transaktion startet, ist die QoS "genau einmal". Wenn jedoch bei Service Callout- oder Publishing-Aktionen der Parameter für ausgehende QoS auf "Best-Effort" (Standardeinstellung) gesetzt ist, führt Oracle Service Bus diese Aktionen außerhalb des Transaktionskontexts aus. Damit Oracle Service Bus diese Aktionen im selben Anforderungstransaktionskontext ausführt, müssen Sie den QoS für diese Aktionen auf "genau einmal" setzen.
Der Service behält sein Nachrichtenmuster (synchron, asynchron, einseitig) unabhängig von der Einstellung für diese Option bei.
Bei Transaktions-Timeouts gilt der globale Timeout-Wert für Transaktionen, der in der Oracle WebLogic Server-Konsole konfiguriert wird.
Exceptions bei Transaktionen
Oracle Service Bus ruft bei nicht erfolgreichen Transaktionen den System Error Handler auf. Nicht erfolgreiche Transaktions-Exceptions können nicht in einem vom Benutzer konfigurierten Error Handler abgefangen werden. Bei synchronen Mustern wird eine Transaktions-Exception über die Antwort wiedergegeben. Bei asynchronen Mustern, bei denen die Transaktion für die Festschreibung in der Anforderung entwickelt wurde, wird die Exception zurück an den Anforderungs-Thread gesendet.
Hinweis: In einem asynchronen Muster hat ein Fehler in der Antwort, der nach dem Speichern der Transaktion auftritt, keine Auswirkungen auf die Transaktion.
Gleiche Transaktion für Antwort
Diese Option gilt nur für unidirektionale und asynchrone Nachrichtenmuster.

Wenn Sie diese Option aktivieren, propagiert Oracle Service Bus den Transaktionskontext vom Anforderungsthread an den Antwortthread.
Außerdem wird das Nachrichtenmuster automatisch synchron, unabhängig von der ursprünglichen Einstellung für das Nachrichtenmuster (z.B. asynchron oder einseitig).
Sie können diese Option z.B. verwenden, wenn der Business-Service in der Anforderung voraussetzt, dass eine Transaktion vor dem Senden der Antwort gespeichert wird, wie beim unidirektionalen Muster.
Bei Transaktions-Timeouts gilt der globale Timeout-Wert für Transaktionen, der in der Oracle WebLogic Server-Konsole konfiguriert wird.
Content-Streaming
Aktivieren Sie diese Option, wenn Nachrichtencontent eingelesen und nicht zwischengespeichert werden soll.
Aktivieren Sie das Kontrollkästchen Aktiviert, und wählen Sie eine der folgenden Optionen:
  • Zwischencontent im Speicher (Speicherpuffer) oder in einer Datenträgerdatei (Datenträgerpuffer) puffern
  • Komprimierung aktivieren
Weitere Informationen finden Sie unter "Streaming body Content" im Oracle Fusion Middleware Administrator's Guide for Oracle Service Bus unter http://www.oracle.com/pls/as111170/lookup?id=OSBAG314.
XOP-/MTOM-Unterstützung
Oracle Service Bus unterstützt XOP/MTOM mit den folgenden Transporten:
  • HTTP/S
  • Lokal
  • SB
Aktivieren Sie das Kontrollkästchen Aktiviert, damit ausgehende Nachrichten vom Proxy-Service im MTOM/XOP-Format decodiert und geparst werden. Hinweis: Diese Option ist bei importierten Proxy-Services, die auf Konfigurationen von früheren Releases basieren, deaktiviert.
Wenn die XOP/MTOM-Unterstützung aktiviert ist, wählen Sie eine der folgenden Optionen, um festzulegen, wie Binärdaten in den Nachrichtenkontextvariablen $header und $body verarbeitet werden sollen:
  • Binärdaten nach Wert aufnehmen: Ersetzen Sie in einer eingehenden Anforderungsnachricht beim Erstellen der Nachrichtenkontextvariablen $header und $body die xop:Include-Elemente durch ctx:binary-Kontentelemente.
  • Binärdaten nach Wert aufnehmen: Ersetzen Sie in einer eingehenden Anforderungsnachricht beim Erstellen der Nachrichtenkontextvariablen $header und $body die xop:Include-Elemente durch Base64-codierte Textversionen von entsprechenden Binärdaten.
Verwenden Sie die Option "Binärdaten nach Referenz aufnehmen", wenn Sie Direktzugriff auf Binärdaten benötigen, z.B. um Daten an ein Java-Callout oder eine MFL-(Message Format Language)Umwandlung weiterzugeben.
Verwenden Sie die Option "Binärdaten nach Wert aufnehmen" in den folgenden Fällen:
  • Zum Herstellen einer Brücke zwischen MTOM- und nicht-MTOM-Services. Beispiel: Stellen Sie sich einen MTOM-fähigen Proxy-Service vor, der eine Anforderung empfängt, die dann an einen nicht-MTOM-fähigen Service weitergeleitet wird. Sie können diese Option verwenden, um vorhandene Standards beim Senden von Binärdaten in XML in Base64-codierter Form einzuhalten.
  • Zum Validieren des Contents der Nachricht mit einem XML-Schema, das voraussetzt, dass ein base64binary-Element anstelle von Binärdaten verwendet wird
Hinweis: Wenn die XOP/MTOM-Unterstützung für einen Proxy-Service aktiviert ist, muss nicht jede eingehende Nachricht im MTOM-Format codiert werden. Diese Einstellung gibt stattdessen an, dass, wenn eine MTOM-formatierte Nachricht eingeht, der Proxy-Service diese entsprechend verarbeiten muss. Beachten Sie, dass auch wenn Proxy-Services, die nicht für die XOP-/MTOM-Unterstützung aktiviert sind, eine MTOM-formatierte Nachricht empfangen, der Service die Nachricht ablehnt und einen Laufzeitfehler meldet.
Anhänge
Oracle Service Bus unterstützt das Streaming von MIME-Anhängen mittels der folgenden Transporte:
  • HTTP/S
  • Lokal (wenn über einen HTTP-Proxy verkettet und das Streaming von Anhängen aktiviert ist)
Aktivieren Sie das Kontrollkästchen Seitenanhänge auf Datenträger, um den Proxy-Service für das Streaming von MIME-Anhängen zu aktivieren. Wenn die Option bei HTTP-Proxy-Services aktiviert wird, gilt sie für eingehende Proxy-Service-Anforderungsnachrichten.
Hinweis: Wenn Sie die XOP/MTOM-Unterstützung aktivieren, ist die Option "Anhänge" nur verfügbar, wenn Sie unter "XOP-/MTOM-Unterstützung" die Option Binärdaten nach Referenz aufnehmen aktivieren. Zudem müssen Payloads mit Anhängen RFC 822 entsprechen. Besonders Zeilen mit Internet-Headern müssen mit CRLF (Wagenrücklauf/Zeilenvorschub) beendet werden.
