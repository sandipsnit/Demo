Seite "Transportkonfiguration"

Die SeiteProxy-Service erstellen/bearbeiten - Transportkonfigurationist eine von einer Serie zum Erstellen und Bearbeiten von Proxy-Services, wie in Proxy-Services erstellen und konfigurieren und Proxy-Service-Konfigurationen beschrieben.

Verwenden Sie diese Seite, um ein Transportprotokoll für den Proxy-Service auszuwählen und andere allgemeine Transportkonfigurationseinstellungen festzulegen. Tabelle 19-3 beschreibt, wie die Seite zu verwenden ist.

Hinweis: Eingehende Sicherheit auf Transportebene trifft für Clientanwendungen und Oracle Service Bus Proxy-Services zu. Ausgehende Sicherheit auf Transportebene trifft auf die Verbindungen zwischen Oracle Service Bus Proxy-Services und Business-Services zu. Mehr Informationen über Sicherheit auf Transportebene finden Sie unter "Sicherheit auf Transportebene konfigurieren" im Oracle Fusion Middleware Developer's Guide for Oracle Service Bus unter http://www.oracle.com/pls/as111170/lookup?id=OSBDV1557.

Verwendung
Protokoll
Wählen Sie aus der Liste ein Transportprotokoll. Die verfügbaren Protokolle hängen vom erstellten Service-Typ ab:
  • WSDL Web-Service: http, jca, jms, local, sb, ws
  • Transporttyp: jejb
  • Messaging Service: email, file, ftp, http, jms, local, mq (falls verfügbar), sftp, tuxedo
  • Jeder SOAP-Service: http, jms, local, sb
  • Jeder XML-Service: email, file, ftp, http, jms, local, mq (falls verfügbar), sb, sftp, tuxedo
Endpunkt-URI
Geben Sie einen Endpunkt-URI ein, in einem Format basierend auf dem im Feld Protokoll gewählten Transportprotokoll oben: Die Formate sind:
  • email - mailfrom:mail-server-hostname:mail-server-port
  • file - file:///drivename:/somename
  • ftp - ftp://hostname:port/directory
  • http - someName

    • Beim HTTP-Transport werden sowohl HTTP- als auch HTTPS-Endpunkte unterstützt.

  • jca - jca://<resource_adapter_jndi>
  • jejb - Das URL-Format ist ejb_jndi_name. Der für einen JEJB-Proxyservice konfigurierte URI wird der globale JNDI-Name zum Suchen des statuslosen Session Beans, das der JEJB-Transport von der Remote-/Business-Schnittstelle im Client-JAR-Archiv generiert hat.

    • Hinweis: Für EJB 3.0, ist ejb_jndi_name das mappedName-Attribut für die @javax.ejb.Stateless-Anmerkung im generierten Bean. Dem Lookup-JNDI-Namen für den generierten EJB-Service wird #interface_class nachgestellt. Dies ist der vollständig angegebene Name der Business-Schnittstelle.

    Sie können auf den JEJB-Proxy-Service zugreifen wie folgt:

    EJB 2.1: protocol://host:port/ejb_jndi_name

    EJB 3.0: protocol://host:port/ejb_jndi_name#interface_class

    Hinweis: Wenn die EJBs auf IBM WebSphere ausgeführt werden, muss ejb_jndi_name folgendes Format haben:

    cell/nodes/node_name/servers/server_name/ejb_jndi_name

    or

    cell/clusters/cluster_name/ejb_jndi_name

    Beschreibungen dazu finden Sie in der IBM WebSphere-Dokumentation unter http://publib.boulder.ibm.com/infocenter/wasinfo/v6r1/index.jsp?topic=/com.ibm.websphere.nd.iseries.doc/info/iseriesnd/ae/rnam_example_prop3.html.

  • jms - jms://host:port/factoryJndiName/destJndiName

    • Wenn Sie ein JMS-Ziel für mehrere Server adressieren möchten, verwenden Sie das folgende URI-Format:

    jms://host1:port,host2:port/QueueConnectionFactory/DestName

    Beachten Sie: Wenn Sie einen Proxy-Service erstellen, können Sie einen JMS-Endpunkt-URI konfigurieren, auch wenn der Server an diesem Endpunkt nicht verfügbar ist. Wenn Sie jedoch im Fall von JMS die Session aktivieren, muss der Endpunkt verfügbar sein. Mehr darüber erfahren Sie unter Sessions aktivieren.

    Hinweis: Während bei Oracle WebLogic Server Vorwärtsschrägstriche in JNDI-Namen (Beispiel: "myqueues/myqueue") zulässig sind, stören JNDI-Namen mit Vorwärtsschrägstrichen das URI-Format, das für Oracle Service Bus erforderlich ist, und Sie können diese Namen nicht verwenden. Um dieses Problem zu umgehen, definieren Sie einen JMS-Fremdserver, und referenzieren Sie diesen Fremdserver im URI. Weitere Informationen finden Sie unter "Configure foreign servers" in der Oracle Fusion Middleware Oracle WebLogic Server Administration Console Online Help unter http://www.oracle.com/pls/as111170/lookup?id=WLACH.

    In einem Cluster: Die Hostnamen im JMS-URI müssen genau mit den Hostnamen der Cluster Server übereinstimmen, da sie in WebLogic Server konfiguriert werden.

  • local

    • Dieser Transport erfordert keinen Endpunkt-URI.

  • mq - mq://<local-queue-name>?conn=<mq-connection-resource-ref>

    • local-queue-name ist der Name der MQ-Queue, von der der Proxy-Service Nachrichten liest.

    mq-connection-resource-ref ist der Pfad (Projekt/Ordner) und Name der MQ-Verbindungsressource, z.B. default/my_MQconnection.

    Hinweis: Der Endpunkt-URI darf keine Leerzeichen enthalten, Sie dürfen also keine MQ-Verbindungsressourcen oder Projekte/Ordner mit Leerstellen im Namen erstellen.

    Informationen dazu, wie Sie einen MQ-Transport in Oracle Service Bus verfügbar machen, finden Sie unter MQ-Verbindungen.

Endpunkt-URI (Fortsetzung)
  • sb - service_name

    • service_name ist die eindeutige ID für den Proxy-Service. Standardmäßig ist dieser Name der Proxy-Service-Name.

    service_name darf nur in URIs zugelassene Zeichen enthalten (wie in RFC2396 unter http://www.ietf.org/rfc/rfc2396.txt beschrieben), folgende Zeichen dürfen nicht enthalten sein: (/) oder (:).

  • sftp - sftp://hostname:port/directory
  • tuxedo - exportname

    • Der URI exportname entspricht einem WTC-Export, den die Remote-Tuxedo-Domain als Tuxedo-Service erkennt.

    Wenn mehr als ein URI angegeben wurde, müssen Sie eindeutige Ressourcennamen für die Endpunkte haben. Wenn kein Remote-Name angegeben wurde, ist sein Wert der Wert des Ressourcennamens. Wenn keine Remote-Namen angegeben wurden, oder der Remote- und der Ressourcenname der gleiche sind, ist nur ein URI zugelassen. In diesem Fall haben Remote- und Ressourcenname den gleichen Wert. Das ermöglicht es Benutzern, bereits definierte WTC-Importe zu verwenden, um WTC-Load-Balancing und -Failover verwenden zu können.

    Hinweis: Wenn Sie zwei identische URI konfigurieren, zeigt ein Fehler an, dass der Service-Name bereits vorhanden ist.

  • ws - /contextPath

    • contextPathmuss für Proxy-Services, die entweder HTTP- oder WS-Transport verwenden, eindeutig sein.

Alle Header abrufen
Wählen Sie Ja, um alle Header vom Transport abzurufen.
Wählen Sie Nein, um eine bestimmte Header-Gruppe abzurufen. Wenn Sie Nein wählen, geben Sie im Feld Header eine Header-Gruppe ein, klicken Sie dann Hinzufügen. (Dieser Schritt entfällt bei Local-Transport.)
Hinweis: Oracle Service Bus gibt den HTTP-Autorisierungs-Header von der Anforderung nicht an die Pipeline weiter, denn dadurch entsteht eine Sicherheitslücke. Sie könnten versehentlich einen Log-Vorgang erstellen, der den Benutzernamen und das uncodierte Kennwort in eine Log-Datei schreibt. Wenn Ihr Design-Muster voraussetzt, dass der HTTP-Autorisierungs-Header in der Pipeline steht, gehen Sie wie folgt vor:

a. Setzen Sie im Startbefehl von Oracle Service Bus folgende Systemeigenschaft auf "true": com.bea.wli.sb.transports.http.GetHttpAuthorizationHeaderAllowed

b. In der Oracle Service Bus Administration-Konsole auf der Seite "Transportkonfiguration" wählen Sie Alle Header abrufen oder Benutzerdefinierte Headers und geben die Autorisierung an.
c. Starten Sie Oracle Service Bus neu.

Oracle Service Bus wird den Autorisierungs-Header an die Pipeline weiterreichen.

Wenn Sie fertig sind

Klicken Sie auf Weiter, um mit der Konfiguration dieses Service auf der nächsten Seite fortzufahren. Oder klicken Sie auf Letzte, um diese Konfiguration auf der Seite Zusammenfassung unter Proxy-Service erstellen/bearbeiten zu prüfen und zu speichern.
