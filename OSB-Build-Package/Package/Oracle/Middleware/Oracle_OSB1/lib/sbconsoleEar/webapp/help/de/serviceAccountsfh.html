Service-Accounts angeben

Sie können ein und denselben Service-Account für mehrere Business und Proxy-Services verwenden. Um Benutzernamen und Kennwort anzugeben, verwenden Sie eine der folgenden Methoden:

  • Statisch

    • Sie müssen hierbei Benutzername und Kennwort zusammen mit der Accountkonfiguration speichern. Der Service-Account codiert diesen Benutzernamen und das Kennwort in der ausgehenden Anforderung.

  • Passthrough von Benutzername und Kennwort

    • Veranlasst den Service-Account, den Benutzernamen und das Kennwort anzugeben, die er von der eingehenden Clientanforderung erhält. Beispiel: Wenn eine eingehende HTTP-Basisanforderung "pat" und "patspassword" als Benutzername und Kennwort enthält, codiert der Service-Account "pat" und "patspassword" in der ausgehenden Anforderung.

    Da diese Methode voraussetzt, dass Clientanforderungen Benutzernamen und Kennwörter in Klartext enthalten, ist sie nur für Clientanforderungen anwendbar, die entweder das HTTP-Basisprotokoll, das ist eine Authentifizierung des Web Services Security-Benutzernametokens mit Klartext-Kennwort, oder ein benutzerdefiniertes Benutzernamen- und Kennworttoken verwenden.

    Oracle empfiehlt die Verwendung dieser Methode nur, wenn Oracle Service Bus und der Endpunkt zur selben Authentifizierungsdomain gehören. Beispiel: Sie verwenden diese Methode, wenn Sie Nachrichten innerhalb einer Organisation weiterleiten und beide, Oracle Service Bus und der Nachrichtenverbraucher, sich gegenüber einem gemeinsamen LDAP-Server authentifizieren.

    Folgende Einschränkungen gelten bei dieser Methode:

    • Sie kann nicht in ausgehenden Anforderungen verwendet werden, die Oracle Service Bus gegenüber einem lokalen oder einem Remote-Server oder einer Systemressource, wie einem FTP-Server oder einem JMS-Server, authentifizieren.
    • Sie kann nicht mit der XQuery-Funktion fn-bea:lookupBasicCredentials verwendet werden. Weitere Informationen finden Sie unter "XQuery Implementation" im Oracle Fusion Middleware Administrator's Guide for Oracle Service Bus auf der Website http://www.oracle.com/pls/as111170/lookup?id=OSBAG444.
      • Hinweis: Wenn Ihr Proxy eine aktive WSS-Zwischenstufe ist, können Sie WS-Security zum Codieren eines WS-Security-Benutzernametokens oder einen benutzerdefinierten Benutzernamen/Kennwort verwenden. In diesem Fall funktioniert der Benutzername/Kennwort-Passthrough, da der Proxy zunächst die Anforderung verschlüsselt und dann erst Zugriff auf Benutzername und Kennwort in Klartext erhält.
  • Benutzerzuordnung

    • Erfordert, dass Sie den Benutzernamen, das ist das Ergebnis der Authentifizierung einer eingehenden Anforderung von einem Client (der lokale Benutzername), mit einem Benutzernamen und Kennwort korrelieren (zuordnen), das Sie angegeben haben (der Remote-Benutzername und Kennwort). Wenn der Service-Account eine Anforderung von einem authentifizierten Client erhält, der zugeordnet wurde, liefert dieser Service-Account den entsprechenden Remote-Benutzernamen und das Kennwort für die ausgehende Anforderung des Business oder Proxy-Service.

    Wenn der Client sowohl Transportebene als auch Nachrichtenebene authentifiziert, ordnet der Service-Account den Benutzernamen auf Nachrichtenebene dem Remote-Benutzernamen und Kennwort zu.

    Sie können auch einen anonymen Benutzernamen zu einem Remote-Benutzernamen und Kennwort zuordnen.

    Folgende Einschränkungen gelten bei dieser Methode:

    • Sie kann nicht in ausgehenden Anforderungen verwendet werden, die Oracle Service Bus gegenüber einem lokalen oder einem Remote-Server oder einer Systemressource, wie einem FTP-Server oder einem JMS-Server, authentifizieren.
    • Sie kann nicht mit der XQuery-Funktion fn-bea:lookupBasicCredentials verwendet werden. Weitere Informationen finden Sie unter "XQuery Implementation" im Oracle Fusion Middleware Administrator's Guide for Oracle Service Bus auf der Website http://www.oracle.com/pls/as111170/lookup?id=OSBAG444.
