Proxy-Service-Testdaten konfigurieren

Tabelle 32-1 listet die Konfigurationsoptionen für das Testen von Proxy-Services. Die Felder zum Akzeptieren von Eingaben im Anforderungsdokument hängen vom Service-Typ ab.

Tabelle 32-1 Konfigurationsoptionen für Proxy-Services

Beschreibung

Ausführen
Test ausführen.

Eingabewerte zurücksetzen.

Testkonfiguration
Testkonfigurationsoptionen festlegen.
Direkter Aufruf
Eine Nachricht an den Proxy-Service senden, ohne die Transportschicht von Oracle Service Bus zu verwenden. Die in die Testkonsole eingegebenen Daten müssen denen entsprechen, die vom Client an den Proxy-Service gesendet werden.
Das Gegenteil von einem direkten Aufruf ist ein indirekter Aufruf. Er kann durch Deaktivieren der Option Direkter Aufruf aktiviert werden. Er wird standardmäßig für Business-Services ausgeführt. Der indirekte Aufruf sendet Nachrichten über die Transportschicht. Dabei müssen die in die Testkonsole eingegebenen Daten denen entsprechen, die vom Proxy-Service an den aufgerufenen Service gesendet werden.
Einschließlich Tracing
Tracing zeigt den Status der Nachricht auf ihrem Weg durch das System an.
Anforderungsdokument
Die Eingabefelder generieren die Anforderungsnachricht, die an den Proxy-Service gesendet wird. Klicken Sie auf Ausführen, um den Test mit den eingegebenen Werten auszuführen. Die Testkonsole zeigt die Anforderungsnachricht, die Antwortnachricht des Service sowie Metadaten an.
Eingaben im Bereich "Anforderungsdokument" sind abhängig vom Service-Typ. In den folgenden Bereichen sind die Service-Typen und eine Beschreibung der jeweils erforderlichen Eingaben aufgelistet.
Beliebiges XML
Service-Anforderungseingabe in Form von Payload. Payload ist der Content der zu sendenden Nachricht. Der Content muss ein XML-Dokument sein. Sie können nach einer Datei suchen oder den Nachrichtencontent in das Textfeld eingeben.
Anforderungsdokument - Fortsetzung
Beliebiges SOAP
Service-Anforderungseingabe in Form von Payload. Payload ist der Content der zu sendenden Nachricht. Der Content muss ein SOAP-Envelope sein. Sie können nach einer Datei suchen oder den Nachrichtencontent in das Textfeld eingeben.
Messaging
Messaging Services legen vier mögliche Eingabetypen fest: Keine, XML, Binär oder Text. Der Service verlangt nur eine Eingabe - entweder dateibasiert oder textbasiert - mit Ausnahme des Typs Keine. Dieser Eingabetyp erfordert keinerlei Eingabe.
Oracle empfiehlt binäre Eingaben aus Dateien. Daten, die Sie in das Textfeld eingeben, werden von der Systemcodierung in eine binäre Eingabe umgewandelt.
Aus Dateien eingegebene Daten für Text-Services müssen in Text konvertiert werden. Das Codierungseingabefeld dient dazu, die Codierung anzugeben, die bei der Konvertierung angewendet werden soll. Ist das Feld nicht konfiguriert, wird die Systemcodierung verwendet. Standardmäßig wird das Codierungsfeld mit dem Codierungswert initialisiert, der am Proxy-Service-Endpunkt konfiguriert wurde.
XML
Service-Anforderungseingabe in Form von Payload. Payload ist der Content der zu sendenden Nachricht. Der Content muss ein XML-Dokument sein. Sie können nach einer Datei suchen oder den Nachrichtencontent in das Textfeld eingeben.
Wenn Ihr Proxy-Service ein WSDL-basierter Service mit mehreren definierten Vorgängen ist, generiert die Testkonsole ein Beispieldokument, das sie zum Testen des Service bereitstellt. Sie können diese Beispieldaten direkt verwenden, bearbeiten und dann den Test ausführen, oder Sie können eigene Testdaten angeben.
Alle Vorgänge sind oben auf der Seite aufgelistet.
Anforderungsdokument - Fortsetzung
SOAP-Dokument
Bei SOAP-Dokumenten besteht der SOAP-Envelope in der Regel aus keinem oder mehreren Headern und einer Body Payload. Über die Registerkarten Form und XML können Sie den Content näher spezifizieren.
In der Registerkarte Form befinden sich ein Feld für den SOAP-Header und eines für den SOAP-Body. Der Inhalt des Feldes "SOAP-Header" muss ein SOAP-Header Tag sein (so kann mehr als ein Header angegeben werden). Das Feld "SOAP-Body" enthält die eigentlichen Daten, die als Teil der Nachricht gesendet werden. Der Content muss ein XML-Dokument sein. Sowohl Header wie Body werden zum Generieren des SOAP-Envelopes verwendet.
Anforderungsdokument - Fortsetzung
SOAP-RPC
Bei SOAP-RPC besteht der SOAP-Envelope aus keinem oder mehreren Headern und keinem oder mehreren Argumenten.
Über die Registerkarten Form und XML können Sie den Content näher spezifizieren.
In der Registerkarte Form befinden sich ein Eingabefeld für SOAP-Header und jeweils ein Feld für die Argumente. (Der Name des Eingabefelds entspricht dem Namen des Arguments.) Der Content des Feldes "SOAP-Header" muss ein SOAP-Header Tag sein (so kann mehr als ein Header angegeben werden).
Die WSDL dient zum Erkennen des Typs der Argumente. Für einfache Typen gibt es ein einzeiliges Eingabefeld, für XML-Typen entsprechend ein mehrzeiliges Feld. Ein Bespieldokument wird automatisch generiert, um das Testen zu vereinfachen.
Die Header und Argumente werden von der Testkonsole verwendet, um den SOAP-Envelope zu generieren.
In der Registerkarte XML befindet sich nur ein Eingabefeld. Hier ist der SOAP-Envelope einzugeben, der gesendet werden soll. Die Payload (XML-Eingabe) kann datei- oder textbasiert sein. Das Referenzieren einer Datei für die Eingabe hat Priorität vor Texteingaben. Suchen Sie nach der Datei, mit der Sie den Test ausführen möchten.
Web Service Security
Dieser Bereich steht für SOAP-Services nur zur Verfügung, wenn der gewählte Vorgang eine Web Service Security Policy (WSS-Policy) hat.
Service-Provider
Der Test-Service ruft alle clientseitigen PKI-Zugangsdaten (Schlüsselpaar) für Web Service Security-Vorgänge (digitale Signatur und Verschlüsselung) von diesem Service-Provider ab. Dies ist ein fakultatives Feld. Bestimmte Szenarien verlangen, je nach der Anforderungs-/Antwort-Policy für den Vorgang, einen Service-Provider. Weitere Informationen finden Sie in "Testing Services with Web Service Security" im Oracle Fusion Middleware Administrator's Guide for Oracle Service Bus auf der Website http://www.oracle.com/pls/as111170/lookup?id=OSBAG384.
Benutzername
Benutzername in Web Service Security-Benutzernamentoken, die während des Test-Service erstellt wurden. Dies ist ein fakultatives Feld. Der Benutzername wird nur in bestimmten Szenarien benötigt, in denen die Anforderungs-Policy des Vorgangs eine Identity Assertion vorschreibt.
Verwechseln Sie dieses Feld nicht mit dem Benutzernamensfeld im Transportsicherheitskontext.
Es muss sich um einen gültigen Benutzername und ein gültiges Kennwort der lokalen Sicherheits-Realm handeln. Ein ungültiger Benutzername bzw. ein ungültiges Kennwort führen auf Clientseite zu einem Test-Service-Fehler.
Kennwort
Kennwort in Web Service Security-Benutzernamentoken, die während des Test-Service erstellt wurden.
Transport
Der Bereich Transport ist standardmäßig ausgeblendet. Die Felder und Werte hängen von der Testkonfiguration ab.
Authentifizierung
Benutzername
Benutzername zum Festlegen des Sicherheitskontexts, den der Test-Service verwendet, wenn der Proxy-Service aufgerufen wird.
Wenn der Proxy-Service die Nachricht an einen Business-Service weiterleitet, der ein SAML-Token erwartet, ist dies die Identität, die das Token darstellt. Weitere Informationen finden Sie unter "Using SAML for Authentication" im Oracle Fusion Middleware Developer's Guide for Oracle Service Bus auf der Website http://www.oracle.com/pls/as111170/lookup?id=OSBDV1647.
Verwechseln Sie dieses Feld nicht mit dem Benutzernamensfeld für die Web Service Security (WSS). Es muss sich um einen gültigen Benutzernamen und ein gültiges Kennwort der lokalen Sicherheits-Realm handeln. Ein ungültiger Benutzername bzw. ein ungültiges Kennwort führen auf Clientseite zu einem Test-Service-Fehler.
Hinweis: Wenn die Testkonsole ein Proxy mit Authentifizierung durch benutzerdefinierten Token auf HTTP-Ebene aufruft, wird die Authentifizierungsprüfung nicht durchgeführt.
Kennwort
Zugehöriges Kennwort. Weitere Informationen finden Sie unter Benutzername.
Aufrufmodus
Anforderung/Antwort
Diese Option wird nur angezeigt, wenn ein SOAP- oder XML-Proxy-Service getestet wird. Deaktivieren Sie das Kontrollkästchen Anforderung/Antwort für einseitige Service-Aufrufe.
