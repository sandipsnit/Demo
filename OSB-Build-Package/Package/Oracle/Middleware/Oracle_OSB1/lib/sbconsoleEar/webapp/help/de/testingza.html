Anforderungsdokument
Die Eingabefelder generieren die Anforderungsnachricht, die an den Business-Service gesendet wird. Klicken Sie auf Ausführen, um den Test mit den eingegebenen Werten auszuführen. Die Testkonsole zeigt die Anforderungsnachricht und die Antwortnachricht des Service an.
Eingaben im Bereich "Anforderungsdokument" sind abhängig vom Service-Typ. In den folgenden Bereichen sind die Service-Typen und eine Beschreibung der jeweils erforderlichen Eingaben aufgelistet.
Beliebiges XML
Der Service erfordert Eingabe in Form einer Payload. Die Payload ist der Content der zu sendenden Nachricht. Als Content wird ein XML-Dokument erwartet. Sie können nach einer Datei suchen oder den Nachrichtencontent in das Textfeld eingeben.
Beliebiges SOAP
Service-Anforderungseingabe in Form von Payload. Payload ist der Content der zu sendenden Nachricht. Der Content muss ein SOAP-Envelope sein. Sie können nach einer Datei suchen oder den Nachrichtencontent in das Textfeld eingeben.
Anforderungsdokument - Fortsetzung
Messaging
Messaging Services definieren 4 mögliche Eingabetypen: keine, XML, Binär oder Text. Der Service fordert eine einzelne Eingabe entweder datei- oder textbasiert, außer für den Typ keine, dieser erfordert keine Eingabe.
Oracle empfiehlt die Eingabe von binärem Input aus einer Datei. Daten, die in den Textbereich eingegeben werden, werden mit der Systemcodierung zu binären Daten konvertiert.
Aus Dateien eingegebene Daten für Text-Services müssen in Text konvertiert werden. Das Codierungseingabefeld dient dazu, die Codierung anzugeben, die bei der Konvertierung angewendet werden soll. Ist das Feld nicht konfiguriert, wird die Systemcodierung verwendet. Standardmäßig wird das Codierungsfeld mit dem Codierungswert initialisiert, der am Proxy-Service-Endpunkt konfiguriert wurde.
Anforderungsdokument - Fortsetzung
XML
Der Service erfordert Eingabe in Form einer Payload. Die Payload ist der Content der zu sendenden Nachricht. Als Content wird ein XML-Dokument erwartet. Sie können nach einer Datei suchen oder den Nachrichtencontent in das Textfeld eingeben.
Dies ist ein WSDL-basierter Service mit mehreren definierten Vorgängen. Oracle liefert ein Musterdokument, das beim Testen dieses Service zu verwenden ist. Alle Vorgänge sind oben auf der Seite aufgelistet, ein Pfeil markiert die gewählten Vorgänge.
SOAP-Dokument
Bei einem SOAP-Dokument besteht der SOAP-Envelope normalerweise aus null oder mehreren Headern und einer Hauptteil-Payload.
Über die Registerkarten Form und XML können Sie den Inhalt näher spezifizieren.
In der Registerkarte Form befinden sich ein Feld für den SOAP-Header und eines für den SOAP-Body. Der Inhalt des Felds "SOAP-Header" muss ein SOAP-Header Tag sein (so kann mehr als ein Header angegeben werden). Das Feld "SOAP-Body" enthält die eigentlichen Daten, die als Teil der Nachricht gesendet werden. Der Inhalt muss ein XML-Dokument sein. Sowohl Header wie Body werden zum Generieren des SOAP-Envelope verwendet.
Die Registerkarte XML enthält ein Eingabefeld. Als Content des Feldes wird der zu sendende SOAP-Envelope erwartet.
Anforderungsdokument - Fortsetzung
SOAP-RPC
Bei SOAP-RPC besteht der SOAP-Envelope aus keinem oder mehreren Headern und keinem oder mehreren Argumenten.
Über die Registerkarten Form und XML können Sie den Content näher spezifizieren.
In der Registerkarte Form befinden sich ein Eingabefeld für SOAP-Header und jeweils ein Feld für die Argumente (Name des Eingabefelds entspricht dem Namen des Arguments). Der Inhalt des Feld "SOAP-Header" muss ein SOAP-Header Tag sein (so kann mehr als ein Header angegeben werden).
Die WSDL dient zum Erkennen des Typs der Argumente. Für einfache Typen gibt es ein einzeiliges Eingabefeld, für XML-Typen entsprechend ein mehrzeiliges Feld. Ein Beispieldokument wird automatisch generiert, um das Testen zu vereinfachen.
Die Header und Argumente werden von der Testkonsole verwendet, um den SOAP-Envelope zu generieren.
In der Registerkarte XML befindet sich nur ein Eingabefeld. Hier ist der SOAP-Envelope einzugeben, der gesendet werden soll. Die Payload (XML-Eingabe) kann datei- oder textbasiert sein. Das Referenzieren einer Datei für die Eingabe hat Priorität vor Texteingaben. Suchen Sie nach der Datei, mit der Sie den Test ausführen möchten.
Web Service Security
Dieser Abschnitt ist nur verfügbar für SOAP-Services, wenn der gewählte Vorgang über eine Web Service-Security-Policy verfügt.
Service-Provider
Der Test-Service ruft alle clientseitigen PKI-Zugangsdaten (Schlüsselpaar) für Web Service Security-Vorgänge (digitale Signatur und Verschlüsselung) von diesem Service-Provider ab. Dies ist ein fakultatives Feld. Bestimmte Szenarien verlangen, je nach der Anforderungs-/Antwort-Policy für den Vorgang, einen Service-Provider. Weitere Informationen finden Sie in "Testing Services with Web Service Security" im Oracle Fusion Middleware Administrator's Guide for Oracle Service Bus auf der Website http://www.oracle.com/pls/as111170/lookup?id=OSBAG384.
Web Service Security Fortsetzung
Benutzername
Benutzername in Web Service Security-Benutzernamentoken, die während des Test-Service erstellt wurden. Dies ist ein fakultatives Feld. Der Benutzername wird nur in bestimmten Szenarien benötigt, in denen die Anforderungs-Policy des Vorgangs eine Identity Assertion vorschreibt.
Verwechseln Sie dieses Feld nicht mit dem Benutzernamenfeld im Transportsicherheitskontext. Hier muss ein gültiger Benutzername und ein Kennwort in der lokalen Sicherheits-Realm angegeben sein. Ein ungültiger Benutzername oder ein ungültiges Kennwort führen zu einem clientseitigen Fehler auf dem Test-Service.
In einigen Szenarien können dieser Benutzername und das Kennwort auch verwendet werden, wenn der Test-Services eine SAML-Assertion generiert.
Kennwort
Das in den vom Test-Service generierten Benutzernamentoken von Web Service Security generierte Kennwort .
Transport
Der Transportabschnitt ist standardmäßig ausgeblendet. Die Felder und Werte hängen von der Testkonfiguration ab.
Authentifizierung
Benutzername
Der Benutzername für das Einrichten des vom Test-Service beim Aufruf des Business-Service verwendeten Sicherheitskontexts. Wenn der Business-Service ein SAML-Token erwartet, darf diese Identität im SAML-Token propagiert werden. Siehe "SAML zur Authentifizierung verwenden" im Oracle Fusion Middleware Developer's Guide for Oracle Service Bus unter http://www.oracle.com/pls/as111170/lookup?id=OSBDV1647. In der lokalen Sicherheits-Realm muss dies ein gültiger Benutzername und ein gültiges Kennwort sein. Ein ungültiger Benutzername oder ein ungültiges Kennwort werden im Test-Service zu einem clientseitigen Fehler führen.
Kennwort
Zugehöriges Kennwort. Weitere Informationen finden Sie unter Benutzername.
Service-Provider
Das Feld wird verwendet, wenn ein HTTPS-Business-Service mit CLIENT-CERT-Authentifizierung getestet wird. Mit dem Service-Provider müssen SSL-Client-Zugangsdaten verknüpft sein. Der Test-Service wird diese Zugangsdaten beim SSL-Handshake verwenden.
Aufrufmodus
Anforderung/Antwort
Diese Option wird nur angezeigt, wenn ein beliebiger SOAP oder beliebiger XML-Business-Service getestet wird. Deaktivieren Sie das Kontrollkästchen Anforderung/Antwort für unidirektionale Service-Aufrufe.
