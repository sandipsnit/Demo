Página Configuración de Transporte

La página Crear/Editar un Servicio de Negocio: Configuración de Transporte es una de una serie de páginas para la creación y edición de servicios de negocio, como se describe en Creación y Configuración de Servicios de Negocio y Edición de Configuraciones de Servicio de Negocio.

Utilice esta página para seleccionar un protocolo de transporte para el servicio de negocio y para definir otros valores de configuración general de transporte.

La seguridad de nivel de transporte saliente se aplica a las conexiones entre los servicios de negocio y los servicios de proxy de Oracle Service Bus. Para obtener más información sobre la seguridad de nivel de transporte, consulte la sección sobre configuración de seguridad de nivel de transporte en Oracle Fusion Middleware Developer's Guide for Oracle Service Bus (Guía del Desarrollador de Oracle Fusion Middleware para Oracle Service Bus) en http://www.oracle.com/pls/as111170/lookup?id=OSBDV1557.

Protocolo
Seleccione un protocolo de transporte de la lista. Los protocolos disponibles dependen del tipo de servicio que desee crear:
  • Servicio Web WSDL: soa-direct, bpel-10g, dsp, http, jca, jms, sb, ws
  • Servicio de Transporte según Configuración: ejb, flujo, jejb
  • Servicio de mensajes: Correo electrónico, archivo, ftp, http, jms, mq (si está disponible), sftp, tuxedo
  • Cualquier Servicio SOAP: dsp, http, jms, sb
  • Cualquier Servicio XML: dsp, correo electrónico, archivo, ftp, http, jms, mq (si está disponible), sb, sftp, tuxedo
Algoritmo de Equilibrio de Carga
Seleccione uno de los siguientes algoritmos de equilibrio de carga:
  • asignación en rueda (round-robin): Este algoritmo ordena de forma dinámica las URL introducidas en el campo URI de Punto Final para este servicio de negocio. Si falla la primera, se intenta con la siguiente y así sucesivamente hasta que se agota el número de reintentos.

    • Para cada nuevo mensaje, hay un nuevo orden de URL.

  • aleatorio: Este algoritmo ordena de forma aleatoria la lista de URL introducidas en el campo URI de Punto Final para este servicio de negocio. Si falla la primera, se intenta con la siguiente y así sucesivamente hasta que se agota el número de reintentos.
  • ponderado de forma aleatoria: Este algoritmo ordena de forma aleatoria la lista de URL introducidas en el campo URI de Punto Final para este servicio de negocio, pero algunas se intentan más veces que otra dependiendo del valor introducido en el campo Ponderación.
  • ninguno: Este algoritmo ordena la lista de URL introducidas en el campo URI de Punto Final para este servicio de negocio de arriba a abajo.
URI de Punto Final
Introduzca un URI de punto final en el formato basado en el protocolo de transporte que seleccione en el campo Protocolo. A continuación se muestran las descripciones de los formatos de URI para cada transporte. Los elementos URI opcionales se muestran entre corchetes []. Para obtener más información sobre los URI de transporte, consulte los respectivos capítulos en Oracle Fusion Middleware Developer's Guide for Oracle Service Bus (Guía del Desarrollador de Oracle Fusion Middleware para Oracle Service Bus) en http://www.oracle.com/pls/as111170/lookup?id=OSBDV.
  • bpel-10g: protocolo://host[:puerto] [/ruta_acceso_protocolo]/dominio/proceso[/versión[/enlace_partner/rol]]
  • dsp: t3://dirección_ip_dsp:puerto/nombre_aplicación_dsp
  • ejb: ejb:nombre_proveedor_jndi:nombre_ejb_jndi

    • En el URI, nombre_proveedor_jndi es el nombre del proveedor del recurso de JNDI y nombre_ejb_jndi es el nombre JNDI del servidor de JNDI para EJB.

    Si el proveedor de JNDI está en el mismo servidor, no es necesario especificar el proveedor de JNDI. El URI sería ejb::nombre_ejb_jndi.

    Nota: Si sus EJB se están ejecutando en IBM WebSphere, nombre_jndi_ejb debe tener el siguiente formato:

    cell/nodes/nombre_nodo/servers/nombre_servidor/nombre_jndi_ejb

    o bien

    cell/clusters/nombre_cluster/nombre_jndi_ejb

    como se describe en la documentación de IBM WebSphere en http://publib.boulder.ibm.com/infocenter/wasinfo/v6r1/index.jsp?topic=/com.ibm.websphere.nd.iseries.doc/info/iseriesnd/ae/rnam_example_prop3.html.

  • correo electrónico: Puede introducir varios URI de punto final en cualquier combinación de los siguientes formatos:
    • mailto:nombre@nombre_dominio.com
    • mailto:nombre@nombre_dominio.com?smtp=recurso_servidor_smtp
    • mailto:nombre@nombre_dominio.com?mailsession=sesión_correo_jndi
  • archivo: file:///nombre_unidad:/algún_nombre
  • flujo: flow:<ruta de acceso de referencia al recurso de división-unión>

    • La ruta de acceso de referencia al recurso de división-unión es la ruta de acceso (proyecto/carpeta) y el nombre de la división-unión; por ejemplo, batchorderProcessing/batchorder.

  • ftp : ftp://host:puerto/directorio
  • http: http://host:puerto/algún_servicio

    • El transporte de HTTP soporta puntos finales de HTTP y HTTPS.

  • jca: jca://<jndi_adaptador_recurso>
  • jejb - jejb:nombre_proveedor_jndi:nombre_jndi_ejb

    • Ejemplo de EJB 2.1: jejb:miProveedor:osb.jejb.miJejbBiz21

    Ejemplo de EJB 3.0: jejb:miProveedor:miBiz31#osb.jejb.miJejbBiz

    Nota: Si sus EJB se están ejecutando en IBM WebSphere, nombre_jndi_ejb debe tener el siguiente formato:

    cell/nodes/nombre_nodo/servers/nombre_servidor/nombre_jndi_ejb

    o bien

    cell/clusters/nombre_cluster/nombre_jndi_ejb

    como se describe en la documentación de IBM WebSphere en http://publib.boulder.ibm.com/infocenter/wasinfo/v6r1/index.jsp?topic=/com.ibm.websphere.nd.iseries.doc/info/iseriesnd/ae/rnam_example_prop3.html.

  • jms: jms://host:puerto/nombre_jndi_fábrica/nombre_jndi_destino

    • Para que el destino de JMS sean varios servidores, utilice el siguiente formato de URI:
    jms://host1:puerto,host2:puerto/fábrica_conexiones_cola/nombre_destino

    Nota: aunque Oracle WebLogic Server permite el uso de barras inclinadas en los nombres de JNDI (por ejemplo, "miscolas/micola"), los nombres de JNDI con barras inclinadas interfieren en el formato de URI que necesita Oracle Service Bus, por lo que no se pueden utilizar esos nombres. Para solucionar este problema, defina un servidor ajeno de JMS y haga referencia a él en el URI. Para obtener más información, consulte la sección sobre configuración de servidores ajenos en Oracle Fusion Middleware Oracle WebLogic Server Administration Console Online Help (Ayuda en Pantalla de la Consola de Administración de Oracle Fusion Middleware Oracle WebLogic Server), en http://www.oracle.com/pls/as111170/lookup?id=WLACH.

    En un cluster: Los nombres de host del URI de JMS deben coincidir con de los servidores del cluster configurados en WebLogic Server.

URI de Punto Final (continuación)
  • mq: mq://<nombre_cola_local>?conn=<referencia_recurso_conexión_mq>

    • nombre_cola_local es el nombre de la cola de MQ desde la que el servicio de negocio lee los mensajes.

    referencia_recurso_conexión_mq es la ruta de acceso (proyecto/carpeta) y el nombre del recurso de conexión de MQ; por ejemplo, default/my_MQconnection.

    Nota: El URI de punto final no puede contener espacios, por lo tanto, no cree recursos de conexión de MQ o proyectos/carpetas con espacios en los nombres.

    Para que el transporte de MQ esté disponible en Oracle Service Bus, consulte Conexiones de MQ.

  • sb: sb://[<nombre_proveedor_jndi/>]nombre_servicio

    • nombre_proveedor_jndi (opcional) es el nombre del recurso de proveedor JNDI de Oracle Service Bus. Si se omite, se utiliza el contexto por defecto.

    nombre_servicio es un servicio de destino y se corresponde al URI de servicio de proxy remoto.

  • sftp: sftp://nombre_host:puerto/directorio
  • soa-direct: [protocolo://autoridad]/valor_por_defecto/nombre_compuesto[!número_versión[*etiqueta]]/nombre_servicio

    • El protocolo y la autoridad son opcionales si los servicios de SOA están en el mismo servidor que Oracle Service Bus.

  • tuxedo: tuxedo:nombre_recurso[/nombre_remoto]

    • En el URI, nombre_recurso corresponde a un nombre de importación de WTC y nombre_remoto corresponde al nombre del servicio exportado por el dominio remoto de Tuxedo. El nombre_recurso de URI es necesario y el nombre_remoto es opcional.

    Si se especifica más de un URI, debe tener nombres de recursos únicos para los puntos finales. Si no se especifica ningún nombre remoto, su valor es el valor del nombre del recurso. Si no se introduce ningún nombre remoto o si el nombre remoto y de recurso es el mismo, sólo se permite un URI. En este caso, el nombre de recurso y remoto tienen el mismo valor. Esto permite a las importaciones de WTC ya definidas utilizar el equilibrio de carga y failover de WTC.

  • ws: http://host:puerto/algún_nombre o https://host:puerto/algún_nombre
Haga clic en Agregar para agregar uno o varios URI adicionales. En tiempo de ejecución, las URL se seleccionan en función del algoritmo de equilibrio de carga seleccionado en el campo Algoritmo de Equilibrio de Carga.
Si ha seleccionado ponderado de forma aleatoria en el campo Algoritmo de Equilibrio de Carga, puede introducir una ponderación en el campo URI de Punto Final. El valor por defecto es 1.
Si ha definido varios puntos finales y ha seleccionado ninguno en el campo Algoritmo de Equilibrio de Carga, es importante el orden de los puntos finales. Puede cambiar el orden con las flechas Arriba y Abajo de la columna Opciones.
Oracle Service Bus no soporta URI de punto final duplicados en el mismo servicio de negocio.
Recuento de Reintentos
En caso de fallo de entrega al enviar solicitudes salientes, especifique el número de veces que se intenta cada punto final de la URL; es decir, el número de intentos de failover.
Por ejemplo, un servicio de negocio tiene un URI configurado (U1) y el número de reintentos se ha definido en 3. Si falla U1 en el primer intento, el sistema reintenta el punto final U1 3 veces más.
Si un servicio de negocio tiene 2 URI configurados (U1 y U2) y un recuento de reintentos de 3, si falla el primer reintento (por ejemplo, para U1), el sistema lo intenta (failover) con el siguiente URI (U2). Si también falla, el sistema realiza dos intentos más, uno para U1 y otro para U2.
Intervalo de Iteración de Reintentos
Especifique el número de segundos para la pausa del sistema antes de iterar todos los URI de punto final de la lista de nuevo.
Por ejemplo, un servicio de negocio tiene dos URI configurados (U1 y U2) y un recuento de reintentos de 2 con un intervalo de iteración de reintentos de 5 segundos. Si el primer intento (para U1) falla, el sistema lo intenta de inmediato con U2. Pero si U2 falla también, el sistema espera durante 5 segundos y vuelve a intentarlo con U1 una vez más.
Reintentar tras Errores de Aplicación
Seleccione Sí o No.
En caso de fallo de entrega al enviar solicitudes salientes, especifique si se deben reintentar los URI de punto final según los errores de la aplicación (por ejemplo, un fallo de SOAP).
Este campo sólo está disponible para estos transportes: HTTP, EJB, JMS, DSP, Tuxedo, SB y WS.
