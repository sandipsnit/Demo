Página Manejo de Mensajes

La página Crear/Editar un Servicio de Proxy: Manejo de Mensajes es una de una serie de páginas para la creación y edición de servicios de proxy, como se describe en Creación y Configuración de Servicios de Proxy y Edición de Configuraciones de Servicio de Proxy.

Utilice esta página para especificar si el servicio de proxy debe hacer fluir el contenido del mensaje, cómo debe el servicio descodificar los mensajes de solicitud recibidos y codificar los mensajes de respuesta enviados, y si el servicio de proxy debe hacer fluir los anexos MIME en lugar de almacenar en buffer el contenido del anexo en memoria.

Con esta página, puede activar el servicio de proxy para hacer fluir el contenido del mensaje en lugar de almacenarlo en memoria. También puede activar el servicio de proxy para descodificar y analizar los mensajes entrantes con formato MTOM/XOP y enviar respuestas con el formato MTOM/XOP, cuando corresponda. El mecanismo de optimización de transmisión de mensajes (MTOM) de SOAP es un método de envío de datos binarios y desde servicios web. MTOM utiliza el paquete optimizado XML binario (XOP) para transferir los datos binarios.

Con esta página, también puede activar el servicio de proxy para almacenar el contenido de los anexos MIME en un archivo del disco y, a continuación, procesar los datos en forma de flujo sin almacenar el contenido del anexo en el buffer de la memoria. Esto permite al servicio de proxy procesar grandes anexos de forma segura y eficaz.

Transacción Necesaria
Seleccione esta opción para garantizar que Oracle Service Bus ejecuta el flujo de mensajes del servicio de proxy en el contexto de una transacción. Si ya existe una transacción global, el proveedor de transporte lo propaga en la solicitud (incluso si no selecciona esta opción). Si no existe ninguna transacción global, el tiempo de ejecución del flujo de mensajes inicia una transacción. Si el tiempo de ejecución del flujo de mensajes inicia una transacción, el contexto de la transacción se inicia antes de que se valide o ejecute la configuración del servicio (por ejemplo, comprobación de seguridad o validación de WS-I), y antes de la ejecución del flujo de mensajes, garantizando que todo el procesamiento y ejecución se produce en el contexto de la transacción.
Si el tiempo de ejecución del flujo de mensajes inicia una transacción, la calidad del servicio es Una Sola Vez. Sin embargo, si las acciones de llamadas de servicio o de publicación tienen el parámetro de calidad del servicio saliente definido en Mayor Esfuerzo (valor por defecto), Oracle Service Bus ejecuta dichas acciones fuera del contexto de la transacción. Para que Oracle Service Bus ejecute dichas acciones en el mismo contexto de la transacción de solicitud, defina la calidad del servicio de dichas acciones en Una Sola Vez.
El servicio mantiene su patrón de mensajes (síncrono, asíncrono, unidireccional) independientemente del valor de esta opción.
Para timeouts de transacción, se aplica el valor de timeout de transacción global configurado en la consola de Oracle WebLogic Server.
Excepciones en Transacciones
Oracle Service Bus llama al manejador de errores del sistema para transacciones fallidas. No puede capturar excepciones de transacciones fallidas en un manejador de errores configurado por el usuario. Para patrones síncronos, se devuelve una excepción de transacción a través de la respuesta. Para patrones asíncronos, en los que la transacción está designada para confirmarse en la solicitud, la excepción se devuelve en el thread de solicitud.
Tenga en cuenta que en patrones asíncronos, un error en la respuesta que se produce después de la confirmación de la transacción en la solicitud no afecta a la transacción.
Misma Transacción para Respuesta
Esta opción se aplica sólo a los patrones de mensajes unidireccionales y asíncronos.

Si selecciona esta opción, Oracle Service Bus propaga el contexto de la transacción del thread de solicitud al thread de respuesta.
Si selecciona esta opción, el patrón de mensaje se convierte en asíncrono automáticamente, independientemente del valor del patrón de mensaje inicial (como asíncrono o unidireccional).
No debe utilizar esta opción, por ejemplo, si el servicio de negocio de la solicitud ha requerido una confirmación de transacción antes de enviar la respuesta, como en un patrón unidireccional.
Para timeouts de transacción, se aplica el valor de timeout de transacción global configurado en la consola de Oracle WebLogic Server.
Flujo de Contenido
Seleccione esta opción para hacer fluir el contenido del mensaje en lugar de almacenarlo en la memoria.
Active la casilla de control Activado y seleccione lo siguiente:
  • Si se va a almacenar en buffer el contenido intermedio en memoria (Buffer de Memoria) o en un archivo de disco (Buffer de Disco)
  • Si se va a activar Compresión
Para obtener más información, consulte la sección sobre flujo del contenido del cuerpo en Oracle Fusion Middleware Administrator's Guide for Oracle Service Bus (Guía del Administrador de Oracle Fusion Middleware para Oracle Service Bus) en http://www.oracle.com/pls/as111170/lookup?id=OSBAG314.
Soporte XOP/MTOM
Oracle Service Bus soporta XOP/MTOM con los siguientes transportes:
  • HTTP/S
  • Local
  • SB
Active la casilla de control Activado para permitir que el servicio de proxy descodifique y analice los mensajes entrantes en formato MTOM/XOP y para enviar respuestas mediante el formato MTOM/XOP, cuando corresponda. Tenga en cuenta que esta opción está desactivada para los servicios de proxy importados basados en configuraciones de versiones anteriores.
Si se activa el soporte XOP/MTOM, seleccione la forma de manejar los datos binarios en las variables de contexto de mensaje $header y $body entre las siguientes opciones:
  • Incluir Datos Binarios por Referencia: (Valor por defecto) En un mensaje de solicitud entrante, sustituya los elementos xop:Include por elementos ctx:binary-content al definir las variables de contexto de mensaje $header y $body.
  • Incluir Datos Binarios por Valor: En un mensaje de solicitud entrante, sustituya los elementos xop:Include por las versiones de texto codificado con Base64 correspondientes a los datos binarios al definir las variables de contexto de mensaje $header y $body.
Utilice Incluir Datos Binarios por Referencia cuando necesite acceso directo a datos binarios, por ejemplo, para transferir datos a una llamada de Java o una transformación de Lenguaje de Formato de Mensajes (MFL).
Utilice Incluir Datos Binarios por Valor en los siguientes casos:
  • Para establecer puentes entre los servicios MTOM y no MTOM. Por ejemplo, considere un servicio de proxy activado para MTOM que recibe una solicitud que, a continuación, se direcciona a un servicio activado no para MTOM. Puede utilizar esta opción para cumplir los estándares existentes para enviar datos binarios en XML con formato codificado en Base64.
  • Para validar el contenido del mensaje con respecto a un esquema XML que necesita que se utilice un elemento base64binary en lugar de datos binarios.
Tenga en cuenta que si está activado el soporte XOP/MTOM para un servicio de proxy, no es necesario que cada mensaje entrante tenga formato MTOM. En su lugar, este valor especifica que llega un mensaje con formato MTOM, el servicio de proxy debe manejarlo según corresponda. Tenga en cuenta también que cuando los servicios de proxy no activados para soporte XOP/MTOM reciben un mensaje con formato MTOM, el servicio rechaza el mensaje y emite un error en tiempo de ejecución.
Anexos
Oracle Service Bus soporta el flujo de anexos MIME con los transportes siguientes:
  • HTTP/S
  • Local (cuando se encadenan mediante un proxy HTTP con anexos de flujo activados)
Active la casilla de control Paginar Anexos en Disco para permitir que el servicio de proxy haga fluir anexos MIME. Si se activa para servicios de proxy HTTP, la opción se aplica a mensajes de solicitud entrantes del servicio de proxy.
Tenga en cuenta que si selecciona Soporte XOP/MTOM, la opción Anexos sólo está disponible si selecciona la opción Incluir Datos Binarios por Referencia en Soporte XOP/MTOM. Tenga en cuenta también que las cargas útiles que contienen anexos deben cumplir con la norma RFC 822. En concreto, las líneas que contienen cabeceras de Internet deben terminar en CRLF (salto de línea de retorno de carro).
