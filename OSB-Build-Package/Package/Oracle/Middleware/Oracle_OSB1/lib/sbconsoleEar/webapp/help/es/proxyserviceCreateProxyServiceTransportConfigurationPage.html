Página Configuración de Transporte

La página Crear/Editar un Servicio de Proxy: Configuración de Transporte es una de una serie de páginas para la creación y edición de servicios de proxy, como se describe en Creación y Configuración de Servicios de Proxy y Edición de Configuraciones de Servicio de Proxy.

Utilice esta página para seleccionar un protocolo de transporte para el servicio de proxy y para definir otros valores de configuración general de transporte. En la Tabla 19-3 se describe cómo utilizar la página.

Nota: La seguridad a nivel de transporte entrante se aplica a las aplicaciones cliente y servicios de proxy de Oracle Service Bus. La seguridad a nivel de transporte saliente se aplica a las conexiones entre los servicios de negocio y los servicios de proxy de Oracle Service Bus. Para obtener más información sobre la seguridad a nivel de transporte, consulte la sección sobre configuración de seguridad a nivel de transporte en Oracle Fusion Middleware Developer's Guide for Oracle Service Bus (Guía del Desarrollador de Oracle Fusion Middleware para Oracle Service Bus) en http://www.oracle.com/pls/as111170/lookup?id=OSBDV1557.

Protocolo
Seleccione un protocolo de transporte de la lista. Los protocolos disponibles dependen del tipo de servicio que desee crear:
  • Servicio Web WSDL: http, jca, jms, local, sb, ws
  • Servicio de Transporte según Configuración: jejb
  • Servicio de Mensajes: Correo electrónico, archivo, ftp, http, jms, local, mq (si está disponible), sftp, tuxedo
  • Servicio Cualquier SOAP: http, jms, local, sb
  • Servicio Cualquier XML: Correo electrónico, archivo, ftp, http, jms, local, mq (si está disponible), sb, sftp, tuxedo
URI de Punto Final
Introduzca un URI de punto final con el formato basado en el protocolo de transporte seleccionado en el campo Protocolo anterior. Los formatos son:
  • correo electrónico: mailfrom:nombre_host_servidor_correo:puerto_servidor_correo
  • archivo: file:///nombre_unidad:/algún_nombre
  • ftp: ftp://nombre_host:puerto/directorio
  • http: algúnNombre

    • El transporte de HTTP soporta puntos finales de HTTP y HTTPS.

  • jca: jca://<jndi_adaptador_recurso>
  • jejb: El formato de URL es nombre_jndi_ejb. El URI configurado para un servicio de proxy de JEJB se convierte en el nombre de JNDI global para buscar el bean de sesión sin estado generado por el transporte de JEJB desde la interfaz remota/de negocio del JAR de cliente.

    • Nota: Para EJB 3.0, nombre_jndi_ejb es el atributo mappedName de la anotación @javax.ejb.Stateless del bean generado. El nombre de la búsqueda de JNDI para el servicio de EJB generado tiene el sufijo #interface_class, que es el nombre totalmente cualificado de la interfaz de negocio.

    Puede acceder al servicio de proxy de JEJB como:

    EJB 2.1: protocolo://host:puerto/nombre_jndi_ejb

    EJB 3.0: protocolo://host:puerto/nombre_jndi_ejb#interface_class

    Nota: Si sus EJB se están ejecutando en IBM WebSphere, nombre_jndi_ejb debe tener el siguiente formato:

    cell/nodes/nombre_nodo/servers/nombre_servidor/nombre_jndi_ejb

    o bien

    cell/clusters/nombre_cluster/nombre_jndi_ejb

    como se describe en la documentación de IBM WebSphere en http://publib.boulder.ibm.com/infocenter/wasinfo/v6r1/index.jsp?topic=/com.ibm.websphere.nd.iseries.doc/info/iseriesnd/ae/rnam_example_prop3.html.

  • jms: jms://host:puerto/nombre_jndi_fábrica/nombre_jndi_destino

    • Para que el destino de JMS sean varios servidores, utilice el siguiente formato de URI:

    jms://host1:puerto,host2:puerto/fábrica_conexiones_cola/nombre_destino

    Tenga en cuenta que al crear un servicio de proxy, puede configurar un URI de punto final de JMS incluso si el servidor de dicho punto final no está disponible. Sin embargo, en el caso de JMS, al activar la sesión, el punto final debe estar disponible. Para obtener más información, consulte Activación de Sesiones.

    Nota: aunque Oracle WebLogic Server permite el uso de barras inclinadas en los nombres de JNDI (por ejemplo, "miscolas/micola"), los nombres de JNDI con barras inclinadas interfieren en el formato de URI que necesita Oracle Service Bus, por lo que no se pueden utilizar esos nombres. Para solucionar este problema, defina un servidor ajeno de JMS y haga referencia a él en el URI. Para obtener más información, consulte la sección sobre configuración de servidores ajenos en Oracle Fusion Middleware Oracle WebLogic Server Administration Console Online Help (Ayuda en Pantalla de la Consola de Administración de Oracle Fusion Middleware Oracle WebLogic Server), en http://www.oracle.com/pls/as111170/lookup?id=WLACH.

    En un cluster: Los nombres de host del URI de JMS deben coincidir con de los servidores del cluster configurados en WebLogic Server.

  • local

    • Este transporte no necesita ningún URI de punto final.

  • mq: mq://<nombre_cola_local>?conn=<referencia_recurso_conexión_mq>

    • nombre_cola_local es el nombre de la cola de MQ desde la que el servicio de proxy lee los mensajes.

    referencia_recurso_conexión_mq es la ruta de acceso (proyecto/carpeta) y el nombre del recurso de conexión de MQ; por ejemplo, default/my_MQconnection.

    Nota: El URI de punto final no puede contener espacios, por lo tanto, no cree recursos de conexión de MQ o proyectos/carpetas con espacios en los nombres.

    Para que el transporte de MQ esté disponible en Oracle Service Bus, consulte Conexiones de MQ.

URI de Punto Final (continuación)
  • sb: nombre_servicio

    • nombre_servicio es el identificador único para el servicio de proxy. Por defecto, este nombre será el nombre del servicio de proxy.

    nombre_servicio sólo debe contener caracteres permitidos en los URI (como se describe en RFC2396 en http://www.ietf.org/rfc/rfc2396.txt), y no puede contener caracteres de barra inclinada (/) ni dos puntos (:).

  • sftp: sftp://nombre_host:puerto/directorio
  • tuxedo: nombre_exportación

    • El nombre_exportación del URI corresponde a una exportación de WTC que el dominio de Tuxedo remoto identifica como un servicio de Tuxedo.

    Si se especifica más de un URI, debe tener nombres de recursos únicos para los puntos finales. Si no se especifica ningún nombre remoto, su valor es el valor del nombre del recurso. Si no se introduce ningún nombre remoto o si el nombre remoto y de recurso es el mismo, sólo se permite un URI. En este caso, el nombre de recurso y remoto tienen el mismo valor. Esto permite a las importaciones de WTC ya definidas utilizar el equilibrio de carga y failover de WTC.

    Nota: Si configura dos URI idénticos, un error indica que el nombre de servicio ya existe.

  • ws: /ruta_acceso_contexto

    • ruta_acceso_contexto debe ser única para servicios de proxy que utilizan el transporte HTTP o WS.

Obtener Todas las Cabeceras
Seleccione Sí para recuperar todas las cabeceras del transporte.
Seleccione No para recuperar un juego definido de cabeceras. Si selecciona No, introduzca un juego de cabeceras en el campo Cabecera y, a continuación, haga clic en Agregar. (Este paso no se aplica al transporte local.)
Nota: Oracle Service Bus no transfiere la cabecera de autorización HTTP de la solicitud al pipeline porque abre una vulnerabilidad de seguridad. Puede crear sin querer una acción de log que escriba el nombre de usuario y contraseña descifrada en un archivo log. Si el patrón de diseño necesita que la cabecera de autorización HTTP esté en el pipeline, realice lo siguiente:

a. En el comando de inicio de Oracle Service Bus, defina la siguiente propiedad del sistema en true: com.bea.wli.sb.transports.http.GetHttpAuthorizationHeaderAllowed

b. En la consola de administración de Oracle Service Bus, en la página Configuración de Transporte, seleccione Obtener Todas las Cabeceras o seleccione Cabeceras Especificadas por el Usuario y especifique la autorización.
c. Reinicie Oracle Service Bus.

Oracle Service Bus transferirá la cabecera de autorización al pipeline.
