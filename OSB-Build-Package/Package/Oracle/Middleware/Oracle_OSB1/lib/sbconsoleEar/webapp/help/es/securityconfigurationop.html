Descripción de Usuarios, Grupos, Roles de Seguridad y Políticas

Esta sección incluye los siguientes temas:

  • Usuarios
  • Grupos
  • Roles
  • Políticas de Control de Acceso
  • Sesiones y Datos de Configuración de Seguridad

Usuarios

Los usuarios son entidades que se pueden autenticar. Un usuario puede ser una persona o una entidad de software, como por ejemplo un cliente de servicios web. Debe dotar a cada usuario de una identidad única (nombre) dentro del dominio de seguridad.

Por lo general, los usuarios que creará se clasificarán en dos categorías:

  • Usuarios cliente con capacidad de acceso a los servicios de proxy o de negocio.

    • Si crea un elevado número de usuarios cliente, piense en la posibilidad de organizarlos en grupos de seguridad.

  • Usuarios administrativos que puedan usar la consola de administración de Oracle Service Bus para crear o modificar servicios de proxy, servicios de negocio y otros recursos de Oracle Service Bus.

    • Oracle Service Bus utiliza un sistema de seguridad basado en roles para sus funciones administrativas. En lugar de proporcionar privilegios de acceso directamente a los usuarios, Oracle Service Bus otorga privilegios administrativos sólo a los roles de seguridad. Para proporcionar privilegios administrativos a un usuario, deberá incluir al usuario en uno de los grupos de seguridad por defecto que esté en uno de los roles de seguridad predefinidos.

Grupos

Para facilitar las tareas de administración de un número elevado de usuarios, puede organizar a los usuarios por grupos, cada grupo con su nombre propio. Después, en lugar de otorgar privilegios de acceso o identidades de roles a cada usuario por individual, podrá proporcionar privilegios o identidades a grupos completos.

Grupos de Seguridad Administrativos

Oracle Service Bus proporciona grupos de seguridad por defecto para que sea más fácil dar a los usuarios acceso a funciones administrativas como, por ejemplo, crear servicios de proxy. Cada grupo se encuadra dentro de un rol de seguridad de Oracle Service Bus predefinido con privilegios administrativos.

Para obtener más información, consulte la sección sobre configuración de la seguridad administrativa en Oracle Fusion Middleware Developer's Guide for Oracle Service Bus (Guía del Desarrollador de Oracle Fusion Middleware para Oracle Service Bus) en http://www.oracle.com/pls/as111170/lookup?id=OSBDV1539.

Roles

Los roles de seguridad son identidades que se pueden otorgar a un usuario o a un grupo según condiciones del entorno de tiempo de ejecución. Cuando se crean políticas de control de acceso, éstas se pueden otorgar a un rol, a un grupo o a un usuario.

Por ejemplo, supongamos que crea dos grupos: "MisClientesEste" y "MisClientesOeste". Digamos que crea un rol de seguridad llamado "ClienteConPrivilegios" y después crea condiciones de modo que el grupo MisClientesOeste desempeñe el rol de 8 a.m. a 8 p.m. hora WT, y el grupo MisClientesEste lo haga de 8 p.m. a 8 a.m. EST. A continuación, crea una política de control de acceso para un servicio de proxy que proporciona acceso con rol ClienteConPrivilegios al servicio. Diferentes usuarios tendrán acceso a diferentes horas dependiendo de si se encuentran en el grupo MisClientesEste o MisClientesOeste.

Roles de Seguridad Administrativos

Oracle Service Bus proporciona cuatro roles de seguridad predefinidos (además de cuatro roles predefinidos de WebLogic Server) que otorgan privilegios administrativos. Si bien los privilegios de acceso de los roles de seguridad administrativos de Oracle Service Bus no se pueden cambiar, sí se pueden cambiar las condiciones aplicables a un usuario o a un grupo dentro de unos de los roles.

Para obtener más información acerca de estos roles y los privilegios disponibles para cada rol, consulte la sección sobre configuración de la seguridad administrativa en Oracle Fusion Middleware Developer's Guide for Oracle Service Bus (Guía del Desarrollador de Oracle Fusion Middleware para Oracle Service Bus) en http://www.oracle.com/pls/as111170/lookup?id=OSBDV1539.

Políticas de Control de Acceso

Las políticas de control de acceso especifican las condiciones bajo las cuales usuarios, grupos o roles pueden acceder a un servicio de proxy. Por ejemplo, puede crear una política que permita a los usuarios en el rol ClienteOro acceder siempre a un servicio de proxy y que permita a los usuarios en el rol ClientePlata acceder al servicio de proxy sólo las noches de la semana a partir de las 12 p.m.

Para todos los servicios de proxy, puede crear una política de nivel de transporte, que aplique una comprobación de seguridad cuando un cliente intenta establecer una conexión con el servicio de proxy. Sólo se permite procesar solicitudes de usuarios que se muestran en la política de nivel de transporte.

Una política de control de acceso de nivel de mensaje aplica una comprobación de seguridad cuando un cliente intenta llamar a un servicio de proxy con seguridad a nivel de mensaje. Puede crear una política de control de acceso de nivel de mensaje en los siguientes casos:

  • Para servicios de proxy que son intermediarios de seguridad de servicios web activos.
  • Para servicios de proxy que tienen sistema de autenticación personalizado en el nivel de mensajes.

Sólo los usuarios que se muestran en la política de nivel de mensaje pueden llamar a la operación.

Sesiones y Datos de Configuración de Seguridad

Los usuarios, los grupos y los roles se conservan en los proveedores de seguridad, los cuales no gobiernan las sesiones de Oracle Service Bus. Así pues, podrá crear o modificar estos datos estando dentro de una sesión o fuera. Las adiciones y modificaciones de estos datos surtirán efecto de forma inmediata y quedarán disponibles para todas las sesiones. Si desecha una sesión en la que ha agregado o ha modificado datos, los datos de seguridad no se desecharán.

Las políticas de control de acceso las conservan los proveedores de autorización. Y ahora hay una referencia a ellas en el repositorio de Oracle Service Bus.

Las políticas de control de acceso se gestionan desde dentro de una sesión de diseño de Oracle Service Bus y no desde fuera de la sesión. Dado que los cambios se hacen desde dentro de una sesión, podrá omitir o desechar estos cambios igual que con cualquier otro recurso.

Aunque las ACL se pueden gestionar desde la consola de administración de Oracle Service Bus, es posible cambiar las políticas desde fuera de Oracle Service Bus. No obstante, si se cambian políticas desde fuera de Oracle Service Bus, es posible que las referencias en Oracle Service Bus queden obsoletas y sin validez.

Así pues, para poder gestionarlas de forma coherente, bien íntegramente desde fuera de sesiones de Oracle Service Bus (con los MBean de proveedor de autorización o con herramientas de proveedor de autorización de terceros) o gestionarlas íntegramente desde dentro de sesiones de Oracle Service Bus. Si se combinan estos dos acercamientos, sea cual sea la combinación, podría obtenerse una vista incoherente de las políticas.
