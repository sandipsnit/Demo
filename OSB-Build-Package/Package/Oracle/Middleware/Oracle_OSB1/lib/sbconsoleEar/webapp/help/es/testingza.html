Documento de Solicitud
Los campos de entrada generan el mensaje de solicitud que se envía al servicio de negocio. Haga clic en Ejecutar para ejecutar la prueba con los valores introducidos. La consola de prueba muestra el mensaje de solicitud y el mensaje de respuesta del servicio.
La entrada de la sección Documento de Solicitud es específica del tipo de servicio. En las siguientes secciones se enumeran los tipos de servicio y la descripción de la entrada solicitada para cada uno.
Cualquier XML
El servicio requiere una entrada en forma de carga útil. La carga útil es el contenido del mensaje que se envía. Se espera que el contenido sea un documento XML. Puede buscar un archivo o introducir el contenido del mensaje en el cuadro de texto.
Cualquier SOAP
El servicio requiere una entrada en forma de carga útil. La carga útil es el contenido del mensaje que se envía. Se espera que el contenido sea el elemento Envelope de SOAP. Puede buscar un archivo o introducir el contenido del mensaje en el cuadro de texto.
Documento de Solicitud (Continuación)
Mensajes
Los servicios de mensajes definen cuatro posibles tipos de entradas: Ninguno, XML, Binario o Texto. El servicio requiere una sola entrada basada en archivo o basada en texto, excepto para el tipo Ninguno, que no requiere ninguna entrada.
Oracle recomienda que se introduzca una entrada binaria desde un archivo. Los datos introducidos en el área de texto se convierten a binarios mediante la codificación del sistema.
Los datos introducidos desde archivos para servicios de texto deben convertirse a texto. El campo de entrada de codificación se utiliza para especificar la codificación que se aplica durante la conversión. La codificación del sistema se utiliza si este campo no está configurado. Por defecto, el campo de codificación se inicializa con el valor de codificación configurado en el punto final del servicio de proxy.
Documento de Solicitud (Continuación)
XML
El servicio requiere una entrada en forma de carga útil. La carga útil es el contenido del mensaje que se envía. Se espera que el contenido sea un documento XML. Puede buscar un archivo o introducir el contenido del mensaje en el cuadro de texto.
Se trata de un servicio basado en WSDL con varias operaciones definidas. Oracle proporciona un documento de ejemplo que puede utilizar para probar este servicio. Todas las operaciones se enumeran en la parte superior de la página, con una flecha que resalta la operación seleccionada.
Documento SOAP
Para un documento de SOAP, el elemento Envelope de SOAP se compone generalmente de una carga útil de un cuerpo y cero o más cabeceras.
Los separadores Pantalla y XML ofrecen métodos alternativos para especificar el contenido.
El separador Pantalla contiene un campo Cabecera SOAP y un campo Cuerpo SOAP. El contenido del campo Cabecera SOAP se espera que sea una etiqueta de cabecera SOAP (esto permite definir más de una cabecera). El campo Cuerpo SOAP contiene los datos actuales que se envían como parte del mensaje. Se espera que el contenido sea un documento XML. Tanto la cabecera como el cuerpo se utilizan para generar elementos Envelope de SOAP.
El separador XML contiene un solo campo de entrada. Se espera que el contenido de este campo sea el elemento Envelope de SOAP que se envía.
Documento de Solicitud (Continuación)
RPC de SOAP
Para RPC de SOAP, el elemento Envelope de SOAP se compone de cero o más cabeceras y de cero o más argumentos.
Los separadores Pantalla y XML ofrecen métodos alternativos para especificar el contenido.
El separador Pantalla contiene una sola entrada para cabeceras SOAP y un campo de entrada para cada argumento (el nombre del campo de entrada corresponde al nombre del argumento). Se espera que el contenido del campo Cabecera SOAP sea una etiqueta de cabecera SOAP (esto permite definir más de una cabecera).
El WSDL se utiliza para detectar el tipo de cada argumento. Se utiliza un campo de entrada de una sola línea para tipos primitivos y un campo de entrada de varias líneas para tipos XML. Automáticamente se genera un documento de ejemplo para facilitar la prueba.
La consola de prueba utiliza las cabeceras y argumentos para generar el elemento Envelope de SOAP.
El separador XML contiene un solo campo de entrada. Se espera que el contenido de este campo sea el elemento Envelope de SOAP que se envía. La carga útil (entrada XML) puede basarse en archivos o en texto. Si se hace referencia a un archivo, éste tendrá preferencia sobre la entrada textual. Busque y seleccione el archivo que desee utilizar en la prueba.
Seguridad de Servicio Web
Esta sección sólo está disponible para servicios SOAP si la operación seleccionada tiene una política de Seguridad de Servicio Web.
Proveedor de Servicios
El servicio de prueba obtiene todas las credenciales PKI (par de claves) de cliente para las operaciones de seguridad de servicio web (firma digital y cifrado) de este proveedor de servicios. Este campo es opcional. Determinados casos requieren un proveedor de servicios, dependiendo de la política de solicitud/respuesta de las operaciones. Para obtener más información, consulte la sección acerca de las prueba de servicios con seguridad de servicio web en Oracle Fusion Middleware Administrator's Guide for Oracle Service Bus (Guía del Administrador de Oracle Fusion Middleware para Oracle Service Bus) en http://www.oracle.com/pls/as111170/lookup?id=OSBAG384.
Seguridad de Servicio Web (Continuación)
Usuario
Nombre de usuario de los elementos de nombre de usuario de seguridad de servicio web generados a partir del servicio de prueba. Este campo es opcional. Este nombre de usuario sólo es necesario en algunos casos en los que la política de solicitud de la operación especifica una afirmación de identidad.
No confunda este campo con el campo de nombre de usuario para el contexto de seguridad de transporte. Debe ser un nombre de usuario y una contraseña válidos en el dominio de seguridad local. Un nombre de usuario no válido o contraseña no válida puede ocasionar un error de cliente en el servicio de prueba.
En algunos casos, este nombre de usuario y contraseña pueden utilizarse cuando el servicio de prueba genera una afirmación de SAML.
Contraseña
Contraseña de los elementos de nombre de usuario de seguridad de servicio web generados a partir del servicio de prueba.
Transporte
La sección Transporte está reducida por defecto. Los campos y valores dependen de la configuración de la prueba.
Autenticación
Usuario
Nombre de usuario para la definición del contexto de seguridad utilizado por el servicio de prueba al llamar al servicio de negocio. Si el servicio de negocio espera un elemento SAML, esta identidad puede propagarse en el elemento SAML. Consulte la sección sobre el uso de SAML para autenticación en Oracle Fusion Middleware Developer's Guide for Oracle Service Bus (Guía del Desarrollador de Oracle Fusion Middleware para Oracle Service Bus) en http://www.oracle.com/pls/as111170/lookup?id=OSBDV1647. Debe ser un nombre de usuario y una contraseña válidos en el dominio de seguridad local. Un nombre de usuario no válido o contraseña no válida puede ocasionar un error de cliente en el servicio de prueba.
Contraseña
Contraseña asociada. Para obtener más información, consulte Usuario.
Proveedor de Servicios
Este campo se utiliza al probar un servicio de negocio HTTPS con autenticación CLIENT-CERT. El proveedor de servicios debe tener asociada una credencial de cliente SSL. El servicio de prueba utilizará esta credencial durante el establecimiento de comunicación de SSL
Modo de Llamada
Solicitud/Respuesta
Esta opción sólo aparece cuando se prueba un servicio de negocio Cualquier SOAP o Cualquier XML. Desactive la casilla de control Solicitud/Respuesta para llamadas de servicio unidireccionales.
