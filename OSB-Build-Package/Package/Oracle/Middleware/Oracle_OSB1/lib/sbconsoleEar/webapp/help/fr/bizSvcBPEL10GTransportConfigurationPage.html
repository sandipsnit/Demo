Page BPEL 10g - Configuration du transport

Cette page permet de configurer les paramètres de transport d'un service fonctionnel à l'aide du protocole de transport BPEL 10g (Oracle BPEL Process Manager). Pour plus d'informations sur l'utilisation d'Oracle Service Bus avec Oracle BPEL Process Manager, reportez-vous à "Oracle BPEL Process Manager Transport" dans le manuel Oracle Fusion Middleware Developer's Guide for Oracle Service Bus à l'adresse suivante : http://www.oracle.com/pls/as111170/lookup?id=OSBDV1147.

Propriété
Description
Rôle
Le transport BPEL permet d'envoyer des messages de demande depuis Oracle Service Bus vers Oracle BPEL Process Manager. Le service fonctionnel peut jouer les rôles suivants :
  • Client synchrone
    Pour les communications synchrones avec un client Oracle Service Bus, la seule information d'emplacement nécessaire est l'adresse BPEL. Cette adresse est capturée de façon statique en tant qu'URI endpoint et/ou de façon dynamique via la réécriture de l'URI.
  • Client asynchrone
    Pour les communications asynchrones avec un client Oracle Service Bus, un callback est envoyé depuis Oracle BPEL Process Manager vers Oracle Service Bus sur une connexion différente de la demande. Vous devez configurer Oracle Service Bus de manière à ce qu'il fournisse l'adresse de callback correcte. Pour plus d'informations, reportez-vous à "Advanced Use Cases (Asynchronous)" dans le manuel Oracle Fusion Middleware Developer's Guide for Oracle Service Bus à l'adresse suivante : http://www.oracle.com/pls/as111170/lookup?id=OSBDV1163.
  • Callback de service
    Si le service fonctionnel est destiné à servir de callback de service vers Oracle BPEL Process Manager (Oracle BPEL Process Manager appelle un service externe via Oracle Service Bus), l'adresse de callback n'est connue qu'à l'exécution. Utilisez un URI endpoint tel que bpel://callback.

    • Si vous configurez le service fonctionnel avec l'URI marqueur, configurez la logique de pipeline de manière à ce qu'elle définisse, de façon dynamique, l'URI sur $outbound. Pour ce faire, vous pouvez, par exemple, utiliser l'action TransportHeader.

    Remarque : Un service fonctionnel de type callback de service ne prend en charge ni l'équilibrage de charge, ni le basculement en cas d'incident.
Proxy de callback
Ce champ facultatif est disponible uniquement pour le rôle Client asynchrone. Dans ce champ, vous indiquez le service proxy (proxy SB ou HTTP de type SOAP) à utiliser pour acheminer les callbacks vers le client Oracle Service Bus ayant effectué la demande. Pour plus d'informations, reportez-vous à "Advanced Use Cases (Asynchronous)" dans le manuel Oracle Fusion Middleware Developer's Guide for Oracle Service Bus à l'adresse suivante : http://www.oracle.com/pls/as111170/lookup?id=OSBDV1163.
Compte de service
Pour la sécurité du contexte JNDI, cette option permet d'accéder au service de distribution Oracle BPEL Process Manager. Cliquez sur Parcourir et sélectionnez un compte de service. Si aucun n'est spécifié, un sujet anonyme est utilisé.
Il n'existe aucune restriction concernant le type de compte de service pouvant être configuré (statique ou pass-through), mais le système doit pouvoir accéder à un nom utilisateur et un mot de passe à l'exécution.
Suspendre la transaction
Cette option permet de rendre le service fonctionnel non transactionnel, même si ce dernier est appelé par une transaction.
Lorsque vous ne sélectionnez pas l'option Suspendre la transaction, le comportement est le suivant :
  • Si le protocole est pris en charge par WebLogic Server (T3, IIOP, HTTP), la transaction est propagée.
  • Si le protocole indique un serveur OC4J (ORMI, OPMN), le transport BPEL génère une exception, car OC4J ne prend pas en charge la propagation des transactions.
Le transport BPEL ne prend pas en compte l'option Suspendre la transaction dans les situations suivantes :
  • Le service fonctionnel est appelé avec la qualité de service (QoS) Au mieux. Le transport BPEL suspend automatiquement les transactions ne prenant pas en charge la qualité de service.
  • Le service fonctionnel est appelé avec la qualité de service (QoS) Une seule fois et il n'existe aucune transaction.
Pour plus d'informations sur la propagation des transactions, reportez-vous à "Transaction Propagation" dans le manuel Oracle Fusion Middleware Developer's Guide for Oracle Service Bus à l'adresse suivante : http://www.oracle.com/pls/as111170/lookup?id=OSBDV1151.
Stratégie de répartition
Sélectionnez l'instance du gestionnaire de travaux WebLogic Server à utiliser dans la stratégie de répartition de cette adresse. Le gestionnaire de travaux par défaut est utilisé s'il n'en existe pas d'autre.
Pour plus d'informations sur les gestionnaires de travaux, reportez-vous à "Using Work Managers to Optimize Scheduled Work" dans le manuel Oracle Fusion Middleware Configuring Server Environments for Oracle WebLogic Server à l'adresse suivante : http://www.oracle.com/pls/as111170/lookup?id=CNFGD112.
