Page HTTP - Configuration du transport

Le transport HTTP prend en charge les adresses HTTP et HTTPS.

La page Créer/Modifier un service fonctionnel - HTTP - Configuration du transport est une des pages permettant de créer et modifier des services fonctionnels, comme indiqué dans Création et configuration de services fonctionnels et Modification des configurations des services fonctionnels.

Utilisez cette page pour configurer les paramètres de transport d'un service fonctionnel utilisant le protocole de transport HTTP. Le tableau 18-10 explique comment utiliser cette page.

Temporisation de la lecture
Indiquez l'intervalle de temporisation de la lecture, en secondes.
La valeur zéro (0) signifie qu'il n'y a jamais temporisation.
Temporisation de la connexion
Indiquez l'intervalle de temporisation de la connexion, en secondes. Si le délai de temporisation est dépassé avant qu'une connexion ait pu être établie, Oracle Service Bus génère une erreur de connexion.
La valeur zéro (0) signifie qu'il n'y a jamais temporisation.
Méthode de demande HTTP
Ce paramètre permet d'utiliser une des méthodes HTTP suivantes dans une demande :
  • POST : cette méthode transmet toutes les données, d'une longueur illimitée, directement à la connexion de socket comme faisant partie du corps de la demande HTTP. L'échange est invisible pour le client et l'URL n'est pas modifiée. Pour les demandes REST, la méthode POST indique au transport d'effectuer une opération de création/remplacement ou d'exécuter une action pour la demande.
  • GET : avec cette méthode, vous pouvez inclure certaines informations descriptives propres à la demande dans cette dernière. Ces informations sont transmises sous la forme d'une suite de caractères ajoutée à la fin de l'URL de la demande dans une chaîne d'interrogation. Vous pouvez utiliser la méthode GET dans un service fonctionnel de type N'importe quel service XML ou Service de messagerie lorsque le type de message de demande est défini sur Aucun. Pour les demandes REST, la méthode GET extrait une représentation d'une ressource distante.
  • PUT : vous pouvez utiliser la méthode PUT dans un service fonctionnel de type N'importe quel service XML ou Service de messagerie. La méthode PUT indique au transport d'effectuer une opération de création/remplacement pour une demande REST, comme télécharger un fichier vers un emplacement donné, par exemple.
  • HEAD : vous pouvez utiliser la méthode HEAD dans un service fonctionnel de type N'importe quel service XML ou Service de messagerie lorsque le type de message de réponse est défini sur Aucun. La méthode HEAD indique au transport d'obtenir les informations d'en-tête d'une ressource distante plutôt que d'obtenir une représentation complète de la ressource pour une demande REST.
  • DELETE : vous pouvez utiliser la méthode DELETE dans un service fonctionnel de type N'importe quel service XML ou Service de messagerie. La méthode DELETE indique au transport d'effectuer une opération de suppression pour une demande REST.
Remarque : si une méthode est définie dans la variable $outbound/transport/request/http:http-method, la valeur indiquée est prioritaire par rapport aux autres méthodes sélectionnées pour Méthode de demande HTTP.
Authentification
Sélectionnez l'une des options suivantes :
  • Aucun : indique qu'aucune authentification n'est nécessaire pour accéder à ce service.
  • De base : indique qu'une authentification de base est nécessaire pour accéder à ce service.

    • Lorsque l'authentification de base est activée, WebLogic Server authentifie le client à l'aide d'un nom utilisateur et d'un mot de passe auprès des fournisseurs d'authentification configurés dans le domaine de sécurité, comme un service d'annuaire LDAP (Lightweight Directory Access Protocol) et Windows Active Directory. Le client doit envoyer ses nom utilisateur et mot de passe dans l'en-tête de demande HTTP.

    Il est fortement déconseillé d'utiliser l'authentification de base sur HTTP, car les mots de passe sont envoyés en clair. Vous pouvez en revanche envoyer des mots de passe via HTTPS en toute sécurité, car ce protocole présente un canal crypté.

    Avertissement : par défaut, tous les utilisateurs (autorisés et anonymes) peuvent accéder à un service fonctionnel. Pour limiter l'accès à ce dernier, vous devez créer une stratégie d'autorisation basée sur le transport. Reportez-vous à Modification des stratégies d'accès au niveau du transport.

  • Certificat client : indique qu'une communication cryptée et une authentification client forte (SSL bidirectionnelle) sont utilisées. Pour plus d'informations, reportez-vous à "Configuring Transport-Level Security" dans le manuel Oracle Fusion Middleware Developer's Guide for Oracle Service Bus à l'adresse suivante : http://www.oracle.com/pls/as111170/lookup?id=OSBDV1557.
Compte de service
Indiquez un compte de service. Un compte de service représente la ressource d'alias d'un nom utilisateur et d'un mot de passe. Ce champ est obligatoire si vous sélectionnez Authentification de base obligatoire.

Stratégie de répartition
Sélectionnez l'instance du gestionnaire de travaux WebLogic Server à utiliser dans la stratégie de répartition de cette adresse. Le gestionnaire de travaux par défaut est utilisé s'il n'en existe pas d'autre. Pour plus d'informations sur les gestionnaires de travaux, reportez-vous à "Using Work Managers to Optimize Scheduled Work" dans le manuel Oracle Fusion Middleware Configuring Server Environments for Oracle WebLogic Server à l'adresse suivante : http://www.oracle.com/pls/as111170/lookup?id=CNFGD112.
Encodage de la demande
Acceptez l'encodage de jeu de caractères par défaut, ISO-8859-1, pour les demandes dans les transports HTTP ou indiquez un encodage de jeu de caractères différent.
Encodage de la réponse
Acceptez l'encodage de jeu de caractères par défaut, ISO-8859-1, pour les réponses dans les transports HTTP ou indiquez un encodage de jeu de caractères différent.
Serveur proxy
Indiquez une ressource de serveur proxy ou cliquez sur Parcourir pour sélectionner une entrée dans la liste des ressources de serveur proxy configurées.
Suivre les réacheminements HTTP
Lorsque vous cochez cette case, les réacheminements HTTP (demandes dont le code de réponse est 3xx) sont suivis de façon automatique. Un réacheminement a lieu lorsque vous envoyez une demande sortante à l'URL d'un service fonctionnel et que ce dernier renvoie un code de réponse (par exemple, 302) signalant que l'URL n'est plus valide et que la demande doit être envoyée à une autre URL.
Lorsque vous cochez la case Suivre les réacheminements HTTP, Oracle Service Bus renvoie automatiquement la demande à la nouvelle URL sans aucune intervention de votre part. Si vous ne souhaitez pas que les réacheminements HTTP soit suivis de façon automatique, désélectionnez cette case.
Utiliser le mode continu par tranches de mémoire
Sélectionnez cette option si vous souhaitez utiliser l'encodage de transfert par tranches de mémoire HTTP lorsque vous envoyez des messages.
Remarque : n'utilisez pas le mode continu par tranches de mémoire si vous sélectionnez l'option Suivre les réacheminements HTTP. Les réacheminements et l'authentification ne peuvent pas être gérés de façon automatique en mode continu par tranches de mémoire.
Affectation de session
Sélectionnez cette option pour utiliser l'affectation de session (également appelée affinité de session) pour les demandes HTTP du service fonctionnel.
Nom d'ID de session affecté
Si l'option Affectation de session est activée, entrez un nom d'identification pour la session.
