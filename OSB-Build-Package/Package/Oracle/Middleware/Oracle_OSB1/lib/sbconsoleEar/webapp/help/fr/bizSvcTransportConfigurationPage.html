Page Configuration du transport

La page Créer/Modifier un service fonctionnel - Configuration du transport est l'une des pages destinées à la création et à la modification de services fonctionnels, décrites dans les rubriques Création et configuration de services fonctionnels et Modification des configurations des services fonctionnels.

Cette page permet de sélectionner un protocole de transport pour le service fonctionnel et de définir d'autres paramètres généraux de configuration du transport.

La sécurité de niveau transport sortant s'applique aux connexions entre les services fonctionnels et les services proxy d'Oracle Service Bus. Pour plus d'informations sur la sécurité de niveau transport, reportez-vous à Configuring Transport-Level Security dans le manuel Oracle Fusion Middleware Developer's Guide for Oracle Service Bus, à l'adresse suivante : http://www.oracle.com/pls/as111170/lookup?id=OSBDV1557.

Protocole
Sélectionnez un protocole de transport dans la liste. Les protocoles disponibles varient selon le type de service que vous créez.
  • Service Web WSDL : soa-direct, bpel-10g, dsp, http, jca, jms, sb, ws
  • Service doté d'un type de transport : ejb, flow, jejb
  • Service de messagerie : email, file, ftp, http, jms, mq (si disponible), sftp, tuxedo
  • N'importe quel service SOAP : dsp, http, jms, sb
  • N'importe quel service XML : dsp, email, file, ftp, http, jms, mq (si disponible), sb, sftp, tuxedo
Algorithme d'équilibrage de charge
Sélectionnez l'un des algorithmes d'équilibrage de charge suivants :
  • Tourniquet : cet algorithme classe de manière dynamique les URL que vous entrez dans le champ URI endpoint pour ce service fonctionnel. Si la première ne fonctionne pas, il essaie la suivante et ainsi de suite jusqu'à ce que le nombre de tentatives soit épuisé.

    • A chaque nouveau message, un nouveau classement des URL est effectué.

  • Aléatoire : cet algorithme classe de manière aléatoire la liste des URL que vous entrez dans le champ URI endpoint pour ce service fonctionnel. Si la première ne fonctionne pas, il essaie la suivante et ainsi de suite jusqu'à ce que le nombre de tentatives soit épuisé.
  • Pondéré-aléatoire : cet algorithme classe de manière aléatoire la liste des URL que vous entrez dans le champ URI endpoint pour ce service fonctionnel, mais certaines sont réessayées plus que d'autres, selon la valeur que vous entrez dans le champ Pondération.
  • Aucun : cet algorithme classe la liste des URL que vous entrez dans le champ URI endpoint pour ce service fonctionnel de haut en bas.
URI endpoint
Entrez une URL endpoint au format basé sur le protocole de transport sélectionné dans le champ Protocole. Vous trouverez ci-dessous les descriptions des formats URI de chaque transport. Les éléments URI facultatifs sont affichés entre crochets []. Pour plus d'informations sur les URI de transport, reportez-vous aux chapitres sur les transports correspondants dans le manuel Oracle Fusion Middleware Developer's Guide for Oracle Service Bus, à l'adresse suivante : http://www.oracle.com/pls/as111170/lookup?id=OSBDV.
  • bpel-10g - protocol://host[:port] [/protocol-path]/domain/process[/version[/partnerlink/role]]
  • dsp - t3://dsp-ip-address:port/dsp-app-name
  • ejb - ejb:jndi_provider_name:ejb_jndi_name

    • Dans l'URI, jndi_provider_name est le nom de la ressource du fournisseur JNDI et ejb_jndi_name, le nom JNDI sur le serveur JNDI pour EJB.

    Si le fournisseur JNDI est situé sur le même serveur, il n'a pas besoin d'être indiqué. L'URI sera alors ejb::ejb_jndi_name.

    Remarque : si vos EJB sont exécutés sur IBM WebSphere, ejb_jndi_name doit avoir le format suivant :

    cell/nodes/node_name/servers/server_name/ejb_jndi_name

    ou

    cell/clusters/cluster_name/ejb_jndi_name

    comme décrit dans la documentation IBM WebSphere, à l'adresse suivante : http://publib.boulder.ibm.com/infocenter/wasinfo/v6r1/index.jsp?topic=/com.ibm.websphere.nd.iseries.doc/info/iseriesnd/ae/rnam_example_prop3.html.

  • adresse électronique : vous pouvez entrer plusieurs URI endpoint en combinant tous les formats suivants :
    • mailto:name@domain_name.com
    • mailto:name@domain_name.com?smtp=smtp_server_resource
    • mailto:name@domain_name.com?mailsession=jndi_mail_session
  • file - file:///drivename:/somename
  • flow - flow:<reference path to split-join resource>

    • reference path to split-join resource est le chemin d'accès (project/folder) et le nom de l'opération Fractionnement-Fusion ; par exemple, batchorderProcessing/batchorder.

  • ftp - ftp://host:port/directory
  • http - http://host:port/someService

    • Le transport HTTP prend en charge les adresses HTTP et HTTPS.

  • jca - jca://<resource_adapter_jndi>
  • jejb - jejb:jndi_provider_name:ejb_jndi_name

    • Exemple EJB 2.1 : jejb:myProvider:osb.jejb.myJejbBiz21

    Exemple EJB 3.0 : jejb:myProvider:myBiz31#osb.jejb.myJejbBiz

    Remarque : si vos EJB sont exécutés sur IBM WebSphere, ejb_jndi_name doit avoir le format suivant :

    cell/nodes/node_name/servers/server_name/ejb_jndi_name

    ou

    cell/clusters/cluster_name/ejb_jndi_name

    comme décrit dans la documentation IBM WebSphere, à l'adresse suivante : http://publib.boulder.ibm.com/infocenter/wasinfo/v6r1/index.jsp?topic=/com.ibm.websphere.nd.iseries.doc/info/iseriesnd/ae/rnam_example_prop3.html.

  • jms - jms://host:port/factoryJndiName/destJndiName

    • Pour envoyer une destination JMS vers plusieurs serveurs, utilisez le format d'URI suivant :
    jms://host1:port,host2:port/QueueConnectionFactory/DestName

    Remarque : lorsque Oracle WebLogic Server autorise les barres obliques dans les noms JNDI, tels que "myqueues/myqueue", les noms JNDI avec des barres obliques interfèrent avec le format URI requis par Oracle Service Bus, et vous ne pouvez pas utiliser ces noms. Pour passer outre ce problème, définissez un serveur étranger JMS et référencez-le dans l'URI. Pour plus d'informations, reportez-vous à Configurer les serveurs étrangers dans l'aide en ligne d'Oracle Fusion Middleware Oracle WebLogic Server Administration Console Online Help à l'adresse http://www.oracle.com/pls/as111170/lookup?id=WLACH.

    Dans un cluster : dans l'URI JMS, les noms d'hôte doivent correspondre exactement aux noms d'hôte des serveurs de cluster tels qu'ils sont configurés sur le serveur WebLogic.

URI endpoint (suite)
  • mq - mq://<local-queue-name>?conn=<mq-connection-resource-ref>

    • local-queue-name est le nom de la file d'attente MQ à partir de laquelle le service fonctionnel lit les messages.

    mq-connection-resource-ref est le chemin (project/folder) et le nom de la ressource de connexion MQ ; par exemple, default/my_MQconnection.

    Remarque : L'URI endpoint ne doit pas contenir d'espaces. Par conséquent, ne créez pas de ressources de connexion MQ ou de projets/dossiers contenant des espaces.

    Pour accéder au transport MQ disponible dans Oracle Service Bus, reportez-vous à Connexions MQ.

  • sb - sb://[<jndi_provider_name/>]service_name

    • jndi_provider_name (facultatif) est le nom de la ressource du fournisseur JNDI d'Oracle Service Bus. S'il n'est pas indiqué, le contexte par défaut est utilisé.

    service_name est un service cible et correspond à l'URI du service proxy distant.

  • sftp - sftp://hostname:port/directory
  • soa-direct - [protocol://authority]/default/compositeName[!versionNumber[*label]]/serviceName

    • Le protocole et l'autorité sont facultatifs lorsque les services SOA sont situés sur le même serveur qu'Oracle Service Bus.

  • tuxedo - tuxedo:resourcename[/remotename]

    • Dans l'URI, resourcename correspond à un nom d'import WTC et remotename, au nom du service exporté par le domaine Tuxedo distant. Le nom resourcename de l'URI est obligatoire alors que le nom remotename est facultatif.

    Lorsque plusieurs URI sont indiqués, les adresses doivent avoir des noms de ressource uniques. Lorsqu'aucun nom distant n'est indiqué, il prend pour valeur le nom de ressource. Lorsqu'aucun nom distant n'est entré ou lorsque les noms de ressource et distant sont identiques, il n'est possible d'indiquer qu'un seul URI. Dans ce cas, les noms de ressource et distant ont la même valeur. Cela permet aux imports WTC déjà définis d'utiliser l'équilibrage de charge et le basculement WTC.

  • ws - http://host:port/somename or https://host:port/somename
Cliquez sur Ajouter pour ajouter au moins un URI. Au moment de l'exécution, les URL sont sélectionnées selon l'algorithme d'équilibrage de charge sélectionné dans le champ Algorithme d'équilibrage de charge.
Si vous avez choisi Pondéré-aléatoire dans le champ Algorithme d'équilibrage de charge, vous pouvez également entrer une pondération dans le champ URI endpoint. La valeur par défaut est 1.
Lorsque plusieurs adresses sont définies et que vous avez sélectionné Aucun dans le champ Algorithme d'équilibrage de charge, l'ordre des adresses est important. Vous pouvez réorganiser vos adresses en utilisant les flèches Haut et Bas de la colonne Options.
Oracle Service Bus ne prend pas en charge les URI endpoint en double dans le même service fonctionnel.
Nombre de tentatives
En cas d'échec de la distribution lors de l'envoi de demandes sortantes, indiquez le nombre de tentatives supplémentaires pour chaque adresse URL, c'est-à-dire le nombre de tentatives de basculement en cas d'incident.
Par exemple, un service fonctionnel a un URI (U1) configuré et le nombre de tentatives est fixé à 3. En cas d'échec d'U1 à la première tentative, le système réessaie l'adresse U1 à trois reprises.
Lorsqu'un service fonctionnel a 2 URI (U1 et U2) configurés et un nombre de tentatives de 3, en cas d'échec de la première tentative (par exemple, avec U1), le système essaie (bascule vers) l'URI suivant (U2). En cas de nouvel échec, le système fait deux autres tentatives, une avec U1 et une avec U2.
Intervalle entre les tentatives
Indiquez le nombre de secondes durant lesquelles le système s'arrête avant de réitérer tous les URI endpoint de la liste.
Par exemple, un service fonctionnel a 2 URI configurés (U1 et U2) et un nombre de tentatives de 2 avec un intervalle d'itération de 5 secondes. En cas d'échec de la première tentative (avec U1), le système essaie immédiatement U2. En cas de nouvel échec, le système attend 5 secondes avant de réessayer U1.
Réessayer les erreurs d'application
Sélectionnez Oui ou Non.
En cas d'échec de la distribution lors de l'envoi de demandes sortantes, indiquez si les URI endpoint doivent être réessayés selon les erreurs d'application (par exemple, une erreur SOAP).
Ce champ est disponible uniquement pour les transports suivants : HTTP, EJB, JMS, DSP, Tuxedo, SB et WS.
