Configuration de paramètres opérationnels pour les services fonctionnels

Utilisez la page Paramètres opérationnels pour activer et désactiver les services fonctionnels ainsi que leur surveillance. Vous pouvez également spécifier ces mêmes paramètres opérationnels sur la page Opérations > Recherche intelligente. Cette deuxième méthode ne permet cependant pas de définir un intervalle d'agrégation ou d'indiquer un niveau de gravité d'alerte. Pour plus d'informations, reportez-vous à Recherche et mise à jour des paramètres opérationnels.

L'effet à l'exécution des paramètres définis au niveau du service dépend des paramètres globaux associés. Vous devez activer à la fois les paramètres globaux et ceux définis au niveau du service pour que ce dernier soit activé entièrement à l'exécution. L'état du service doit également être activé. Reportez-vous à Activation des paramètres globaux.

Vous pouvez modifier et enregistrer les paramètres de configuration de la surveillance même si le service n'est pas activé à l'exécution. Par exemple, vous pouvez modifier et enregistrer le paramètre Intervalle d'agrégation même si l'option Surveillance des services est désactivée. Vous pouvez ainsi modifier des paramètres et les activer plus tard.

  1. Si vous ne l'avez pas déjà fait, cliquez sur Créer afin de créer une session ou cliquez sur Modifier pour accéder à une session existante. Reportez-vous à Utilisation du centre des modifications.
  2. Sélectionnez Navigateur Web de ressources > Services fonctionnels.
  3. Cliquez sur le nom du service fonctionnel.

    4. La page Visualiser un service fonctionnel affiche les informations présentées dans le tableau 25-8.

    Tableau 25-8 Informations sur le service fonctionnel 
    Propriété
    Description
    Dernière modification par
    Utilisateur ayant créé ou modifié le service, ou l'ayant importé dans la configuration.
    Dernière modification le
    Date et heure à laquelle l'utilisateur a créé ou modifié le service, ou l'a importé dans la configuration. Cliquez sur le lien de date et d'heure pour visualiser l'historique des modifications apportées à cette ressource. Reportez-vous à Page Visualiser l'historique des modifications.
    Références
    Nombre d'objets référencés par le service. Lorsque des références existent, cliquez sur le lien numérique pour visualiser la liste des objets en question. Reportez-vous à Visualisation des références aux ressources.
    Référencé par
    Nombre d'objets référençant le service. Lorsque des références existent, cliquez sur le lien numérique pour visualiser la liste des objets en question. Reportez-vous à Visualisation des références aux ressources.
    Description
    Description du service, le cas échéant.

  4. Sélectionnez l'onglet Paramètres opérationnels.

    5. Vous devez être dans une session active pour configurer les paramètres opérationnels des services fonctionnels présentés dans le tableau 25-9.

    Tableau 25-9 Configuration de paramètres opérationnels pour les services fonctionnels 
    Opération
    Procédure
    Désactiver l'état
    Désélectionnez la case Activé. Lorsque vous activez la session, le service arrête le traitement des messages.
    La désactivation d'un service n'entraîne plus la suppression des statistiques de surveillance précédemment collectées par le service.
    Activer l'état
    Cochez la case Activé. Lorsque vous activez la session, le service reprend le traitement des messages.
    Activer les URI endpoint de service fonctionnel à mettre hors ligne
    Définir l'intervalle entre les tentatives pour les URI hors ligne
    Cochez la case Activer avec un intervalle entre les tentatives.
    Lorsque vous sélectionnez cette option, le service fonctionnel enlève (met hors ligne) les URI endpoint qui ne répondent pas, à l'exécution. Ainsi seuls les URI actifs sont utilisés lors des nouvelles tentatives et pour le traitement des demandes ultérieures.
    Si vous souhaitez que les URI inactifs restent hors ligne jusqu'à procéder à une action corrective et les réactiver ensuite, n'indiquez aucun intervalle entre les tentatives. Par exemple, un intervalle de zéro signifie que l'URI endpoint reste hors ligne pour une durée indéterminée. Pour le remettre en ligne, utilisez l'action Marquer l'adresse comme étant en ligne sur la page URI endpoint. Reportez-vous à Visualisation des mesures des URI endpoint de service fonctionnel.
    Si vous souhaitez que le service fonctionnel mette les URI inactifs hors ligne et qu'il tente à nouveau d'accéder à ces mêmes URI endpoint pour le traitement des messages ultérieurs après un intervalle spécifique, indiquez un intervalle entre les tentatives. Spécifiez une valeur dans les champs heures, minutes et seconde(s), qui correspondra à la durée pendant laquelle les URI inactifs restent en dehors da la boucle de traitement des URI actifs.
    L'URI est mis hors ligne pendant l'intervalle indiqué, puis une nouvelle tentative est effectuée. Si l'URI endpoint répond, il est remis en ligne ; s'il ne répond pas, il reste hors ligne et ainsi de suite.
    Vous ne pouvez indiquer un intervalle entre les tentatives que si vous avez sélectionné l'option Activer avec un intervalle entre les tentatives.
    Désactiver la surveillance
    Désélectionnez la case Activé. Lorsque vous activez la session, le système arrête la collecte des statistiques de surveillance relatives au service et supprime les statistiques recueillies précédemment.
    Activer la surveillance
    Cochez la case Activé. Lorsque vous activez la session, le système démarre la collecte des statistiques de surveillance relatives au service.
    Définir l'intervalle d'agrégation d'un service
    Indiquez un intervalle (en heures et minutes), puis cliquez sur Mettre à jour. Si vous spécifiez un nombre d'heures supérieur à 1, la valeur par défaut indiquée pour les minutes est toujours 0. Si vous spécifiez la valeur 0 ou 1 pour les heures, vous pouvez également indiquer des minutes. Reportez-vous à Définition de l'intervalle d'agrégation d'un service.
    Vous pouvez modifier et enregistrer le paramètre Intervalle d'agrégation, que la surveillance soit activée ou non.
    Pour activer la surveillance à l'exécution, assurez-vous que la case Activer la surveillance est sélectionnée.
    Désactiver les alertes de contrat de niveau de service
    Désélectionnez la case Activé. Le système arrête l'évaluation des règles d'alerte configurées pour le service. Par conséquent, vous ne recevez plus les alertes de contrat de niveau de service associées à ce service.
    Activer les alertes de contrat de niveau de service
    Cochez la case Activé. Dans la liste déroulante Activer les alertes au niveau, limitez les alertes de contrat de niveau de service au niveau indiqué ou à un niveau supérieur, à savoir (du niveau le plus large au niveau le plus limité) : Normal (valeur par défaut), Avertissement, Mineur, Majeur, Critique et Fatal.
    La génération d'alertes de contrat de niveau de service dépend à la fois de l'état de surveillance globale et de celui relatif au service. Ces deux états doivent être activés pour que la génération d'alertes de contrat de niveau de service le soit également à l'exécution. Reportez-vous à Activation globale des alertes de contrat de niveau de service.
    Limiter le flux de messages envoyés à l'ensemble des adresses configurées dans le service fonctionnel
    Cochez la case Activé en regard de l'option Etat du ralentissement. Lorsque ce paramètre est activé, les messages sont traités par ordre de priorité. Vous pouvez éventuellement affecter une priorité aux messages à l'aide des options de routage. Sinon, les messages sont retirés de la file d'attente suivant l'ordre FIFO (First-In, First-Out - Premier entré, premier sorti).
    Pour plus d'informations, reportez-vous à "Throttling" dans le manuel Oracle Fusion Middleware Administrator's Guide for Oracle Service Bus à l'adresse suivante : http://www.oracle.com/pls/as111170/lookup?id=OSBAG614.
    Limiter le nombre de messages répartis sur un ensemble d'adresses configurées dans le service fonctionnel
    Si le paramètre Etat du ralentissement est activé, ce champ est obligatoire.
    Indiquez une valeur pour Nombre maximal de traitements simultanés. Il doit s'agir d'un entier positif différent de zéro.
    Si la valeur indiquée est dépassée, les messages entrants sont placés dans une file d'attente (de ralentissement), le cas échéant. Sinon, ils sont supprimés.
    Lorsque la file d'attente atteint sa capacité maximale, les messages mis en file d'attente dont la priorité est la plus faible sont enlevés et une erreur d'exécution est générée. Si un gestionnaire d'erreur est configuré, celui-ci est déclenché. Les nouveaux messages entrants seront mis en file d'attente.
    Indiquer si les messages envoyés aux services fonctionnels ayant dépassé le nombre maximal de messages simultanés autorisés doivent être conservés dans une file d'attente
    Définir le nombre maximal de messages conservés dans la file d'attente
    Si le paramètre Etat du ralentissement est activé, ce champ est obligatoire.
    Indiquez une valeur pour File d'attente de ralentissement (il doit s'agir d'un entier positif) ou conservez la valeur par défaut, zéro, signifiant qu'il n'existe aucune file d'attente de ralentissement. Lorsque cette valeur est égale à zéro (aucune file d'attente de ralentissement) et que le nombre de messages dépasse la valeur indiquée pour Nombre maximal de traitements simultanés, les messages sont supprimés et une erreur est générée. Si un gestionnaire d'erreur est configuré, celui-ci est déclenché.
    Si vous désactivez le ralentissement à l'exécution, tous les messages placés dans la file d'attente de ralentissement seront traités.
    Lorsque vous supprimez ou renommez un service fonctionnel, tous les messages placés dans la file d'attente de ralentissement sont supprimés.
    Indiquer la durée maximale de conservation des messages dans la file d'attente
    Si le paramètre Etat du ralentissement est activé, ce champ est facultatif.
    Indiquez une valeur, en millisecondes, pour Expiration des messages. Il doit s'agir d'un entier positif. La valeur zéro indique que les messages n'arriveront jamais à expiration.
    Ce paramètre est défini au niveau de la file d'attente, ce qui signifie que la valeur Expiration des messages est identique pour tous les messages de la file d'attente. Lorsque cette valeur est dépassée, les messages sont enlevés de la file d'attente sans être traités.
    Activer la fonction de trace de message pour un service fonctionnel

    Cochez la case Activé en regard de Fonction de trace de message. Une fois la fonction de trace de message activée, le système journalise les messages échangés entre le pipeline Oracle Service Bus et le service fonctionnel (messages de demande et de réponse sortants).

    Remarque : pour visualiser la trace dans le fichier journal ou de sortie standard (console du serveur), vous devez définir la journalisation d'Oracle WebLogic Server sur les niveaux de gravité suivants :

    Niveau de gravité minimal à journaliser : Info
    Fichier journal : Info
    Sortie standard : Info

    Pour plus d'informations sur la configuration des niveaux de gravité de journalisation, reportez-vous à Using Log Severity Levels dans le manuel Oracle Fusion Middleware Configuring Log Files and Filtering Log Messages for Oracle WebLogic Server à l'adresse http://www.oracle.com/pls/as111170/lookup?id=WLLOG141.

    Le cas échéant, les messages sortants journalisés peuvent également inclure le nombre de tentatives, le code d'erreur et le message d'erreur.
    Dans la liste déroulante Niveau de détail, sélectionnez le niveau de détail souhaité parmi les options suivantes :
    • Terse : affiche la date, l'heure, le type de service, le nom du service et l'URI.
    • En-têtes : affiche les informations du niveau Terse, ainsi que la représentation XML des métadonnées du message.
    • Complet : affiche les informations du niveau En-têtes, ainsi que les données traitées RAW, y compris les pièces jointes, le cas échéant.
    Si vous sélectionnez l'option Complet dans la liste déroulante Niveau de détail, vous devez indiquer la taille maximale (en kilo-octets) des données traitées de message dans le champ Limite de la fonction de trace des données traitées. Vous devez également spécifier l'encodage par défaut utilisé pour la journalisation des données traitées dans le champ Encodage par défaut. Ce paramètre s'avère utile lors de la journalisation de données traitées binaires ou de messages SOAP comportant des pièces jointes binaires.
    L'encodage par défaut peut être Base64 ou tout encodage pris en charge par Java, tel que défini à l'adresse suivante : http://java.sun.com/javase/6/docs/technotes/guides/intl/encoding.doc.html.
    Si vous n'indiquez aucune valeur dans le champ Encodage par défaut, Oracle Service Bus utilise l'encodage par défaut de l'hôte pour les données traitées. L'encodage par défaut dépend de la Java Virtual Machine (JVM), du système d'exploitation sous-jacent et des paramètres locaux du système d'exploitation.
    Si le paramètre indiqué dans le champ Encodage par défaut ne peut pas être utilisé (par exemple, s'il ne s'agit pas d'une option valide pour la configuration), Oracle Service Bus emploie l'encodage en Base64 pour les données traitées.
    Désactiver la fonction de trace de message pour un service fonctionnel
    Désélectionnez la case Activé en regard de Fonction de trace de message.
    Activer et désactiver la mise en mémoire cache des résultats
    Afin d'activer la mise en mémoire cache des résultats pour le service fonctionnel, cochez la case Activé. Vous devez vérifier que la mise en mémoire cache des résultats est également activée au niveau global. Reportez-vous à Activation globale de la mise en mémoire cache des résultats.
    Afin de désactiver la mise en mémoire cache des résultats pour un service fonctionnel, désélectionnez la case Activé et cliquez sur Mettre à jour. Tous les résultats mis en mémoire cache pour ce service fonctionnel sont vidés.
