Page Gestion des messages

La page Créer/Modifier un service proxy - Gestion des messages est l'une des pages destinées à la création et à la modification de services proxy, décrites dans les rubriques Création et configuration de services proxy et Modification des configurations des services proxy.

Cette page permet d'indiquer si le service proxy doit transmettre en continu le contenu des messages, la façon dont le service doit décoder les messages de demande reçus et encoder les messages de réponse envoyés, et si le service proxy doit transmettre en continu les pièces jointes MIME au lieu de placer leur contenu dans la mémoire tampon.

En utilisant cette page, vous pouvez activer le service proxy pour qu'il transmette en continu le contenu des messages plutôt que de le stocker en mémoire. Vous pouvez aussi faire en sorte qu'il décode et analyse les messages entrants au format MTOM/XOP, et envoie des réponses à l'aide du format MTOM/XOP, si nécessaire. La méthode SOAP MTOM (Message Transmission Optimization Mechanism) permet d'envoyer les données binaires vers et depuis des services Web. Elle utilise XML-binary Optimized Packaging (XOP) pour transférer les données binaires.

Grâce à cette page, vous pouvez aussi permettre au service proxy de stocker le contenu des pièces jointes MIME sur un fichier de disque, puis de traiter les données en mode transmission continue, sans placer le contenu des pièces jointes dans la mémoire tampon. Cela permet au service proxy de traiter les pièces jointes volumineuses de manière fiable et efficace.

Transaction obligatoire
Sélectionnez cette option pour garantir l'exécution par Oracle Service Bus du flux de messages du service proxy dans le contexte d'une transaction. S'il existe déjà une transaction globale, le fournisseur de transport la propage dans la demande (même si vous ne sélectionnez pas cette option). S'il n'existe pas de transaction globale, l'exécution du flux de messages démarre une transaction. Dans ce cas, le contexte de transaction commence avant que la configuration du service soit validée ou exécutée (par exemple, contrôle de sécurité ou validation WS-I), et avant l'exécution du flux de messages, ce qui garantit que l'ensemble des traitements et de l'exécution survient dans le contexte de la transaction.
Lorsque l'exécution du flux de messages démarre une transaction, la qualité de service est "Une seule fois". Cependant, si le paramètre de qualité de service sortante est défini sur "Au mieux" (par défaut) pour les actions Appels de service ou Publication, Oracle Service Bus exécute ces actions en dehors du contexte de transaction. Pour qu'Oracle Service Bus exécute ces actions dans le même contexte de transaction de demande, définissez le paramètre de qualité de service sur "Une seule fois" pour ces actions.
Le service conserve son modèle de messagerie (Synchrone, Asynchrone, Unidirectionnel), indépendamment du paramétrage de cette option.
Pour les temporisations de transaction, la valeur de temporisation de transaction globale configurée dans la console d'Oracle WebLogic Server s'applique.
Exceptions dans les transactions
Oracle Service Bus appelle le gestionnaire d'erreur système pour les échecs de transaction. Vous ne pouvez pas détecter les exceptions de transaction en échec dans un gestionnaire d'erreur configuré par l'utilisateur. Pour les modèles synchrones, une exception de transaction est renvoyée dans la réponse. Pour les modèles asynchrones dans lesquels la transaction est conçue pour être validée dans la demande, l'exception est renvoyée sur le thread de demande.
Dans les modèles asynchrones, une erreur survenue dans la réponse après la validation de la transaction dans la demande n'affecte pas la transaction.
Même transaction pour la réponse
Cette option s'applique uniquement aux modèles de messagerie asynchrone et unidirectionnelle.

Si vous sélectionnez cette option, Oracle Service Bus propage le contexte de transaction du thread de demande vers le thread de réponse.
Si vous sélectionnez cette option, le modèle de message devient automatiquement synchrone, indépendamment du paramètre de modèle de message initial (asynchrone ou unidirectionnel).
N'utilisez pas cette option, par exemple, si le service fonctionnel de la demande a exigé la validation de la transaction avant l'envoi de la réponse, comme dans un modèle unidirectionnel.
Pour les temporisations de transaction, la valeur de temporisation de transaction globale configurée dans la console d'Oracle WebLogic Server s'applique.
Transmission en continu de contenu
Sélectionnez cette option pour transmettre en continu le contenu des messages plutôt que de le stocker en mémoire.
Cochez la case Activé et choisissez :
  • De placer le contenu intermédiaire dans la mémoire tampon (Mémoire tampon en mémoire) ou sur un fichier de disque (Mémoire tampon sur disque)
  • D'activer éventuellement Compression
Pour plus d'informations, reportez-vous à Streaming body Content dans le manuel Oracle Fusion Middleware Administrator's Guide for Oracle Service Bus, à l'adresse suivante : http://www.oracle.com/pls/as111170/lookup?id=OSBAG314.
Prise en charge de XOP/MTOM
Oracle Service Bus prend en charge le format XOP/MTOM via les transports suivants :
  • HTTP/S
  • Local
  • SB
Cochez la case Activé pour permettre au service proxy de décoder et d'analyser les messages entrants au format MTOM/XOP, et d'envoyer des réponses à l'aide du format MTOM/XOP, si nécessaire. Cette option est désactivée pour les services proxy importés basés sur les configurations de versions précédentes.
Si vous choisissez d'activer la prise en charge XOP/MTOM, sélectionnez une méthode de gestion des données binaires dans les variables de contexte de message $header et $body, parmi les options suivantes :
  • Inclure les données binaires par référence : (par défaut) dans un message de demande entrant, remplacez les éléments xop:Include par des éléments ctx:binary-content lorsque vous paramétrez les variables de contexte de message $header et $body.
  • Inclure les données binaires par valeur : dans un message de demande entrant, remplacez les éléments xop:Include par les données binaires correspondantes des versions de texte encodé Base64 lorsque vous paramétrez les variables de contexte de message $header et $body.
Utilisez Inclure les données binaires par référence pour accéder directement à des données binaires, par exemple pour transmettre des données à un appel Java ou à une transformation MFL (Message Format Language).
Utilisez Inclure les données binaires par valeur dans les cas suivants :
  • Pour faire le lien entre des services MTOM et non-MTOM. Par exemple, prenez un service proxy MTOM recevant une requête qui est ensuite acheminée vers un service non MTOM. Cette option permet de garantir la conformité aux normes existantes pour l'envoi de données binaires encodées en Base64 au format XML.
  • Pour valider les contenus de messages par rapport à un schéma XML qui exige un élément binaire Base64 à utiliser à la place de données binaires.
Si la prise en charge de XOP/MTOM est activée pour un service proxy, tous les messages entrants ne doivent pas obligatoirement être au format MTOM. A l'inverse, ce paramètre indique que lorsqu'un message au format MTOM arrive, le service proxy doit le gérer en conséquence. Lorsque les services proxy dont la fonction XOP/MTOM n'est pas activée reçoivent un message au format MTOM, le service rejette ce message et envoie une erreur d'exécution.
Pièces jointes
Oracle Service Bus prend en charge les pièces jointes MIME transmises en continu à l'aide des transports suivants :
  • HTTP/S
  • Local (en cas de chaînage via un proxy HTTP avec activation de la transmission en continu des pièces jointes)
Cochez la case Pièces jointes de page vers le disque pour permettre au service proxy de transmettre les pièces jointes MIME en continu. Lorsqu'elle est activée pour les services proxy HTTP, cette option s'applique aux messages de demande entrants du service proxy.
Lorsque vous sélectionnez Prise en charge de XOP/MTOM, l'option Pièces jointes est disponible uniquement si vous choisissez Inclure les données binaires par référence dans Prise en charge de XOP/MTOM. Les données traitées contenant des pièces jointes doivent être conformes à la norme RFC 822. Plus précisément, les lignes contenant des en-têtes Internet doivent se terminer par un retour chariot (CRLF).
