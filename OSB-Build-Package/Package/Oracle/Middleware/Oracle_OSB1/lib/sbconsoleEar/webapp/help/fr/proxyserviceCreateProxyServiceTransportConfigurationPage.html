Page Configuration du transport

La page Créer/Modifier un service proxy - Configuration du transport est l'une des pages destinées à la création et à la modification de services proxy, décrites dans les rubriques Création et configuration de services proxy et Modification des configurations des services proxy.

Cette page permet de sélectionner le protocole de transport du service proxy et de définir d'autres paramètres généraux de configuration de transport. Le tableau 19-13 explique comment l'utiliser.

Remarque : La sécurité de niveau transport entrant s'applique aux applications client et aux services proxy Oracle Service Bus. La sécurité de niveau transport sortant s'applique aux connexions entre les services proxy et les services fonctionnels Oracle Service Bus. Pour en savoir plus sur la sécurité de niveau transport, reportez-vous à Configuring Transport-Level Security dans le manuel Oracle Fusion Middleware Developer's Guide for Oracle Service Bus, à l'adresses suivante : http://www.oracle.com/pls/as111170/lookup?id=OSBDV1557.

Protocole
Sélectionnez un protocole de transport dans la liste. Les protocoles disponibles varient selon le type de service que vous créez.
  • Service Web WSDL : http, jca, jms, local, sb, ws
  • Doté d'un type de transport : jejb
  • Service de messagerie : courrier électronique, fichier, ftp, http, jms, local, mq (si disponible), sftp, tuxedo
  • N'importe quel service SOAP : http, jms, local, sb
  • N'importe quel service XML : courrier électronique, fichier, ftp, http, jms, local, mq (si disponible), sb, sftp, tuxedo
URI endpoint
Entrez un URI endpoint au format du protocole de transport sélectionné dans le champ Protocole au-dessus. Les formats sont les suivants :
  • adresse électronique : mailfrom:mail-server-hostname:mail-server-port
  • fichier : file:///drivename:/somename
  • ftp : ftp://hostname:port/directory
  • http : someName

    • Le transport HTTP prend en charge les adresses HTTP et HTTPS.

  • jca : jca://<resource_adapter_jndi>
  • jejb : le format d'URL est ejb_jndi_name. L'URI configuré pour un service proxy JEJB devient le nom JNDI global permettant de localiser le bean de session sans conservation de statut (stateless) généré par le transport JEJB à partir de l'interface distante/fonctionnelle dans le JAR client.

    • Remarque : pour EJB 3.0, ejb_jndi_name correspond à l'attribut mappedName de l'annotation @javax.ejb.Stateless dans le bean généré. Le suffixe du nom JNDI de recherche pour le service EJB généré est #interface_class, qui est le nom qualifié complet de l'interface fonctionnelle.

    Vous pouvez accéder au service proxy JEJB comme suit :

    EJB 2.1: protocol://host:port/ejb_jndi_name

    EJB 3.0 : protocol://host:port/ejb_jndi_name#interface_class

    Remarque : si vos EJB sont exécutés sur IBM WebSphere, ejb_jndi_name doit avoir le format suivant :

    cell/nodes/node_name/servers/server_name/ejb_jndi_name

    ou

    cell/clusters/cluster_name/ejb_jndi_name

    comme décrit dans la documentation IBM WebSphere, à l'adresse suivante : http://publib.boulder.ibm.com/infocenter/wasinfo/v6r1/index.jsp?topic=/com.ibm.websphere.nd.iseries.doc/info/iseriesnd/ae/rnam_example_prop3.html.

  • jms : jms://host:port/factoryJndiName/destJndiName

    • Pour cibler une destination JMS vers plusieurs serveurs, utilisez le format d'URI suivant :

    jms://host1:port,host2:port/QueueConnectionFactory/DestName

    Lorsque vous créez un service proxy, vous pouvez configurer un URI endpoint JMS même si le serveur à cette adresse n'est pas disponible. Cependant, dans le cas de JMS, lorsque vous activez la session, l'adresse doit être disponible. Pour en savoir plus, reportez-vous à Activation de sessions.

    Remarque : lorsque Oracle WebLogic Server autorise les barres obliques dans les noms JNDI, tels que "myqueues/myqueue", les noms JNDI avec des barres obliques interfèrent avec le format URI requis par Oracle Service Bus, et vous ne pouvez pas utiliser ces noms. Pour passer outre ce problème, définissez un serveur étranger JMS et référencez-le dans l'URI. Pour plus d'informations, reportez-vous à Configurer les serveurs étrangers dans l'aide en ligne d'Oracle Fusion Middleware Oracle WebLogic Server Administration Console Online Help à l'adresse http://www.oracle.com/pls/as111170/lookup?id=WLACH.

    Dans un cluster : dans l'URI JMS, les noms d'hôte doivent correspondre exactement aux noms d'hôte des serveurs de cluster tels qu'ils sont configurés sur le serveur WebLogic.

  • local

    • Ce transport n'exige pas d'URI endpoint.

  • mq - mq://<local-queue-name>?conn=<mq-connection-resource-ref>

    • local-queue-name est le nom de la file d'attente MQ à partir de laquelle le service proxy lit les messages.

    mq-connection-resource-ref est le chemin (project/folder) et le nom de la ressource de connexion MQ ; par exemple, default/my_MQconnection.

    Remarque : L'URI endpoint ne doit pas contenir d'espaces. Par conséquent, ne créez pas de ressources de connexion MQ ou de projets/dossiers contenant des espaces.

    Pour accéder au transport MQ disponible dans Oracle Service Bus, reportez-vous à Connexions MQ.

URI endpoint (suite)
  • sb :service_name

    • service_name est l'identificateur unique du service proxy. Par défaut, ce nom est le nom du service proxy.

    service_name doit contenir uniquement les caractères autorisés dans les URI (conformément à la norme RFC2396 disponible sur http://www.ietf.org/rfc/rfc2396.txt), mais ne peut pas contenir les caractères barre oblique (/) ou deux-points (:).

  • sftp : sftp://hostname:port/directory
  • tuxedo : exportname

    • L'URI exportname correspond à un export WTC que le domaine Tuxedo distant identifie comme un service Tuxedo.

    Lorsque plusieurs URI sont indiqués, vous devez avoir des noms de ressource uniques pour les adresses. Lorsqu'aucun nom distant n'est indiqué, sa valeur est le nom de ressource. Lorsqu'aucun nom distant n'est entré ou lorsque les noms de ressource et distant sont identiques, un seul URI est autorisé. Dans ce cas, les noms de ressource et distant ont la même valeur. Cela permet aux utilisateurs employant des imports WTC déjà définis de se servir des fonctions d'équilibrage de charge et de basculement WTC.

    Remarque : si vous configurez deux URI identiques, une erreur indique que le nom de service existe déjà.

  • ws : /contextPath

    • contextPath doit être unique pour les services proxy qui utilisent le transport HTTP ou WS.

Obtenir tous les en-têtes
Sélectionnez Oui pour extraire tous les en-têtes du transport.
Sélectionnez Non pour n'extraire qu'un ensemble défini d'en-têtes. Si vous sélectionnez Non, entrez un ensemble d'en-têtes dans le champ En-tête, puis cliquez sur Ajouter. (Cette étape ne s'applique pas au transport local.)
Remarque : Oracle Service Bus ne transmet pas l'en-tête d'autorisation HTTP de la demande au pipeline car cela entraîne une vulnérabilité de la sécurité. Vous pouvez créer par inadvertance une action de journalisation qui écrit le nom utilisateur et son mot de passe décrypté dans un fichier journal. Si le modèle de conception exige que l'en-tête d'autorisation HTTP soit dans le pipeline, procédez comme suit :

a. Dans la commande de démarrage d'Oracle Service Bus, définissez la propriété système suivante sur True : com.bea.wli.sb.transports.http.GetHttpAuthorizationHeaderAllowed

b. Dans la console d'administration Oracle Service Bus, sur la page de configuration du transport, sélectionnez Obtenir tous les en-têtes ou En-têtes spécifiés par l'utilisateur et indiquez Autorisation.
c. Redémarrez Oracle Service Bus.

Oracle Service Bus transmettra l'en-tête d'autorisation au pipeline.
