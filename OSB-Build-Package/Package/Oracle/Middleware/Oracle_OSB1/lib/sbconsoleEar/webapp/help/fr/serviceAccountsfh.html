Spécification de comptes de service

Vous pouvez utiliser le même compte de service pour plusieurs services fonctionnels et services proxy. Pour indiquer le nom utilisateur et le mot de passe que fournit le compte de service, vous avec le choix entre les techniques suivantes :

  • Statique

    • Exige que vous enregistriez un nom utilisateur et un mot de passe avec la configuration du compte de service. Le compte de service encode ces nom utilisateur et mot de passe dans la demande sortante.

  • Pass-through de nom utilisateur et de mot de passe

    • Amène le compte de service à fournir le nom utilisateur et le mot de passe qu'il reçoit d'une demande client entrante. Par exemple, si une demande BASIC HTTP entrante contient "pat" et "patspassword" comme nom utilisateur et mot de passe, le compte de service encode "pat" et "patspassword" dans la demande sortante.

    Dans la mesure où cette technique exige que les demandes client incluent des nom utilisateur et mot de passe en clair, elle ne s'applique qu'aux demandes client qui utilisent le protocole BASIC HTTP, une authentification par jeton du nom utilisateur de sécurité des services Web avec un mot de passe en clair ou un jeton personnalisé de nom utilisateur et de mot de passe.

    Oracle recommande d'utiliser cette technique uniquement lorsque Oracle Service Bus et l'adresse appartiennent au même domaine d'authentification. Par exemple, utilisez cette technique lorsque vous acheminez des messages dans une seule organisation, et qu'Oracle Service Bus et le destinataire de messages procèdent à l'authentification par rapport à un serveur LDAP commun.

    Les restrictions suivantes s'appliquent à cette technique :

    • Elle ne peut pas être utilisée dans les demandes sortantes qui authentifient Oracle Service Bus avec un serveur local ou distant, ou avec une ressource système, tel un serveur FTP ou JMS.
    • Elle ne peut pas être utilisée avec la fonction XQuery fn-bea:lookupBasicCredentials. Pour plus d'informations, reportez-vous à "XQuery Implementation" dans le manuel Oracle Fusion Middleware Administrator's Guide for Oracle Service Bus à l'adresse http://www.oracle.com/pls/as111170/lookup?id=OSBAG444.
      • Remarque : Si votre proxy est un intermédiaire WSS actif, vous pouvez utiliser la sécurité WS pour crypter un jeton de nom utilisateur de sécurité WS ou une combinaison nom utilisateur/mot de passe personnalisée. Dans cette instance, le pass-through de nom utilisateur/mot de passe fonctionne car le proxy décrypte d'abord la demande et accède ensuite au nom utilisateur/mot de passe en clair.
  • Correspondance d'utilisateurs

    • Exige que vous mettiez en corrélation (en correspondance) le nom utilisateur qui résulte de l'authentification d'une demande entrante provenant d'un client (nom utilisateur local) avec un nom utilisateur et un mot de passe que vous spécifiez (nom utilisateur et mot de passe distants). Lorsque le compte de service reçoit une demande d'un client authentifié qui a été mis en correspondance, il fournit les nom utilisateur et mot de passe distants appropriés pour la demande sortante du service fonctionnel ou du service proxy.

    Si le client est authentifié au niveau du transport et du message, le compte de service met en correspondance le nom utilisateur du niveau message avec les nom utilisateur et mot de passe distants.

    Vous pouvez également mettre en correspondance un nom utilisateur anonyme avec les nom utilisateur et mot de passe distants.

    Les restrictions suivantes s'appliquent à cette technique :

    • Elle ne peut pas être utilisée dans les demandes sortantes qui authentifient Oracle Service Bus avec un serveur local ou distant, ou avec une ressource système, tel un serveur FTP ou JMS.
    • Elle ne peut pas être utilisée avec la fonction XQuery fn-bea:lookupBasicCredentials. Pour plus d'informations, reportez-vous à "XQuery Implementation" dans le manuel Oracle Fusion Middleware Administrator's Guide for Oracle Service Bus à l'adresse http://www.oracle.com/pls/as111170/lookup?id=OSBAG444.
