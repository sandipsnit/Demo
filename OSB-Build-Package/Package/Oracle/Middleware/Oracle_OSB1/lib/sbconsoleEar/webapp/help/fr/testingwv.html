Configuration des données de test des services proxy

Le tableau 32-1 répertorie les options de configuration pouvant être utilisées lors du test des services proxy. Les champs permettant d'accepter l'entrée dans le document de demande dépendent du type de service.

Tableau 32-1 Options de configuration des services proxy
Section
Options/Champs
Description
Nom
Nom du service proxy testé.
Opérations disponibles
Opérations associées au service proxy.
Actions de la console de test
Exécuter
Exécute le test.
Réinitialiser
Réinitialise les valeurs d'entrée.
Fermer
Ferme la console de test et n'exécute pas le test.
Tester la configuration
Définit les options de configuration de test.
Appel direct
Envoie un message au service proxy sans utiliser la couche transport Oracle Service Bus. Les données d'entrée dans la console de test doivent être identiques à celles qui sont envoyées par le client au service proxy.
L'appel indirect est le contraire de l'appel direct. Vous pouvez l'utiliser en désélectionnant l'option Appel direct. L'appel indirect est exécuté par défaut pour les services fonctionnels. Il envoie les messages via la couche transport. Dans ce cas, les données d'entrée dans la console de test doivent être identiques à celles qui sont envoyées par un service proxy au service appelé.
Inclure la fonction de trace
La fonction de trace indique l'état du message lors de sa transmission dans le système.
Document de demande
Les champs d'entrée génèrent le message de demande envoyé au service proxy. Cliquez sur Exécuter pour exécuter le test avec les valeurs saisies. La console de test affiche le message de demande, ainsi que les métadonnées et le message de réponse du service.
Les données entrées dans la section Document de demande sont propres au type de service. Les sections suivantes décrivent les différents types de service et les entrées requises pour chacun d'eux.
Tout XML
Ce type de service exige une entrée sous forme de données traitées. Les données traitées correspondent au contenu du message envoyé. Le contenu doit être un document XML. Vous pouvez rechercher un fichier ou entrer le contenu du message dans la zone de texte.
Document de demande (suite)
Tout SOAP
Ce type de service exige une entrée sous forme de données traitées. Les données traitées correspondent au contenu du message envoyé. Ce contenu est censé être l'enveloppe SOAP. Vous pouvez rechercher un fichier ou entrer le contenu du message dans la zone de texte.
Messagerie
Les services de messagerie acceptent quatre types d'entrée : Aucun, XML, Binaire ou Texte. Le service exige une entrée unique, basée sur un fichier ou sur un texte, sauf pour le type Aucun qui ne nécessite aucune entrée.
Oracle recommande de saisir une entrée binaire à partir d'un fichier. Les données saisies dans la zone de texte sont converties en entrées binaires à l'aide de l'encodage système.
Les données saisies à partir de fichiers pour les services de texte doivent être converties en texte. Le champ d'entrée d'encodage permet d'indiquer l'encodage à appliquer au cours de la conversion. Si ce champ n'est pas configuré, l'encodage système est utilisé. Par défaut, le champ d'encodage est initialisé avec la valeur d'encodage configurée sur l'adresse du service proxy.
XML
Ce type de service exige une entrée sous forme de données traitées. Les données traitées correspondent au contenu du message envoyé. Le contenu doit être un document XML. Vous pouvez rechercher un fichier ou entrer le contenu du message dans la zone de texte.
Si votre service proxy est basé sur le WSDL et que plusieurs opérations sont définies pour celui-ci, la console de test génère et fournit un exemple de document à utiliser lors du test du service. Vous pouvez utiliser les données échantillon directement, les modifier, puis exécuter le test, ou fournir vos propres données de test.
Toutes les opérations sont répertoriées en haut de la page.
Document de demande (suite)
Document SOAP
Dans le cas d'un document SOAP, l'enveloppe SOAP contient généralement soit aucun en-tête, soit plusieurs, ainsi que des données traitées de corps. Les onglets Panneau et XML fournissent d'autres méthodes de spécification du contenu.
L'onglet Form contient les champs En-tête SOAP et Corps SOAP. Le contenu du champ En-tête SOAP doit être une balise d'en-tête SOAP (cela permet la définition de plusieurs en-têtes). Le champ Corps SOAP contient les données envoyées avec le message. Le contenu doit être un document XML. A la fois l'en-tête et le corps sont utilisés pour générer l'enveloppe SOAP.
Document de demande (suite)
SOAP RPC
Dans le cas d'un document SOAP RPC, l'enveloppe SOAP contient soit aucun en-tête, soit plusieurs et soit aucun argument, soit plusieurs.
Les onglets Panneau et XML fournissent d'autres méthodes de spécification du contenu.
L'onglet Panneau contient une entrée unique pour les en-têtes SOAP et un champ d'entrée pour chaque argument (le nom du champ d'entrée correspond au nom de l'argument). Le contenu du champ En-tête SOAP doit être une balise d'en-tête SOAP (cela permet la définition de plusieurs en-têtes).
Le WSDL permet de détecter le type de chaque argument. Un champ d'entrée monoligne est utilisé pour les types de primitive et un champ d'entrée multiligne pour les types XML. Un exemple de document est automatiquement généré pour faciliter le test.
Les en-têtes et les arguments sont utilisés par la console de test pour générer l'enveloppe SOAP.
L'onglet XML contient un champ d'entrée unique. Le contenu de ce champ est censé être l'enveloppe SOAP envoyée. Les données traitées (entrée XML) peuvent être basées sur un fichier ou sur un texte. Le référencement d'un fichier pour entrée est prioritaire sur l'entrée textuelle. Sélectionnez le fichier à utiliser dans votre test.
Web Service Security
Cette section n'est disponible que pour les services SOAP lorsque l'opération sélectionnée contient une stratégie de sécurité de service Web (WSS).
Fournisseur de services
Le service de test obtient toutes les informations d'identification et de connexion (paire de clés) PKI côté client pour les opérations de sécurité de service Web (signature numérique et cryptage) à partir de ce fournisseur de services. Ce champ est facultatif. Dans certains scénarios, un fournisseur de services est nécessaire en fonction de la stratégie de demande/réponse de l'opération. Pour plus d'informations, reportez-vous à "Testing Services with Web Service Security" dans le manuel Oracle Fusion Middleware Administrator's Guide for Oracle Service Bus à l'adresse suivante : http://www.oracle.com/pls/as111170/lookup?id=OSBAG384.
Nom utilisateur
Nom utilisateur employé dans les jetons de nom utilisateur de sécurité de service Web générés à partir du service de test. Ce champ est facultatif. Ce nom utilisateur n'est nécessaire que dans certains scénarios pour lesquels la stratégie de demande de l'opération indique une assertion d'identité.
Ne pas confondre ce champ avec le champ de nom utilisateur de contexte de sécurité de transport.
Il doit s'agir d'un nom utilisateur et d'un mot de passe valides dans le domaine de sécurité local. Un nom utilisateur ou mot de passe non valide entraîne une erreur côté client sur le service de test.
Mot de passe
Mot de passe employé dans les jetons de nom utilisateur de sécurité de service Web générés à partir du service de test.
Transport
La section Transport est réduite par défaut. Les champs et les valeurs varient en fonction de la configuration de test.
Authentification
Nom utilisateur
Nom utilisateur servant à définir le contexte de sécurité utilisé par le service de test lors de l'appel du service proxy.
Si le service proxy achemine le message vers un service fonctionnel attendant un jeton SAML, il s'agit de l'identité qui sera représentée par le jeton. Pour plus d'informations, reportez-vous à "Using SAML for Authentication" dans le manuel Oracle Fusion Middleware Developer's Guide for Oracle Service Bus à l'adresse suivante : http://www.oracle.com/pls/as111170/lookup?id=OSBDV1647.
Ne pas confondre ce champ avec le champ de nom utilisateur de sécurité de service Web (WSS). Il doit s'agir d'un nom utilisateur et d'un mot de passe valides dans le domaine de sécurité local. Un nom utilisateur ou mot de passe non valide entraîne une erreur côté client sur le service de test.
Remarque : lorsque la console de test appelle un service proxy disposant d'une authentification de jeton personnalisé HTTP, aucune vérification de l'authentification n'est effectuée.
Mot de passe
Mot de passe associé. Pour plus d'informations, reportez-vous à Nom utilisateur.
Mode d'appel
Demande/Réponse
Cette option n'apparaît que lors du test d'un service proxy de type "n'importe quel service SOAP" ou "n'importe quel service XML". Désélectionnez la case Demande/Réponse pour les appels de service unidirectionnels.

Remarque : le message SOAP sécurisé est affiché avec des caractères non imprimables supplémentaires.
