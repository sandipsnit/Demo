Pagina Configurazione trasporto

La pagina Crea/Modifica un servizio proxy - Configurazione trasporto fa parte di una serie di pagine per la creazione e la modifica dei servizi proxy, come descritto in Creazione e configurazione dei servizi proxy e Modifica delle configurazioni servizi proxy.

Utilizzare questa pagina per selezionare un protocollo di trasporto per il servizio proxy e specificare altre impostazioni di configurazione del trasporto generiche. Nella tabella 19-3 viene descritto come utilizzare la pagina.

Nota: la sicurezza a livello di trasporto in entrata è applicata alle applicazioni client e ai servizi proxy Oracle Service Bus. La sicurezza a livello di trasporto in uscita è applicata alle connessioni tra i servizi business e i servizi proxy Oracle Service Bus. Per ulteriori informazioni sulla sicurezza a livello di trasporto, vedere "Configuring Transport-Level Security" in Oracle Fusion Middleware Developer's Guide for Oracle Service Bus all'indirizzo http://www.oracle.com/pls/as111170/lookup?id=OSBDV1557.

Protocollo
Selezionare un protocollo di trasporto dalla lista. I protocolli disponibili differiscono a seconda del tipo di servizio creato.
  • Servizio Web WSDL: http, jca, jms, local, sb, ws
  • Servizio tipo di trasporto: jejb
  • Servizio messaggistica: email, file, ftp, http, jms, local, mq (se disponibile), sftp, tuxedo
  • Qualsiasi servizio SOAP: http, jms, local, sb
  • Qualsiasi servizio XML: email, file, ftp, http, jms, local, mq (se disponibile), sb, sftp, tuxedo
URI endpoint
Immettere un URI endpoint nel formato basato sul protocollo di trasporto selezionato nel campo Protocollo. I formati disponibili sono i seguenti:
  • email - mailfrom:mail-server-hostname:mail-server-port
  • file - file:///drivename:/somename
  • ftp - ftp://hostname:port/directory
  • http - someName

    • Il trasporto HTTP supporta entrambi gli endpoint HTTP che HTTPS.

  • jca - jca://<resource_adapter_jndi>
  • jejb: il formato URL è ejb_jndi_name. L'URI configurato per un servizio proxy JEJB diventa il nome JNDI globale per l'individuazione del bean di sessione senza conservazione dello stato generato dal trasporto JEJB dall'interfaccia remota o business nel file JAR client.

    • Nota: per EJB 3.0, ejb_jndi_name è l'attributo mappedName dell'annotazione @javax.ejb.Stateless nel bean generato. Il nome JNDI di ricerca per il servizio EJB generato ha il suffisso #interface_class, che corrisponde al nome completamente qualificato dell'interfaccia business.

    È possibile accedere al servizio proxy JEJB nei seguenti modi:

    EJB 2.1: protocol://host:port/ejb_jndi_name

    EJB 3.0: protocol://host:port/ejb_jndi_name#interface_class

    Nota: se gli EJB sono in esecuzione su IBM WebSphere, ejb_jndi_name deve essere nel seguente formato:

    cell/nodes/node_name/servers/server_name/ejb_jndi_name

    Oppure

    cell/clusters/cluster_name/ejb_jndi_name

    come descritto nella documentazione IBM WebSphere all'indirizzo http://publib.boulder.ibm.com/infocenter/wasinfo/v6r1/index.jsp?topic=/com.ibm.websphere.nd.iseries.doc/info/iseriesnd/ae/rnam_example_prop3.html.

  • jms - jms://host:port/factoryJndiName/destJndiName

    • Per specificare una destinazione JMS a più server, utilizzare il seguente formato URI:

    jms://host1:port,host2:port/QueueConnectionFactory/DestName

    Quando si crea un servizio proxy, è possibile configurare un URI di endpoint JMS anche se il server su tale endpoint non è disponibile. Tuttavia, nel caso di JMS, quando si attiva la sessione, l'endpoint deve essere disponibile. Per ulteriori informazioni, vedere Attivazione delle sessioni.

    Nota: anche se Oracle WebLogic Server consente l'utilizzo delle barre nei nomi JNDI, ad esempio "myqueues/myqueue", i nomi JNDI contenenti barre interferiscono con il formato URI richiesto da Oracle Service Bus e non sono pertanto utilizzabili. Una soluzione alternativa a questo problema consiste nel definire un server JMS esterno e farvi riferimento nell'URI. Per ulteriori informazioni, vedere "Configure foreign servers" in Oracle Fusion Middleware Oracle WebLogic Server Administration Console Online Help all'indirizzo http://www.oracle.com/pls/as111170/lookup?id=WLACH.

    In un cluster: i nomi host nell'URI JMS devono corrispondere esattamente ai nomi host dei server cluster configurati in WebLogic Server.

  • local

    • Questo trasporto non richiede un URI endpoint.

  • mq - mq://<local-queue-name>?conn=<mq-connection-resource-ref>

    • local-queue-name è il nome della coda MQ dalla quale il servizio proxy legge i messaggi.

    mq-connection-resource-ref è il percorso (progetto/cartella) e il nome della risorsa di connessione MQ, ad esempio default/my_MQconnection.

    Nota: l'URI endpoint non può contenere spazi, pertanto non creare risorse di connessione MQ o progetti/cartelle contenenti spazi nel nome.

    Per rendere disponibile il trasporto MQ in Oracle Service Bus, vedere Connessioni MQ.

URI endpoint (continua)
  • sb - service_name

    • service_name è l'identificativo univoco per il servizio proxy. Per impostazione predefinita, corrisponde al nome del servizio proxy.

    service_name deve contenere solo i caratteri consentiti negli URI (come descritto in RFC2396 all'indirizzo http://www.ietf.org/rfc/rfc2396.txt), ad eccezione della barra (/) o dei due punti (:).

  • sftp - sftp://hostname:port/directory
  • tuxedo - exportname

    • L'exportname dell'URI corrisponde a un'esportazione WTC che l'interfaccia del dominio Tuxedo remoto identifica come un servizio Tuxedo.

    Se si specificano più URI, è necessario disporre di nomi di risorsa univoci per gli endpoint. Se non si specifica alcun nome remoto, il relativo valore è quello del nome della risorsa. Se non si immette alcun nome remoto o il nome remoto e il nome della risorsa corrispondono, è consentito un solo URI. In questo caso, il nome della risorsa e il nome remoto avranno lo stesso valore. Ciò consente agli utenti che utilizzano importazioni WTC già definite di usare il failover e il bilanciamento del carico WTC.

    Nota: se si configurano due URI identici, un errore indica che il nome del servizio esiste già.

  • ws - /contextPath

    • contextPath deve essere univoco per i servizi proxy che utilizzano il trasporto HTTP o WS.

Recupera tutte le intestazioni
Selezionare Sì per recuperare tutte le intestazioni dal trasporto.
Selezionare No per recuperare un set di intestazioni definito. Se si seleziona No, immettere un set di intestazioni nel campo Intestazione, quindi fare clic su Aggiungi. (questo passo non si applica al trasporto locale).
Nota: Oracle Service Bus non passa l'intestazione dell'autorizzazione HTTP dalla richiesta alla pipeline per evitare di rendere vulnerabile la sicurezza. Si potrebbe creare involontariamente un'azione di log che scrive il nome utente e la password non cifrata in un file di log. Se il proprio pattern di progettazione richiede che l'intestazione dell'autorizzazione HTTP sia inclusa nella pipeline, effettuare le seguenti operazioni:

a. Nel comando di avvio per Oracle Service Bus, impostare la seguente proprietà del sistema su true: com.bea.wli.sb.transports.http.GetHttpAuthorizationHeaderAllowed

b. Nella console di amministrazione di Oracle Service Bus, nella pagina Configurazione trasporto, selezionare Recupera tutte le intestazioni o Intestazioni specificate dall'utente e specificare Autorizzazione.
c. Riavviare Oracle Service Bus.

Oracle Service Bus passerà l'intestazione dell'autorizzazione alla pipeline.

Al termine della procedura

Fare clic su Successivo per proseguire con la configurazione di questo servizio nella pagina successiva oppure fare clic su Ultimo per esaminare e salvare la configurazione nella pagina Crea/Modifica un servizio proxy - Riepilogo.
