Specifica degli account di servizio

È possibile utilizzare lo stesso account di servizio per più servizi business e servizi proxy. Per specificare il nome utente e la password forniti da un account di servizio, è possibile utilizzare uno dei seguenti metodi:

  • Statico

    • Richiede di salvare un nome utente e una password con la configurazione dell'account di servizio. Quest'ultimo codifica il nome utente e la password nella richiesta in uscita.

  • Nome utente e password pass-through

    • Fa in modo che l'account di servizio fornisca il nome utente e la password ricevuti da una richiesta client in entrata. Ad esempio, se una richiesta HTTP BASIC in entrata contiene "pat" e "patspassword" come nome utente e password, l'account di servizio codifica "pat" e "patspassword" nella richiesta in uscita.

    Con questo metodo le richieste client devono includere nomi utente e password con testo in chiaro. Per questo motivo, tale metodo è applicabile solo per le richieste client che utilizzano il protocollo BASIC HTTP, un'autenticazione token nome utente Web Services Security con una password con testo in chiaro o un token nome utente e password personalizzato.

    Oracle consiglia di utilizzare questa tecnica solo quando Oracle Service Bus e l'endpoint appartengono allo stesso dominio di autenticazione. Ad esempio, utilizzare questa tecnica quando si instradano i messaggi in una singola organizzazione e sia Oracle Service Bus che il consumer messaggi eseguono l'autenticazione su un server LDAP comune.

    A questa tecnica si applicano le seguenti restrizioni:

    • Non può essere utilizzata nella richieste in uscita che autenticano Oracle Service Bus su un server locale o remoto o una risorsa di sistema, ad esempio un server FTP o un server JMS.
    • Non può essere utilizzata con la funzione XQuery fn-bea:lookupBasicCredentials. Per ulteriori informazioni, vedere "XQuery Implementation" in Oracle Fusion Middleware Administrator's Guide for Oracle Service Bus all'indirizzo http://www.oracle.com/pls/as111170/lookup?id=OSBAG444.
      • Nota: se il proxy è un intermediario WSS attivo, è possibile utilizzare WS-Security per cifrare un token di nome utente WS-Security o un nome utente e una password personalizzati. In questo caso, il pass-through nome utente/password funziona perché il proxy per prima cosa decifra la richiesta e quindi accede al nome utente e alla password con testo in chiaro.
  • Mapping dell'utente

    • Richiede di correlare (mappare) il nome utente risultante dall'autenticazione di una richiesta in entrata di un client (il nome utente locale) a un nome utente e una password specificati personalmente (il nome utente e la password remoti). Quando l'account di servizio riceve una richiesta da un client autenticato che è stato mappato, fornisce il nome utente e la password appropriati per la richiesta in uscita del servizio business o proxy.

    Se il client esegue l'autenticazione sia a livello di trasporto che a livello di messaggio, l'account di servizio mappa il nome utente a livello di messaggio al nome utente e alla password remoti.

    È anche possibile mappare un nome utente anonimo a un nome utente e a una password remoti.

    A questa tecnica si applicano le seguenti restrizioni:

    • Non può essere utilizzata nella richieste in uscita che autenticano Oracle Service Bus su un server locale o remoto o una risorsa di sistema, ad esempio un server FTP o un server JMS.
    • Non può essere utilizzata con la funzione XQuery fn-bea:lookupBasicCredentials. Per ulteriori informazioni, vedere "XQuery Implementation" in Oracle Fusion Middleware Administrator's Guide for Oracle Service Bus all'indirizzo http://www.oracle.com/pls/as111170/lookup?id=OSBAG444.
