Configurazione dei dati di test dei servizi proxy

Nella tabella 32-1 sono elencate le opzioni di configurazione per l'esecuzione del test dei servizi proxy. I campi per l'accettazione dell'input per il documento di richiesta dipendono dal tipo di servizio.

Chiamata diretta
Consente di inviare un messaggio al servizio proxy senza utilizzare il layer di trasporto Oracle Service Bus. I dati di input della console di test devono essere quelli inviati dal client al servizio proxy.
L'opposto della chiamata diretta è una chiamata indiretta. Può essere richiamata deselezionando l'opzione Chiamata diretta. Viene eseguita per impostazione predefinita per i servizi business. La chiamata indiretta invia i messaggi tramite il layer di trasporto. In questo caso, i dati di input della console di test devono essere quelli inviati da un servizio proxy al servizio richiamato.
Includi trace
Il trace mostra lo stato del messaggio mentre passa nel sistema.
Documento richiesta
I campi di input generano il messaggio di richiesta inviato al servizio proxy. Fare clic su Esegui per eseguire il test con i valori immessi. La console di test visualizza il messaggio di richiesta e i metadati e il messaggio di risposta del servizio.
L'input della sezione Documento richiesta dipende dal tipo di servizio. Nelle sezioni che seguono sono elencati i tipi di servizio e una descrizione dell'input richiesto da ciascuno di essi.
Qualsiasi XML
Il servizio richiede l'input sotto forma di payload. Il payload è il contenuto del messaggio inviato. Si prevede che il contenuto sia un documento XML. È possibile cercare un file o immettere il contenuto del messaggio nella casella di testo.
Documento richiesta (continua)
Qualsiasi SOAP
Il servizio richiede l'input sotto forma di payload. Il payload è il contenuto del messaggio inviato. È previsto che il contenuto sia un envelope SOAP. È possibile cercare un file o immettere il contenuto del messaggio nella casella di testo.
Messaggistica
I servizi di messaggistica definiscono quattro possibili tipi di input: none, XML, Binary e Text. Il servizio richiede un singolo input, basato su file o basato su testo, ad eccezione del tipo none, che non richiede alcun input.
Oracle consiglia di immettere l'input binario da un file. I dati immessi nell'area di testo vengono convertiti in input binario utilizzando la codifica del sistema.
I dati immessi dai file per i servizi di testo devono essere convertiti in testo. Il campo di input di codifica è utilizzato per specificare la codifica da applicare durante la conversione. Se non si configura questo campo, viene utilizzata la codifica del sistema. Per impostazione predefinita, il campo di codifica è inizializzato con il valore di codifica configurato sull'endpoint del servizio proxy.
XML
Il servizio richiede l'input sotto forma di payload. Il payload è il contenuto del messaggio inviato. Si prevede che il contenuto sia un documento XML. È possibile cercare un file o immettere il contenuto del messaggio nella casella di testo.
Se il servizio proxy è un servizio basato su WSDL con più operazioni definite, la console di test genera e fornisce un documento di esempio da utilizzare quando si esegue il test del servizio. È possibile utilizzare questi dati di esempio direttamente, modificarli e quindi eseguire il test o fornire dati di test personali.
Tutte le operazioni sono elencate nella parte superiore della pagina.
Documento richiesta (continua)
Documento SOAP
Per un documento SOAP, l'envelope SOAP è generalmente costituito da zero o più intestazioni e un payload del corpo. Le schede Form e XML forniscono metodi alternativi per specificare il contenuto.
La scheda Form contiene un campo Intestazione SOAP a un campo Corpo SOAP. Si prevede che il contenuto del campo Intestazione SOAP sia una tag Intestazione SOAP (che consente la definizione di più intestazioni). Il campo Corpo SOAP contiene i dati effettivamente inviati come parte del messaggio. Si prevede che il contenuto sia un documento XML. Sia l'intestazione che il corpo sono utilizzati per generare l'envelope SOAP.
Documento richiesta (continua)
RPC SOAP
Per RPC SOAP, l'envelope SOAP è costituito da zero o più intestazioni e zero o più argomenti.
Le schede Form e XML forniscono metodi alternativi per specificare il contesto.
La scheda Form contiene un singolo input per le intestazioni SOAP e un campo di input per ogni argomento (il nome del campo di input corrisponde al nome dell'argomento). Si prevede che il contenuto del campo Intestazione SOAP sia una tag Intestazione SOAP (che consente la definizione di più intestazioni).
Il WSDL è utilizzato per rilevare il tipo di ciascun argomento. Per i tipi primitivi viene utilizzato un campo di input con una singola riga; per i tipi XML viene utilizzato un campo di input con più righe. Viene generato automaticamente un documento di esempio per facilitare l'esecuzione del test.
Le intestazioni e gli argomenti utilizzati dalla console di test per generare l'envelope SOAP.
La scheda XML contiene un singolo campo di input. Si prevede che il contenuto di questo campo sia l'envelope SOAP inviato. Il payload (l'input XML) può essere basato su file o su testo. Il riferimento a un file per l'input ha precedenza sull'input testuale. Cercare e selezionare il file da utilizzare nel test.
Web Service Security
Questa sezione è disponibile solo per i servizi SOAP quando l'operazione selezionata ha un criterio Web Service Security (WSS).
Provider di servizi
Il servizio di test ottiene tutte le credenziali PKI lato client (coppia di chiavi) per le operazioni di sicurezza dei servizi Web (firma digitale e cifratura) da questo provider di servizi. Questo campo è facoltativo. Determinati scenari richiedono un provider di servizi, in base ai criteri di richiesta/risposta per l'operazione. Per ulteriori informazioni, vedere "Testing Services with Web Service Security" in Oracle Fusion Middleware Administrator's Guide for Oracle Service Bus all'indirizzo http://www.oracle.com/pls/as111170/lookup?id=OSBAG384.
Nome utente
Il nome utente nei token nome utente Web Service Security generati dal servizio di test. Questo campo è facoltativo. Questo nome utente è necessario solo in alcuni scenari in cui il criterio di richiesta dell'operazione specifica un'asserzione delle identità.
Non confondere questo campo con il campo del nome utente del contesto di sicurezza del trasporto.
Deve trattarsi di un nome utente e di una password validi nel realm di sicurezza locale. L'utilizzo di un nome utente e di una password non validi causeranno un errore lato client sul servizio di test.
Password
La password nei token nome utente Web Service Security generati dal servizio di test.
Trasporto
La sezione Trasporto è compressa per impostazione predefinita. I campi e i valori dipendono dalla configurazione del test.
Autenticazione
Nome utente
Il nome utente per l'impostazione del contesto di sicurezza utilizzato dal servizio di test quando si richiama il servizio proxy.
Se il servizio proxy instrada il messaggio a un servizio business per il quale è previsto un token SAML, si tratta dell'identità che verrà rappresentata dal token. Per ulteriori informazioni, vedere "Using SAML for Authentication" in Oracle Fusion Middleware Developer's Guide for Oracle Service Bus all'indirizzo http://www.oracle.com/pls/as111170/lookup?id=OSBDV1647.
Non confondere questo campo con il campo del nome utente Web Service Security (WSS). Deve trattarsi di un nome utente e di una password validi nel realm di sicurezza locale. L'utilizzo di un nome utente e di una password non validi causeranno un errore lato client sul servizio di test.
Nota: quando la console di test richiama un proxy con l'autenticazione del token personalizzato HTTP, il controllo dell'autenticazione non viene eseguito.
Password
La password associata. Per ulteriori informazioni, vedere Nome utente.
Modalità di richiamo
Richiesta/risposta
Questa opzione viene visualizzata solo quando si esegue il test di un servizio proxy di tipo Qualsiasi SOAP o Qualsiasi XML. Deselezionare la casella di controllo Richiesta/risposta per i richiami dei servizi unidirezionali.

Nota: il messaggio SOAP protetto viene stampato con spazi bianchi aggiuntivi.
