メッセージ処理構成ページ

ビジネス・サービスの作成/編集 - メッセージ処理構成ページは、ビジネス・サービスの作成および編集(ビジネス・サービスの作成と構成およびビジネス・サービス構成の編集を参照)に関する一連のページの1つです。

このページでは、Oracle Service Busでメッセージ・コンテンツを処理する方法を指定します。表18-20に、ビジネス・サービスのメッセージ処理のオプションを示します。使用できるオプションは、トランスポート・タイプごとに異なります。

表18-20 ビジネス・サービスの作成/編集 - メッセージ処理構成ページ
オプション
作成または編集の方法
XOP/MTOMサポート
この機能を使用して、MTOM/XOP書式でアウトバウンド・メッセージをエンコードできます。SOAP Message Transmission Optimization Mechanism (MTOM)は、バイナリ・データをWebサービスとの間で送受信する方法です。MTOMでは、XML-binary Optimized Packaging (XOP)を使用してバイナリ・データを転送します。
Oracle Service Busでは、次のトランスポートを使用したXOP/MTOMがサポートされます。
  • HTTP/S
  • ローカル
  • SB
「有効」チェック・ボックスを選択すると、ビジネス・サービスでアウトバウンド・メッセージをMTOM/XOP書式でエンコードできます。このオプションは、前のリリースの構成に基づくインポートされたビジネス・サービスに対しては無効になります。
「XOP/MTOMサポート」が有効になっている場合は、$headerおよび$bodyメッセージ・コンテキスト変数のバイナリ・データを処理する方法を次のオプションから選択します。
  • 「参照によるバイナリ・データを含む」: (デフォルト)アウトバウンド応答メッセージで、$bodyメッセージ・コンテキスト変数の設定時にxop:Include要素をctx:binary-content要素で置換します。
  • 「値によるバイナリ・データを含む」: アウトバウンド応答メッセージで、$bodyメッセージ・コンテキスト変数の設定時にxop:Include要素を対応するバイナリ・データのBase64エンコード・テキスト・バージョンで置換します。
「XOP/MTOMサポート」がビジネス・サービスに対して有効になっている場合は、すべてのアウトバウンド・メッセージがMTOM書式である必要はありません。かわりに、この設定はビジネス・サービスがMTOMペイロードを処理できることを指定します。
Oracle Service BusではMTOMとSwAの組合せがサポートされないため、Oracle Service Busがアウトバウンド要求をビジネス・サービスにディスパッチしようとした場合、およびビジネス・サービスでMTOMとXOPの両方が有効で、$attachmentsメッセージ・コンテキスト変数がnullでない場合は、システムによってランタイム・エラーが発行されます。
添付ファイル
Oracle Service Busでは、HTTP/Sトランスポートを使用したMIME添付ファイルのストリーミングがサポートされます。
この機能を使用して、アウトバウンド応答メッセージ内の添付ファイルをディスク・ファイルに格納し、添付ファイルのコンテンツをメモリーにバッファリングせずにストリーミング形式でデータを処理することもできます。これにより、ビジネス・サービスで、大きな添付ファイルを堅牢かつ効率的に処理できます。
「添付ファイルのディスクへのページング」チェック・ボックスを選択すると、ビジネス・サービスがアウトバウンド応答メッセージ内の添付ファイルをストリーミングできます。
「XOP/MTOMサポート」を有効にした場合、「添付ファイル」オプションは、「XOP/MTOMサポート」で「参照によるバイナリ・データを含む」オプションを選択した場合にのみ使用できます。添付ファイルを含むペイロードはRFC 822に準拠している必要もあります。具体的には、インターネット・ヘッダーを含む行は、CRLF (復帰改行)で終了する必要があります。
結果キャッシュ
結果がほとんど変わらないビジネス・サービスを呼び出す場合は、結果キャッシュを使用すると、外部サービスを直接呼び出すかわりにキャッシュされた結果をクライアントに戻すことにより、ビジネス・サービスのパフォーマンスが向上します。
ビジネス・サービスの結果キャッシュを使用するには、結果キャッシュのグローバルな有効化の説明に従って、結果キャッシュをグローバルに有効化しておく必要があります。
次のガイドラインに従って、「詳細設定」セクションで結果キャッシュを構成します。
  • 「サポートされています」オプションを選択します。選択後、残りのフィールドが有効になります。
  • 「式ネームスペース」 - 式を使用して結果キャッシュを構成している場合は、「キャッシュ・トークン式」、「有効期限」またはその両方を指定すると、式で使用するネームスペースおよび対応する接頭辞を入力できます。このフィールドでは、既存のネームスペースを表示することもできます。
  • 「キャッシュ・トークン式」 - Oracle Service Busでは、キャッシュ・キーを使用して、取得または移入のキャッシュされた結果を特定します。キャッシュ・キーのキャッシュ・トークン部分では、一意の識別子が提供されます。「キャッシュ・トークン式」の式を使用して、ビジネス・サービスのキャッシュされた結果を一意に識別するキャッシュ・キーのキャッシュ・トークン部分を生成します。

    • (ビジネス・サービスを呼び出すプロキシ・サービス・メッセージ・フロー内で)リクエストの値からキャッシュ・トークンを生成するには、パイプライン$body、$header、$operationまたは$transportMetaData ($outbound/ctx:transport/ctx:requestまたは$outbound/ctx:transport/ctx:response)から値を取得する式を使用します。たとえば、メッセージ$bodyの顧客IDからキャッシュ・トークンを移入できます。

    キャッシュ・トークン式は、文字列または単純なコンテンツ(属性や子要素を持たない要素など)の値に解決される必要があります。式がnullに評価されたり、エラーが発生した場合、結果はキャッシュされません。

    注意: ビジネス・サービスの構成で「キャッシュ・トークン式」を設定せずに要求からキャッシュを生成することもできます。これを行うには、$outbound/ctx:transport/ctx:request/ctx:cache-tokenの値をプロキシ・サービス・パイプラインに含めます。このキャッシュ・トークン内の値は、ビジネス・サービス構成のキャッシュ・トークン式をオーバーライドします。

  • 「有効期限」 - 有効期限(生存時間(TTL))は、ビジネス・サービスの結果キャッシュのエントリをフラッシュするタイミングを決定します。ゼロ(0)の期間は、期限がないことを意味します。マイナスの期間は、キャッシュしないことを意味します。次のいずれかを選択します:
 
    • デフォルトの使用 - デフォルトはDOMAIN_HOME/config/osb/coherence/osb-coherence-config.xmlのexpiry-delayの値です。デフォルト値は5分です。
    • 期間 - 特定の長さの時間を指定します。
    • 「XQuery式」 - 式にキャッシュ・トークンを設定する場合と同様に、要求または応答から有効期限を取得するXQuery式を入力できます。「リクエスト」または「レスポンス」を選択します。

      リクエストまたはレスポンスの値から有効期限を生成するには、パイプライン$body、$header、$operationまたは$transportMetaData ($outbound/ctx:transport/ctx:requestまたは$outbound/ctx:transport/ctx:response)から値を取得する式を使用します。たとえば、Cache-Control HTTPヘッダーに設定した値を使用します。

      「有効期限」は、整数(秒を表します)、XQuery dayTimeDuration (XSD型)または秒を表す単純なコンテンツ(属性や子要素を持たない要素など)の整数値に解決される必要があります。式がnullに評価されたり、エラーが発生した場合、結果はキャッシュされません。

      注意: ビジネス・サービス構成の「有効期限」を設定せずに、要求から有効期限を生成することもできます。これを行うには、$outbound/ctx:transport/ctx:request/ctx:cache-ttlの値をプロキシ・サービス・パイプラインに含めます。cache-ttl要素の値は、ビジネス・サービス構成の「有効期限」をオーバーライドします。
結果キャッシュの詳細は、Oracle Fusion Middleware Oracle Service Busのための管理者ガイドのビジネス・サービスの結果キャッシュによるパフォーマンスの向上(http://www.oracle.com/pls/as111170/lookup?id=OSBAG170)を参照してください。
