전송 구성 페이지

비즈니스 서비스 생성/편집 - 전송 구성 페이지는 비즈니스 서비스 생성 및 구성과 비즈니스 서비스 구성 편집에 설명된 것처럼 비즈니스 서비스를 생성하고 편집하기 위한 일련의 페이지 중 하나입니다.

이 페이지에서는 비즈니스 서비스의 전송 프로토콜을 선택하고 기타 일반 전송 구성 설정을 지정할 수 있습니다.

아웃바운드 전송 레벨 보안은 Oracle Service Bus 프록시 서비스와 비즈니스 서비스 간의 접속에 적용됩니다. 전송 레벨 보안에 대한 자세한 내용은 http://www.oracle.com/pls/as111170/lookup?id=OSBDV1557에서 Oracle Fusion Middleware Developer's Guide for Oracle Service Bus의 "Configuring Transport-Level Security"를 참조하십시오.

표 18-3에서는 이 페이지를 사용하는 방법에 대해 설명합니다.

표 18-3 비즈니스 서비스 생성/편집 - 전송 구성 페이지
옵션
생성 또는 편집
프로토콜
목록에서 전송 프로토콜을 선택합니다. 생성한 서비스 유형에 따라 사용할 수 있는 프로토콜은 다음과 같이 다릅니다.
  • WSDL 웹 서비스: SOA-DIRECT, BPEL-10g, DSP, HTTP, JCA, JMS, SB, WS
  • 전송 유형의 서비스: ejb, flow, jejb
  • 메시징 서비스: 전자 메일, 파일, FTP, HTTP, JMS, MQ(사용 가능한 경우), SFTP, Tuxedo
  • 모든 SOAP 서비스: DSP, HTTP, JMS, SB
  • 모든 XML 서비스: DSP, 전자 메일, 파일, FTP, HTTP, JMS, MQ(해당하는 경우), SB, SFTP, Tuxedo
로드 밸런싱 알고리즘
다음 로드 밸런싱 알고리즘 중 하나를 선택합니다.
  • 라운드 로빈 - 이 비즈니스 서비스의 끝점 URI 필드에 입력한 URL의 순서를 동적으로 지정합니다. 첫번째 URL이 실패하면 재시도 횟수 제한에 도달할 때까지 계속해서 다음 URL을 시도합니다.

    • 새 메시지마다 새 URL 순서가 지정됩니다.

  • 임의 - 이 비즈니스 서비스의 끝점 URI 필드에 입력된 URL 목록의 순서를 임의로 지정합니다. 첫번째 URL이 실패하면 재시도 횟수 제한에 도달할 때까지 계속해서 다음 URL을 시도합니다.
  • 랜덤 가중치 - 이 비즈니스 서비스의 끝점 URI 필드에 입력된 URL 목록의 순서를 랜덤으로 지정하지만 가중치 필드에 입력된 값을 기반으로 일부 URL을 다른 URL보다 많이 시도합니다.
  • 없음 - 이 비즈니스 서비스의 끝점 URI 필드에 입력된 URL 목록의 순서를 위에서 아래로 지정합니다.
끝점 URI
프로토콜 필드에서 선택한 전송 프로토콜에 기반한 형식으로 끝점 URL을 입력합니다. 다음은 각 전송의 URI 형식에 대한 설명입니다. 선택 사항인 URI 요소는 대괄호 []로 표시됩니다. 전송 URI에 대한 자세한 내용은 http://www.oracle.com/pls/as111170/lookup?id=OSBDV에서 Oracle Fusion Middleware Developer's Guide for Oracle Service Bus의 해당 전송 장을 참조하십시오.
  • bpel-10g - protocol://host[:port] [/protocol-path]/domain/process[/version[/partnerlink/role]]
  • dsp - t3://dsp-ip-address:port/dsp-app-name
  • ejb - ejb:jndi_provider_name:ejb_jndi_name

    • URI에서 jndi_provider_name은 JNDI 제공자 리소스의 이름이고 ejb_jndi_name은 EJB에 대한 JNDI 서버의 JNDI 이름입니다.

    JNDI 제공자가 동일한 서버에 있으면 JNDI 제공자를 지정할 필요가 없습니다. 그러면 URI는 ejb::ejb_jndi_name이 됩니다.

    주: EJB가 IBM WebSphere에서 실행되는 경우 ejb_jndi_name은 다음 형식이어야 합니다.

    cell/nodes/node_name/servers/server_name/ejb_jndi_name

    또는

    cell/clusters/cluster_name/ejb_jndi_name

    자세한 내용은 http://publib.boulder.ibm.com/infocenter/wasinfo/v6r1/index.jsp?topic=/com.ibm.websphere.nd.iseries.doc/info/iseriesnd/ae/rnam_example_prop3.html에서 IBM WebSphere 설명서를 참조하십시오.

  • 전자 메일 - 다음 형식의 조합으로 끝점 URI를 여러 개 입력할 수 있습니다.
    • mailto:name@domain_name.com
    • mailto:name@domain_name.com?smtp=smtp_server_resource
    • mailto:name@domain_name.com?mailsession=jndi_mail_session
  • 파일 - file:///drivename:/somename
  • 흐름 - flow:<분할-조인 리소스에 대한 참조 경로>

    • 분할-조인 리소스에 대한 참조 경로는 분할-조인의 경로(project/folder) 및 이름입니다(예: batchorderProcessing/batchorder).

  • FTP - ftp://host:port/directory
  • HTTP - http://host:port/someService

    • HTTP 전송은 HTTP 및 HTTPS 끝점을 모두 지원합니다.

  • JCA - jca://<resource_adapter_jndi>
  • jejb - jejb:jndi_provider_name:ejb_jndi_name

    • EJB 2.1 예: jejb:myProvider:osb.jejb.myJejbBiz21

    EJB 3.0 예: jejb:myProvider:myBiz31#osb.jejb.myJejbBiz

    주: EJB가 IBM WebSphere에서 실행되는 경우 ejb_jndi_name은 다음 형식이어야 합니다.

    cell/nodes/node_name/servers/server_name/ejb_jndi_name

    또는

    cell/clusters/cluster_name/ejb_jndi_name

    자세한 내용은 http://publib.boulder.ibm.com/infocenter/wasinfo/v6r1/index.jsp?topic=/com.ibm.websphere.nd.iseries.doc/info/iseriesnd/ae/rnam_example_prop3.html에서 IBM WebSphere 설명서를 참조하십시오.

  • JMS - jms://host:port/factoryJndiName/destJndiName

    • JMS 대상을 다중 서버로 대상을 지정하려면 다음 URI 형식을 사용합니다.
    jms://host1:port,host2:port/QueueConnectionFactory/DestName

    주: Oracle WebLogic Server에서는 "myqueues/myqueue"와 같이 JNDI 이름에 슬래시를 사용할 수 있지만 JNDI 이름에 슬래시가 포함되어 있으면 Oracle Service Bus에 필요한 URI 형식과 충돌하므로 이러한 이름을 사용할 수 없습니다. 이 문제를 해결하려면 JMS 외부 서버를 정의하고 URI에서 이 외부 서버를 참조하십시오. 자세한 내용은 Oracle Fusion Middleware Oracle WebLogic Server Administration Console Online Help(http://www.oracle.com/pls/as111170/lookup?id=WLACH)의 "Configure foreign servers"를 참조하십시오.

    클러스터에서: JMS URI의 호스트 이름은 WebLogic Server에 구성된 클러스터 서버의 호스트 이름과 정확히 일치해야 합니다.

끝점 URI(계속)
  • MQ - mq://<local-queue-name>?conn=<mq-connection-resource-ref>

    • local-queue-name은 비즈니스 서비스에서 읽는 메시지를 가진 MQ 대기열의 이름입니다.

    mq-connection-resource-ref는 MQ 접속 리소스의 경로(프로젝트/폴더) 및 이름입니다. 예를 들면 default/my_MQconnection과 같습니다.

    주: 끝점 URI는 공백을 포함할 수 없으므로 이름에 공백을 넣어 MQ 접속 리소스 또는 프로젝트/폴더를 생성하지 마십시오.

    Oracle Service Bus에서 MQ 전송을 사용 가능하도록 설정하려면 MQ 접속을 참조하십시오.

  • SB - sb://[<jndi_provider_name/>]service_name

    • jndi_provider_name(선택 사항)은 Oracle Service Bus JNDI 제공자 리소스의 이름입니다. 생략할 경우 기본 컨텍스트가 사용됩니다.

    service_name은 대상 서비스이며 원격 프록시 서비스 URI에 해당합니다.

  • SFTP - sftp://hostname:port/directory
  • SOA-DIRECT - [protocol://authority]/default/compositeName[!versionNumber[*label]]/serviceName

    • SOA 서비스가 Oracle Service Bus와 동일한 서버에 있을 경우 protocol과 authority는 선택 사항입니다.

  • Tuxedo - tuxedo:resourcename[/remotename]

    • 이 URI에서 resourcename은 WTC 임포트 이름을 나타내며, remotename은 원격 Tuxedo 도메인에서 익스포트한 서비스 이름을 나타냅니다. URI resourcename은 필수이며 remotename은 선택 사항입니다.

    둘 이상의 URI를 지정할 경우 끝점에 대한 고유한 리소스 이름이 있어야 합니다. 원격 이름을 지정하지 않을 경우 해당 값은 리소스 이름의 값입니다. 원격 이름을 입력하지 않거나 원격 및 리소스 이름이 같을 경우 하나의 URI만 허용됩니다. 이 경우 리소스 이름과 원격 이름은 같은 값을 가집니다. 따라서 이미 정의된 WTC 임포트에서 WTC 로드 밸런싱 및 장애 조치를 활용할 수 있습니다.

  • WS - http://host:port/somename 또는 https://host:port/somename
추가를 눌러 하나 이상의 추가 URI를 추가합니다. 런타임 시 URL은 로드 밸런싱 알고리즘 필드에서 선택한 로드 밸런싱 알고리즘을 기반으로 선택됩니다.
로드 밸런싱 알고리즘 필드에서 랜덤-가중치를 선택한 경우 끝점 URI 필드에 가중치를 입력할 수도 있습니다. 기본값은 1입니다.
다중 끝점이 정의되어 있고 로드 밸런싱 알고리즘 필드에서 없음을 선택한 경우에는 끝점의 순서가 중요합니다. 옵션 열의 위로 및 아래로 화살표를 사용하여 끝점의 순서를 바꿀 수 있습니다.
Oracle Service Bus는 동일한 비즈니스 서비스 내에서 중복 끝점 URI를 지원하지 않습니다.
재시도 횟수
아웃바운드 요청 전송 시 전달이 실패할 경우 개별 URL 끝점 재시도 횟수, 즉 장애 조치 시도 횟수를 지정합니다.
예를 들어 비즈니스 서비스에 하나의 구성된 URI(U1)가 있으며 재시도 횟수는 3으로 설정되어 있습니다. U1이 첫번째 시도에서 실패할 경우 시스템은 U1 끝점을 3회 추가로 재시도합니다.
비즈니스 서비스에 2개의 구성된 URI(U1 및 U2)가 있고 재시도 횟수가 3일 때 첫번째 시도(예: U1)가 실패하면 시스템은 다음 URI(U2)를 시도합니다. 여기서도 실패하면 시스템은 추가로 U1 및 U2에 대해 각각 1회씩 시도합니다.
재시도 반복 간격
목록의 모든 끝점 URI에 대해 다시 반복하기 전에 시스템이 일시 중지하는 시간(초)을 지정합니다.
예를 들어, 비즈니스 서비스는 2개의 구성된 URI(U1 및 U2)와 재시도 반복 간격이 5초인 재시도 횟수 2로 구성되어 있습니다. U1에 대한 첫번째 시도가 실패할 경우 시스템은 즉시 U2를 시도합니다. 그러나 두번째 시도도 실패할 경우 5초 동안 기다린 다음 U1에 대해 다시 한 번 재시도합니다.
재시도 응용 프로그램 오류
예 또는 아니오를 선택합니다.
아웃바운드 요청 전송 시 전달에 실패할 경우 응용 프로그램 오류(예: SOAP 오류)에 기반한 끝점 URI를 재시도할지 여부를 지정합니다.
이 필드는 HTTP, EJB, JMS, DSP, Tuxedo, SB 및 WS 전송에 대해서만 제공됩니다.

마친 후에

다음을 눌러 다음 페이지에서 이 서비스를 계속 구성하거나 마지막을 눌러 비즈니스 서비스 생성/편집 - 요약 페이지에서 이 구성을 검토 및 저장합니다.
