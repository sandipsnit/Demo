전송 구성 페이지

프록시 서비스 생성/편집 - 전송 구성 페이지는 프록시 서비스 생성 및 구성과 프록시 서비스 구성 편집에 설명된 것처럼 프록시 서비스를 생성하고 편집하기 위한 일련의 페이지 중 하나입니다.

이 페이지에서는 프록시 서비스의 전송 프로토콜을 선택하고 기타 일반 전송 구성을 설정할 수 있습니다. 표 19-3에서는 이 페이지를 사용하는 방법에 대해 설명합니다.

주: 인바운드 전송 레벨 보안은 클라이언트 응용 프로그램 및 Oracle Service Bus 프록시 서비스에 적용됩니다. 아웃바운드 전송 레벨 보안은 Oracle Service Bus 프록시 서비스와 비즈니스 서비스 간의 접속에 적용됩니다. 전송 레벨 보안에 대한 자세한 내용은 http://www.oracle.com/pls/as111170/lookup?id=OSBDV1557에서 Oracle Fusion Middleware Developer's Guide for Oracle Service Bus의 "Configuring Transport-Level Security"를 참조하십시오.

표 19-3 프록시 서비스 생성/편집 - 전송 구성 페이지
옵션
생성 또는 편집
프로토콜
목록에서 전송 프로토콜을 선택합니다. 생성한 서비스 유형에 따라 사용할 수 있는 프로토콜은 다음과 같이 다릅니다.
  • WSDL 웹 서비스: HTTP, JCA, JMS, 로컬, SB, WS
  • 전송 유형: JEJB
  • 메시징 서비스: 전자 메일, 파일, FTP, HTTP, JMS, 로컬, MQ(사용 가능한 경우), SFTP, Tuxedo
  • 모든 SOAP 서비스: HTTP, JMS, 로컬, SB
  • 모든 XML 서비스: 전자 메일, 파일, FTP, HTTP, JMS, 로컬, MQ(사용 가능한 경우), SB, SFTP, Tuxedo
끝점 URI
위의 프로토콜 필드에서 선택한 전송 프로토콜 기반 형식으로 끝점 URI를 입력합니다. 형식은 다음과 같습니다.
  • 전자 메일 - mailfrom:mail-server-hostname:mail-server-port
  • 파일 - file:///drivename:/somename
  • FTP - ftp://hostname:port/directory
  • HTTP - someName

    • HTTP 전송은 HTTP 및 HTTPS 끝점을 모두 지원합니다.

  • JCA - jca://<resource_adapter_jndi>
  • jejb - URL 형식은 ejb_jndi_name입니다. JEJB 프록시 서비스에 대해 구성된 URI가 전역 JNDI 이름으로 지정되어 클라이언트 JAR의 원격/비즈니스 인터페이스에서 JEJB 전송에 의해 생성된 Stateless 세션 Bean을 찾는 데 사용됩니다.

    • 주: EJB 3.0의 경우 ejb_jndi_name은 생성된 Bean에 있는 @javax.ejb.Stateless 주석의 mappedName 속성입니다. 생성된 EJB 서비스의 JNDI 조회 이름 뒤에는 비즈니스 인터페이스의 전체 이름인 #interface_class가 붙습니다.

    다음과 같이 JEJB 프록시 서비스에 액세스할 수 있습니다.

    EJB 2.1: protocol://host:port/ejb_jndi_name

    EJB 3.0: protocol://host:port/ejb_jndi_name#interface_class

    주: EJB가 IBM WebSphere에서 실행되는 경우 ejb_jndi_name은 다음 형식이어야 합니다.

    cell/nodes/node_name/servers/server_name/ejb_jndi_name

    또는

    cell/clusters/cluster_name/ejb_jndi_name

    자세한 내용은 http://publib.boulder.ibm.com/infocenter/wasinfo/v6r1/index.jsp?topic=/com.ibm.websphere.nd.iseries.doc/info/iseriesnd/ae/rnam_example_prop3.html에서 IBM WebSphere 설명서를 참조하십시오.

  • jms - jms://host:port/factoryJndiName/destJndiName

    • JMS 대상을 다중 서버로 대상을 지정하려면 다음 URI 형식 사용:

    jms://host1:port,host2:port/QueueConnectionFactory/DestName

    프록시 서비스를 생성하는 경우 해당 끝점에서 서버를 사용할 수 없는 경우에도 JMS 끝점 URI를 구성할 수 있습니다. 그러나, JMS의 경우 세션을 활성화할 때 끝점을 사용할 수 있어야 합니다. 자세한 내용은 세션 활성화를 참조하십시오.

    주: Oracle WebLogic Server에서는 "myqueues/myqueue"와 같이 JNDI 이름에 슬래시를 사용할 수 있지만 JNDI 이름에 슬래시가 포함되어 있으면 Oracle Service Bus에 필요한 URI 형식과 충돌하므로 이러한 이름을 사용할 수 없습니다. 이 문제를 해결하려면 JMS 외부 서버를 정의하고 URI에서 이 외부 서버를 참조하십시오. 자세한 내용은 Oracle Fusion Middleware Oracle WebLogic Server Administration Console Online Help(http://www.oracle.com/pls/as111170/lookup?id=WLACH)의 "Configure foreign servers"를 참조하십시오.

    클러스터에서: JMS URI의 호스트 이름은 WebLogic Server에 구성된 클러스터 서버의 호스트 이름과 정확히 일치해야 합니다.

  • 로컬

    • 이 전송에는 끝점 URI가 필요하지 않습니다.

  • MQ - mq://<local-queue-name>?conn=<mq-connection-resource-ref>

    • local-queue-name은 프록시 서비스에서 읽는 메시지를 가진 MQ 대기열의 이름입니다.

    mq-connection-resource-ref는 MQ 접속 리소스의 경로(프로젝트/폴더) 및 이름입니다. 예를 들면 default/my_MQconnection과 같습니다.

    주: 끝점 URI는 공백을 포함할 수 없으므로 이름에 공백을 넣어 MQ 접속 리소스 또는 프로젝트/폴더를 생성하지 마십시오.

    Oracle Service Bus에서 MQ 전송을 사용 가능하도록 설정하려면 MQ 접속을 참조하십시오.

끝점 URI(계속)
  • SB - service_name

    • service_name은 프록시 서비스의 고유한 식별자입니다. 기본적으로 이 이름은 프록시 서비스 이름입니다.

    service_name에는 URI에 허용된 문자만 포함되어야 합니다(http://www.ietf.org/rfc/rfc2396.txt의 RFC2396 참조). 단, 백슬래시(/) 또는 콜론(:)은 포함될 수 없습니다.

  • sftp - sftp://hostname:port/directory
  • Tuxedo - exportname

    • URI exportname은 원격 Tuxedo 도메인이 Tuxedo 서비스로 식별하는 WTC 익스포트에 해당합니다.

    둘 이상의 URI를 지정할 경우 끝점에 대한 고유한 리소스 이름이 있어야 합니다. 원격 이름을 지정하지 않을 경우 해당 값은 리소스 이름의 값입니다. 원격 이름을 입력하지 않거나 원격 및 리소스 이름이 같을 경우 하나의 URI만 허용됩니다. 이 경우 리소스 이름과 원격 이름은 같은 값을 가집니다. 따라서 이미 정의된 WTC 임포트를 사용하는 사용자는 WTC 로드 밸런싱 및 장애 조치를 활용할 수 있습니다.

    주: 동일한 URI를 두 개 구성하면 서비스 이름이 이미 존재함을 알려주는 오류 메시지가 표시됩니다.

  • WS - /contextPath

    • HTTP 또는 WS 전송을 사용하는 프록시 서버에 대해 contextPath가 고유해야 합니다.

모든 헤더 임포트
예를 선택하면 전송의 모든 헤더를 검색합니다.
아니오를 선택하면 정의된 헤더 세트만 검색합니다. 아니오를 선택한 경우 헤더 필드에 헤더 세트를 입력하고 추가를 누릅니다. (이 단계는 로컬 전송에는 적용되지 않습니다.)
주: 보안상의 취약점 때문에 Oracle Service Bus에서는 요청의 HTTP 권한 부여 헤더를 파이프 라인으로 전달하지 않습니다. 로그 파일에 사용자 이름과 암호화되지 않은 비밀번호를 기록하는 로그 작업을 실수로 생성할 수 있습니다. 파이프 라인에 있는 HTTP 권한 부여 헤더가 필요한 경우 다음을 수행합니다.

a. Oracle Service Bus의 시작 명령에서 다음 시스템 속성을 true로 설정합니다. com.bea.wli.sb.transports.http.GetHttpAuthorizationHeaderAllowed

b. Oracle Service Bus 관리 콘솔의 전송 구성 페이지에서 모든 헤더 가져오기 또는 사용자 지정 헤더를 선택한 다음 권한 부여를 지정합니다.
c. Oracle Service Bus를 재시작합니다.

Oracle Service Bus에서는 권한 부여 헤더를 파이프 라인으로 전달합니다.
