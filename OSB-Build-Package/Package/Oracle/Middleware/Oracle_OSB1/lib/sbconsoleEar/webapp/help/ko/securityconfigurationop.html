  • 보안 구성 데이터 및 세션

사용자

사용자는 인증할 수 있는 엔티티입니다. 사용자는 개인 또는 소프트웨어 엔티티(웹 서비스 클라이언트 등)일 수 있습니다. 보안 영역 내에서 각 사용자에게 고유 ID(이름)를 부여해야 합니다.

일반적으로, 사용자는 다음 두 가지 범주로 나뉩니다.

  • 프록시 서비스 또는 비즈니스 서비스를 액세스할 수 있는 클라이언트 사용자

    • 많은 수의 클라이언트 사용자를 생성하는 경우 보안 그룹으로 구성할 것을 고려하십시오.

  • Oracle Service Bus 관리 콘솔을 사용하여 프록시 서비스, 비즈니스 서비스 및 다른 Oracle Service Bus 리소스를 생성하거나 수정할 수 있는 관리 사용자.

    • Oracle Service Bus는 관리 기능에 대해 역할 기반 보안을 사용합니다. 사용자에게 직접 액세스 권한을 부여하는 대신 Oracle Service Bus는 보안 역할에만 관리 권한을 부여합니다. 사용자에게 관리 권한을 부여하려면 사전 정의된 보안 역할 중 하나인 기본 보안 그룹 중 하나에 사용자를 포함시킵니다.

그룹

많은 사용자를 원활하게 관리하기 위해 이들 사용자를 명명된 그룹으로 구성할 수 있습니다. 그런 다음, 개별 사용자에게 액세스 권한 또는 역할 ID를 부여하는 대신 그룹에 권한이나 ID를 부여합니다.

관리 보안 그룹

Oracle Service Bus는 프록시 서비스 생성 같은 관리 기능에 사용자 액세스를 원활하게 부여하기 위해 기본 보안 그룹을 제공합니다. 각 그룹은 관리 권한이 부여되어 있는 사전 정의된 Oracle Service Bus 보안 역할 중 하나에 있습니다.

자세한 내용은 http://www.oracle.com/pls/as111170/lookup?id=OSBDV1539에서 Oracle Fusion Middleware Developer's Guide for Oracle Service Bus의 "Configuring Administrative Security"를 참조하십시오.

역할

보안 역할은 런타임 환경의 조건을 기반으로 사용자 또는 그룹에 부여될 수 있는 ID입니다. 액세스 제어 정책을 생성할 때 역할, 그룹 또는 사용자에 대한 액세스를 부여할 수 있습니다.

예를 들어, MyCustomersEast와 MyCustomersWest 등 두 개 그룹을 생성할 수 있습니다. PrivilegedCustomer라는 보안 역할을 생성한 다음 MyCustomersWest 그룹은 동부 표준시 오전 8시부터 오후 8시까지의 역할에 있지만 MyCustomersEast 그룹은 동부 표준시 오후 8시부터 오전 8시까지의 역할에 있도록 조건을 생성합니다. 그런 다음 서비스에 대한 PrivilegedCustomer 역할 액세스를 제공하는 프록시 서비스에 대한 액세스 제어 정책을 생성합니다. 다른 사용자는 MyCustomersEast와 MyCustomersWest 중 어느 그룹에 속하는지에 따라 다른 시간에 액세스하게 됩니다.

관리 보안 역할

Oracle Service Bus는 (WebLogic Server에서 사전 정의된 네 가지 역할에 더해) 관리 권한을 부여하는 네 가지 사전 정의된 보안 역할을 제공합니다. Oracle Service Bus 관리 보안 역할에 대한 액세스 권한은 변경할 수는 없지만 사용자나 그룹이 역할 중 하나에 있는 조건은 변경할 수 있습니다.

이러한 롤 및 롤별 사용 가능한 권한에 대한 자세한 내용은 http://www.oracle.com/pls/as111170/lookup?id=OSBDV1539에서 Oracle Fusion Middleware Developer's Guide for Oracle Service Bus의 "Configuring Administrative Security"를 참조하십시오.

액세스 제어 정책

액세스 제어 정책은 사용자, 그룹 또는 역할이 프록시 서비스에 액세스할 수 있는 조건을 지정합니다. 예를 들어, GoldCustomer 역할의 사용자는 항상 프록시 서비스를 액세스할 수 있고 SilverCustomer 역할의 사용자는 평일 밤 자정 이후에만 프록시 서비스를 액세스할 수 있는 정책을 생성할 수 있습니다.

모든 프록시 서비스의 경우 클라이언트가 프록시 서비스와 연결하려고 시도할 때 보안 점검을 적용하는 전송 수준 정책을 생성할 수 있습니다. 전송 수준 정책에 나열된 사용자의 요구만 처리할 수 있습니다.

메시지 수준 액세스 제어 정책은 클라이언트가 메시지 수준 보안을 사용하여 프록시 서비스를 호출하려고 시도할 때 보안 점검을 적용합니다. 다음의 경우에 메시지 수준 액세스 제어 정책을 생성할 수 있습니다.

  • 활성 웹 서비스 보안 중간 단계인 프록시 서비스의 경우
  • 메시지 수준 사용자 정의 인증인 프록시 서비스의 경우

메시지 수준 정책에 나열된 사용자만 작업을 호출할 수 있습니다.

보안 구성 데이터 및 세션

사용자, 그룹 및 역할은 Oracle Service Bus 세션에서 관리되지 않는 보안 제공자에서 그대로 유지됩니다. 따라서, 세션 내부에서나 외부에서 이 데이터를 생성하거나 수정할 수 있습니다. 이 데이터에 추가하거나 수정한 내용은 즉시 모든 세션에 적용됩니다. 데이터를 추가하거나 수정한 세션을 폐기해도 보안 데이터는 폐기되지 않습니다.

액세스 제어 정책은 권한 부여 제공자에 저장됩니다. 이제 Oracle Service Bus 저장소에 액세스 제어 정책에 대한 참조가 있습니다.

액세스 제어 정책은 Oracle Service Bus 디자인 세션 외부가 아닌 내부에서 관리됩니다. 변경은 세션 내에서 이루어지므로 다른 리소스와 마찬가지로 변경 사항을 커밋하거나 폐기할 수 있습니다.

ACL은 Oracle Service Bus 관리 콘솔에서 관리할 수 있지만 정책은 Oracle Service Bus 외부에서 변경할 수 있습니다. 하지만 Oracle Service Bus 외부에서 정책을 변경하면 Oracle Service Bus의 참조가 만료되거나 부적합할 수 있습니다.

따라서 일관성 있게 관리하려면 Oracle Service Bus 세션 외부에서 ACL을 완전히 관리하거나(권한 부여 제공자 MBeans 또는 타사 권한 부여 제공자 툴 사용) Oracle Service Bus 세션 내부에서 ACL을 완전히 관리해야 합니다. 두 방법을 조합하여 사용하면 정책에 일관성이 없을 수 있습니다.
