서비스 계정 지정

다중 비즈니스 서비스와 프록시 서비스에 동일한 서비스 계정을 사용할 수 있습니다. 서비스 계정이 제공하는 사용자 이름과 비밀번호를 지정하기 위해 다음 방법을 사용할 수 있습니다.

  • 정적

    • 서비스 계정 구성과 함께 사용자 이름과 비밀번호를 저장해야 합니다. 서비스 계정은 아웃바운드 요청에서 해당 사용자 이름과 비밀번호를 인코딩합니다.

  • 사용자 이름과 비밀번호 통과

    • 서비스 계정이 들어오는 클라이언트 요청에서 받는 사용자 이름과 비밀번호를 제공하게 됩니다. 예를 들어, 인바운드 HTTP BASIC 요청에 “pat” 과 “patspassword”가 사용자 이름과 비밀번호로 포함된다면 서비스 계정은 아웃바운드 요청으로 “pat”과 “patspassword”를 인코딩합니다.

    이 기법에서는 클라이언트 요청에 일반 텍스트로 구성된 사용자 이름과 비밀번호를 포함해야 하기 때문에 HTTP BASIC 프로토콜을 사용하거나 일반 텍스트 비밀번호와 함께 웹 서비스 보안 사용자 이름 토큰 인증 또는 사용자 정의 사용자 이름 및 비밀번호 토큰을 사용하는 클라이언트 요청에만 이 기법이 적용됩니다.

    Oracle Service Bus와 끝점이 동일한 인증 도메인에 속한 경우에만 이 기법을 사용하는 것이 좋습니다. 예를 들어, 단일 조직 내에서 메시지를 라우팅할 때와 Oracle Service Bus와 메시지 소비자가 모두 공통 LDAP 서버에 대해 인증할 경우 이 기법을 사용합니다.

    이 방법에는 다음 제한 사항이 적용됩니다.

    • Oracle Service Bus를 FTP 서버 또는 JMS 서버 등의 로컬 또는 원격 서버나 시스템 리소스에 인증하는 아웃바운드 요청에서는 사용할 수 없습니다.
    • fn-bea:lookupBasicCredentials XQuery 함수와 함께 사용할 수 없습니다. 자세한 내용은 http://www.oracle.com/pls/as111170/lookup?id=OSBAG444에서 Oracle Fusion Middleware Administrator's Guide for Oracle Service Bus의 "XQuery Implementation"을 참조하십시오.
      • 주: 프록시가 활성 WSS 중간 단계인 경우, WS 보안을 사용하여 WS 보안 사용자 이름 토큰 또는 사용자 정의 사용자 이름/비밀번호를 암호화할 수 있습니다. 이 경우 프록시가 먼저 요청을 해독하고 일반 텍스트로 된 사용자 이름/비밀번호에 액세스하기 때문에 사용자 이름/비밀번호 통과가 작동합니다.
  • 사용자 매핑

    • 클라이언트에서 인바운드 요청을 인증한 결과인 사용자 이름(로컬 사용자 이름)과 지정하는 사용자 이름과 비밀번호(원격 사용자 이름과 비밀번호)의 상관 관계를 지정(매핑)해야 합니다. 서비스 계정이 매핑되고 인증된 클라이언트의 요청을 받으면 비즈니스 서비스 또는 프록시 서비스 아웃바운드 요청에 적절한 원격 사용자 이름과 비밀번호를 제공합니다.

    전송 수준과 메시지 수준 모두에서 클라이언트가 인증하면 서비스 계정은 메시지 수준 사용자 이름을 원격 사용자 이름과 비밀번호에 매핑합니다.

    또한, 익명 사용자 이름을 원격 사용자 이름과 비밀번호에 매핑할 수 있습니다.

    이 방법에는 다음 제한 사항이 적용됩니다.

    • Oracle Service Bus를 FTP 서버 또는 JMS 서버 등의 로컬 또는 원격 서버나 시스템 리소스에 인증하는 아웃바운드 요청에서는 사용할 수 없습니다.
    • fn-bea:lookupBasicCredentials XQuery 함수와 함께 사용할 수 없습니다. 자세한 내용은 http://www.oracle.com/pls/as111170/lookup?id=OSBAG444에서 Oracle Fusion Middleware Administrator's Guide for Oracle Service Bus의 "XQuery Implementation"을 참조하십시오.
