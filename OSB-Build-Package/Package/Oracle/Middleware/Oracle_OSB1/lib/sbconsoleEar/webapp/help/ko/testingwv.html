프록시 서비스 테스트 데이터 구성

표 32-1에는 프록시 서비스 테스트를 위한 구성 옵션이 나열됩니다. 요청 문서에 대한 입력값을 적용하는 데 사용되는 필드는 서비스 유형을 기반으로 합니다.

표 32-1 프록시 서비스 구성 옵션
섹션
옵션/필드
설명
이름
테스트할 프록시 서비스의 이름입니다.
사용 가능한 작업
프록시 서비스와 관련된 작업입니다.
테스트 콘솔 작업
실행
테스트를 실행합니다.
재설정
입력 값을 재설정합니다.
닫기
테스트 콘솔을 닫고 테스트를 실행하지 않습니다.
테스트 구성
테스트 구성 옵션을 설정합니다.
직접 호출
Oracle Service Bus 전송 계층을 사용하지 않고 메시지를 프록시 서비스에 전송합니다. 테스트 콘솔에 대한 입력 데이터는 클라이언트에서 프록시 서비스로 전송되는 데이터여야 합니다.
직접 호출의 반대는 간접 호출입니다. 간접 호출은 직접 호출 옵션을 선택 해제하여 호출할 수 있습니다. 이 작업은 기본적으로 비즈니스 서비스에 대해 수행됩니다. 간접 호출은 전송 계층을 통해 메시지를 전송합니다. 이 경우, 테스트 콘솔에 대한 입력 데이터는 프록시 서비스에서 호출된 서비스로 전송되는 데이터여야 합니다.
추적 포함
추적은 시스템을 통해 전달하는 메시지의 상태를 보여줍니다.
요청 문서
입력 필드는 프록시 서비스로 전송되는 요청 메시지를 생성합니다. 입력한 값을 사용하여 테스트를 실행하려면 실행을 누릅니다. 테스트 콘솔에는 요청 메시지, 서비스의 응답 메시지 및 메타 데이터가 표시됩니다.
요청 문서 섹션에서의 입력은 서비스 유형별로 다릅니다. 서비스 유형 및 각 서비스 유형에 필요한 입력에 대한 설명은 다음 섹션에 나열됩니다.
모든 XML
페이로드 형식의 서비스 요청 입력입니다. 페이로드는 전송되는 메시지의 내용입니다. 이 내용은 XML 문서여야 합니다. 파일을 찾아보거나 텍스트 상자에 메시지 내용을 입력할 수 있습니다.
요청 문서 계속
모든 SOAP
페이로드 형식의 서비스 요청 입력입니다. 페이로드는 전송되는 메시지의 내용입니다. 이 내용은 SOAP 엔벨로프여야 합니다. 파일을 찾아보거나 텍스트 상자에 메시지 내용을 입력할 수 있습니다.
메시징
메시징 서비스는 없음, XML, 이진 또는 텍스트 등 4개의 가능한 입력 유형을 정의합니다. 서비스는 어떤 입력도 필요하지 않은 없음 유형을 제외하고 단일 입력(파일 기반 또는 텍스트 기반 중 하나)을 요청합니다.
파일에서 이진 입력으로 입력하는 것이 좋습니다. 텍스트 영역에 입력한 데이터는 시스템 인코딩을 사용하여 이진으로 변환됩니다.
텍스트 서비스에 대해 파일에서 입력된 데이터는 텍스트로 변환해야 합니다. 인코딩 입력 필드는 변환 시 적용할 인코딩을 지정하는 데 사용됩니다. 이 필드가 구성되지 않은 경우 시스템 인코딩이 사용됩니다. 기본적으로 인코딩 필드는 프록시 서비스 끝점에서 구성된 인코딩 값으로 초기화됩니다.
XML
페이로드 형식의 서비스 요청 입력입니다. 페이로드는 전송되는 메시지의 내용입니다. 이 내용은 XML 문서여야 합니다. 파일을 찾아보거나 텍스트 상자에 메시지 내용을 입력할 수 있습니다.
프록시 서비스가 다중 작업이 정의된 WSDL 기반 서비스인 경우, 테스트 콘솔은 서비스를 테스트할 때 사용할 샘플 문서를 생성 및 제공합니다. 이 샘플 데이터를 직접 사용하여 편집하고 테스트를 실행하거나 사용자가 고유한 테스트 데이터를 제공할 수 있습니다.
모든 작업은 페이지 맨 위에 나열됩니다.
요청 문서 계속
SOAP 문서
SOAP 문서에 대해 SOAP 엔벨로프는 일반적으로 0개 이상의 머리글과 1개의 본문 페이로드로 구성됩니다. 폼 및 XML 탭은 내용을 지정할 수 있는 대체 방법을 제공합니다.
폼 탭에는 SOAP 머리글 필드와 SOAP 본문 필드가 포함되어 있습니다. SOAP 머리글 필드의 내용은 SOAP 머리글 태그(둘 이상의 머리글을 정의할 수 있음)여야 합니다. SOAP 본문 필드에는 실제 메시지의 일부로 전송되는 데이터가 포함되어 있습니다. 이 내용은 XML 문서여야 합니다. 머리글과 본문은 모두 SOAP 엔벨로프를 생성하는 데 사용됩니다.
요청 문서 계속
SOAP RPC
SOAP RPC에 대해 SOAP 엔벨로프는 0개 이상의 머리글과 0개 이상의 인수로 구성됩니다.
폼 및 XML 탭은 내용을 지정할 수 있는 대체 방법을 제공합니다.
폼 탭에는 SOAP 머리글에 대한 단일 입력과 각 인수에 대한 1개의 입력 필드(인수 이름에 해당하는 입력 필드 이름)가 포함되어 있습니다. SOAP 머리글 필드의 내용은 SOAP 머리글 태그(둘 이상의 머리글을 정의할 수 있음)여야 합니다.
WSDL은 각 인수의 유형을 감지하는 데 사용됩니다. 단일 라인 입력 필드는 기본 유형에 사용되고, 복수 라인 입력 필드는 XML 유형에 사용됩니다. 샘플 문서가 자동으로 생성되어 테스트에 도움이 됩니다.
머리글과 인수는 SOAP 엔벨로프를 생성하기 위해 테스트 콘솔에서 사용됩니다.
XML 탭에는 단일 입력 필드가 포함되어 있습니다. 이 필드의 내용은 전송되는 SOAP 엔벨로프여야 합니다. 페이로드(XML 입력)는 파일 기반 또는 텍스트 기반일 수 있습니다. 입력에 대한 파일 참조는 텍스트 입력보다 우선합니다. 테스트에 사용할 파일을 찾아서 선택합니다.
웹 서비스 보안
이 섹션은 선택한 작업에 웹 서비스 보안(WSS) 정책이 존재하는 경우 SOAP 서비스에 대해서만 사용할 수 있습니다.
서비스 제공자
테스트 서비스는 이 서비스 제공자로부터 웹 서비스 보안 작업(디지털 서명 및 암호화)에 대한 모든 클라이언트측 PKI(키 쌍) 인증서를 가져옵니다. 이 필드는 선택 사항입니다. 특정 시나리오에는 작업의 요청/응답 정책에 따라 서비스 제공자가 필요합니다. 자세한 내용은 http://www.oracle.com/pls/as111170/lookup?id=OSBAG384에서 Oracle Fusion Middleware Administrator's Guide for Oracle Service Bus의 "Testing Services with Web Service Security"를 참조하십시오.
사용자 이름
웹 서비스 보안 사용자 이름 토큰의 사용자 이름은 테스트 서비스에서 생성되었습니다. 이 필드는 선택 사항입니다. 이 사용자 이름은 작업의 요청 정책이 ID 어설션을 지정하는 일부 시나리오에서만 필요합니다.
이 필드를 전송 보안 컨텍스트 사용자 이름 필드와 혼동하지 마십시오.
로컬 보안 영역의 사용자 이름과 비밀번호는 적합해야 합니다. 사용자 이름이나 비밀번호가 부적합하면 테스트 서비스에서 클라이언트측 오류가 발생합니다.
비밀번호
테스트 서비스에서 생성된 웹 서비스 보안 사용자 이름 토큰의 비밀번호입니다.
전송
기본적으로 전송 섹션은 축소되어 있습니다. 필드 및 값은 테스트 구성에 따라 다릅니다.
인증
사용자 이름
프록시 서비스를 호출하는 경우 테스트 서비스에서 사용하는 보안 컨텍스트 설정의 사용자 이름입니다.
프록시 서비스가 SAML 토큰을 필요로 하는 비즈니스 서비스로 메시지 경로를 지정할 경우 사용자 이름은 토큰이 나타낼 ID입니다. 자세한 내용은 http://www.oracle.com/pls/as111170/lookup?id=OSBDV1647에서 Oracle Fusion Middleware Developer's Guide for Oracle Service Bus의 "Using SAML for Authentication"을 참조하십시오.
이 필드를 웹 서비스 보안(WSS) 사용자 이름 필드와 혼동하지 마십시오. 로컬 보안 영역의 사용자 이름과 비밀번호는 적합해야 합니다. 사용자 이름이나 비밀번호가 부적합하면 테스트 서비스에서 클라이언트 측 오류가 발생합니다.
주: 테스트 콘솔이 HTTP 사용자 정의 토큰 인증을 사용한 프록시를 호출한 경우, 인증 점검이 완료되지 않습니다.
비밀번호
연관된 비밀번호입니다. 자세한 내용은 사용자 이름을 참조하십시오.
호출 모드
요청/응답
이 옵션은 모든 SOAP 또는 모든 XML 프록시 서비스를 테스트하는 경우에만 표시됩니다. 단방향 서비스 호출인 경우 요청/응답 체크 박스를 선택 취소합니다.
메시지 메타 데이터
표 32-10을 참조하십시오.
전송 머리글
표 32-10을 참조하십시오.

주: 보안 SOAP 메시지는 별도의 공백에 인쇄되어 표시됩니다. 공백은 문서 구문에 영향을 줄 수 있으므로 이 SOAP 메시지를 리터럴 데이터로 항상 사용할 수 있는 것은 아닙니다. 예를 들어, 디지털 서명은 공백을 구분하며 부적합하게 될 수 있습니다.
