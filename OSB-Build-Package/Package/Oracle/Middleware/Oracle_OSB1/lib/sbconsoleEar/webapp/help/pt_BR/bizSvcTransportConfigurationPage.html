Página Configuração do Transporte

A página Criar/Editar um Serviço de Negócios - Configuração do Transporte pertence a uma série de páginas para criação e edição de serviços de negócios, conforme descrito em Criando e Configurando Serviços de Negócios e Editando Configurações do Serviço de Negócios.

Use esta página para selecionar um protocolo de transporte para o serviço de negócios e para estabelecer outras definições gerais de configuração de transporte.

A segurança de transporte de saída aplica-se às conexões entre os serviços de proxy e os serviços de negócios do Oracle Service Bus. Para obter mais informações sobre segurança no nível de transporte, consulte "Configuring Transport-Level Security" no Oracle Fusion Middleware Developer's Guide for Oracle Service Bus em http://www.oracle.com/pls/as111170/lookup?id=OSBDV1557.

Tabela 18-3 descreve como usar esta página.

Tabela 18-3 - Página Criar/Editar um Serviço de Negócios - Configuração do Transporte
Opção
Para criar ou editar...
Protocolo
Selecione um protocolo de transporte na lista. Os protocolos disponíveis variam de acordo com o tipo de serviço que você está criando:
  • Web Service de WSDL: soa-direct, bpel-10g, dsp, http, jca, jms, sb, ws
  • Serviço do Tipo de Transporte: ejb, flow, jejb
  • Serviço de Mensagens: e-mail, arquivo, ftp, http, jms, mq (se disponível), sftp, tuxedo
  • Qualquer Serviço SOAP: dsp, http, jms, sb
  • Qualquer Serviço XML: dsp, e-mail, arquivo, ftp, http, jms, mq (se disponível), sb, sftp, tuxedo
Algoritmo de Balanceamento de Carga
Selecione um destes algoritmos de balanceamento de carga:
  • Revezamento - Este algoritmo ordena dinamicamente os URLs informados no campo URI do Ponto Final para este serviço de negócios. Se o primeiro falhar, será tentado o próximo, e assim por diante, até que a contagem de novas tentativas seja esgotada.

    • Para cada mensagem nova, há uma nova ordem de URLs.

  • Aleatório - Esse algoritmo ordena aleatoriamente a lista de URLs informados no campo URI do Ponto Final deste serviço de negócios. Se o primeiro falhar, será tentado o próximo, e assim por diante, até que a contagem de novas tentativas seja esgotada.
  • Aleatório-ponderado - Esse algoritmo ordena aleatoriamente a lista de URLs informados no campo URI do Ponto Final deste serviço de negócios, mas alguns são tentados novamente mais do que outros, dependendo do valor informado no campo Peso.
  • Nenhum - Este algoritmo ordena a lista de URLs informados no campo URI do Ponto Final deste serviço de negócios de cima para baixo.
URI do Ponto Final
Informe um URL de ponto final no formato com base no protocolo de transporte selecionado no campo Protocolo. A seguir, estão as descrições dos formatos de URI para cada transporte. Elementos opcionais do URI são mostrados entre parênteses []. Para obter mais informações sobre URIs de transporte, consulte os respectivos capítulos de transporte no Oracle Fusion Middleware Developer's Guide for Oracle Service Bus em http://www.oracle.com/pls/as111170/lookup?id=OSBDV.
  • bpel-10g - protocol://host[:port] [/protocol-path]/domain/process[/version[/partnerlink/role]]
  • dsp - t3://dsp-ip-address:port/dsp-app-name
  • ejb - ejb:jndi_provider_name:ejb_jndi_name

    • No URI, jndi_provider_name é o nome do recurso do provedor de JNDI e ejb_jndi_name é o nome de JNDI no servidor JNDI do EJB.

    Se o provedor de JNDI estiver localizado no mesmo servidor, o provedor de JNDI não precisará ser especificado. O URI seria ejb::ejb_jndi_name.

    Observação: Se seus EJBs estiverem sendo executados no IBM WebSphere, ejb_jndi_name deverá estar no seguinte formato:

    cell/nodes/node_name/servers/server_name/ejb_jndi_name

    ou

    cell/clusters/cluster_name/ejb_jndi_name

    conforme descrito na documentação do IBM WebSphere, no endereço http://publib.boulder.ibm.com/infocenter/wasinfo/v6r1/index.jsp?topic=/com.ibm.websphere.nd.iseries.doc/info/iseriesnd/ae/rnam_example_prop3.html.

  • e-mail - É possível informar vários URIs de ponto final em qualquer combinação dos seguintes formatos:
    • mailto:name@domain_name.com
    • mailto:name@domain_name.com?smtp=smtp_server_resource
    • mailto:name@domain_name.com?mailsession=jndi_mail_session
  • arquivo - file:///drivename:/somename
  • fluxo - fluxo:<caminho de referência para o recurso split-join>

    • caminho de referência para o recurso split-join é o caminho (projeto/pasta) e nome do Split-Join; por exemplo, batchorderProcessing/batchorder.

  • ftp - ftp://host:port/directory
  • http - http://host:port/someService

    • O transporte HTTP suporta pontos finais HTTP e HTTPS.

  • jca - jca://<resource_adapter_jndi>
  • jejb - jejb:jndi_provider_name:ejb_jndi_name

    • Exemplo de EJB 2.1: jejb:myProvider:osb.jejb.myJejbBiz21

    Exemplo de EJB 3.0: jejb:myProvider:myBiz31#osb.jejb.myJejbBiz

    Observação: Se seus EJBs estiverem sendo executados no IBM WebSphere, ejb_jndi_name deverá estar no seguinte formato:

    cell/nodes/node_name/servers/server_name/ejb_jndi_name

    ou

    cell/clusters/cluster_name/ejb_jndi_name

    conforme descrito na documentação do IBM WebSphere, no endereço http://publib.boulder.ibm.com/infocenter/wasinfo/v6r1/index.jsp?topic=/com.ibm.websphere.nd.iseries.doc/info/iseriesnd/ae/rnam_example_prop3.html.

  • jms - jms://host:port/factoryJndiName/destJndiName

    • Para direcionar um destino JMS para vários servidores, use o seguinte formato:
    jms://host1:port,host2:port/QueueConnectionFactory/DestName

    Observação: Embora o Oracle WebLogic Server permita o uso de barras em nomes JNDI, como "myqueues/myqueue", os nomes JNDI com barras interferem no formato de URI exigido pelo Oracle Service Bus e você não pode usá-los. Para resolver esse problema, defina e referencie um servidor externo JMS no URI. Para obter mais informações, consulte "Configure foreign servers" em Oracle Fusion Middleware Oracle WebLogic Server Administration Console Online Help em http://www.oracle.com/pls/as111170/lookup?id=WLACH.

    Em um cluster: os nomes de host no URI JMS devem corresponder exatamente aos nomes de host dos servidores de cluster, conforme configurados no WebLogic Server.

URI do Ponto Final (continuação)
  • mq - mq://<local-queue-name>?conn=<mq-connection-resource-ref>

    • local-queue-name é o nome da fila MQ a partir da qual o serviço de negócios lê mensagens.

    mq-connection-resource-ref é o caminho (projeto/pasta) e o nome do recurso de conexão MQ; por exemplo, default/my_MQconnection.

    Observação: O URI do Ponto Final não pode conter espaços, portanto não crie recursos de Conexão MQ ou projetos/pastas com espaços nos nomes.

    Para disponibilizar o transporte MQ no Oracle Service Bus, consulte Conexões MQ.

  • sb - sb://[<jndi_provider_name/>]service_name

    • jndi_provider_name (opcional) é o nome do recurso Provedor de JDNI do Oracle Service Bus. Quando omitido, é usado o contexto default.

    service_name é um serviço de destino e corresponde ao URI do serviço de proxy remoto.

  • sftp - sftp://hostname:port/directory
  • soa-direct - [protocol://authority]/default/compositeName[!versionNumber[*label]]/serviceName

    • O protocolo e a autoridade são opcionais quando os serviços SOA estão colocalizados no mesmo servidor do Oracle Service Bus.

  • tuxedo - tuxedo:resourcename[/remotename]

    • No URI, resourcename corresponde a um nome de Importação do WTC e remotename corresponde ao nome do serviço exportado pelo domínio remoto do Tuxedo. O URI resourcename é obrigatório e o remotename é opcional.

    Se mais de um URI for especificado, você deverá ter nomes de recurso exclusivos para os pontos finais. Se nenhum nome remoto for especificado, seu valor será o valor do nome do recurso. Se nenhum nome remoto for informado ou se o nome remoto e o nome do recurso forem iguais, apenas um URI será permitido. Nesse caso, o nome do recurso e o nome remoto terão o mesmo valor. Isso permite que Importações do WTC já definidas usem o balanceamento de carga e o failover do WTC.

  • ws - http://host:port/somename ou https://host:port/somename
Clique em Adicionar para adicionar ou um mais URIs extras. No run-time, os URLs são selecionados com base no algoritmo de balanceamento de carga selecionado no campo Algoritmo de Balanceamento de Carga.
Se você selecionou Ponderado-aleatório no campo Algoritmo de Balanceamento de Carga, também poderá informar um peso no campo URI do Ponto Final. O default é 1.
Se você tiver vários pontos finais definidos e selecionou Nenhum no campo Carregar Algoritmo de Balanceamento, a ordem dos pontos finais será significante. É possível reordenar os pontos finais usando as setas Para Cima e Para Baixo na coluna Opções.
O Oracle Service Bus não suporta URIs do ponto final duplicados no mesmo serviço de negócios.
Contagem de Repetições
No caso de falha de entrega durante o envio de solicitações de saída, especifique o número de vezes para repetir pontos finais do URL individuais; em outras palavras, o número de tentativas de failover.
Por exemplo, um serviço de negócios possui um URI (U1) configurado e o número de novas repetições está definido como 3. Se o U1 falhar na primeira tentativa, o sistema tentará novamente o ponto final U1 3 vezes mais.
Se um serviço de negócios tiver 2 URIs configurados (U1 e U2) e uma contagem de nova tentativa 3, se a primeira tentativa (por exemplo, em U1) falhar, o sistema tentará (fará failover em) o próximo URI (U2). Se ele também falhar, o sistema fará mais duas tentativas, uma vez em U1 e outra vez em U2.
Intervalo de Iteração de Repetição
Especifique o número de segundos que o sistema pausa antes de iteragir sobre todos os URIs do ponto final na lista novamente.
Por exemplo, um serviço de negócios possui dois URIs configurados (U1 e U2) e uma contagem de novas tentativas de 2 com um intervalo de iteração de nova tentativa de 5 segundos. Se a primeira tentativa (no U1) falhar, o sistema tentará o U2 imediatamente. Mas, se o U2 também falhar, o sistema aguardará 5 segundos e tentará o U1 novamente.
Repetir Erros da Aplicação
Selecione Sim ou Não.
No caso de falha de entrega durante o envio de solicitações de saída, especifique se ou não repetir URIs do ponto final com base em erros da aplicação (por exemplo, uma falha SOAP).
Este campo só é exibido para estes transportes: HTTP, EJB, JMS, DSP, Tuxedo, SB e WS.
