Página Configuração do Transporte

A página Criar/Editar um Serviço de Proxy - Configuração do Transporte pertence a uma série de páginas para criação e edição de serviços de proxy, conforme descrito em Criando e Configurando Serviços de Proxy e Editando Configurações do Serviço de Proxy.

Use esta página para selecionar um protocolo de transporte para o serviço de proxy e para estabelecer outras definições gerais de configuração de transporte. A Tabela 19-3 descreve como usa a página.

Observação: A segurança de transporte de entrada aplica-se a aplicações clientes e a serviços de proxy do Oracle Service Bus. A segurança de transporte de saída aplica-se às conexões entre os serviços de proxy e os serviços de negócios do Oracle Service Bus. Para saber mais sobre segurança de transporte, consulte "Configuring Transport-Level Security" no Oracle Fusion Middleware Developer's Guide for Oracle Service Bus em http://www.oracle.com/pls/as111170/lookup?id=OSBDV1557.

Protocolo
Selecione um protocolo de transporte na lista. Os protocolos disponíveis variam de acordo com o tipo de serviço que você está criando:
  • Web Service de WSDL: http, jca, jms, local, sb, ws
  • Tipo de Transporte: jejb
  • Serviço de Mensagens: e-mail, arquivo, ftp, http, jms, local, mq (se disponível), sftp, tuxedo
  • Qualquer Serviço SOAP: http, jms, local, sb
  • Qualquer Serviço XML: e-mail, arquivo, ftp, http, jms, local, mq (se disponível), sb, sftp, tuxedo
URI do Ponto Final
Informe um URI de ponto final no formato baseado no protocolo de transporte selecionado no campo Protocolo, acima: Os formatos são:
  • email - mailfrom:mail-server-hostname:mail-server-port
  • arquivo - file:///drivename:/somename
  • ftp - ftp://hostname:port/directory
  • http - someName

    • O transporte HTTP suporta pontos finais HTTP e HTTPS.

  • jca - jca://<resource_adapter_jndi>
  • jejb - O formato do URL é ejb_jndi_name. O URI configurado para um serviço de proxy JEJB torna-se o nome da JNDI global para localização do bean de sessão sem preservação de estado gerado pelo transporte JEJB da interface remota/de negócios no cliente JAR.

    • Observação: Para o EJB 3.0, ejb_jndi_name é o atributo mappedName da anotação @javax.ejb.Stateless no bean gerado. O nome da JNDI de pesquisa para o serviço do EJB gerado será sufixado com #interface_class, que é o nome totalmente qualificado da interface de negócios.

    É possível acessar o serviço de proxy JEJB como:

    EJB 2.1: protocol://host:port/ejb_jndi_name

    EJB 3.0: protocol://host:port/ejb_jndi_name#interface_class

    Observação: Se seus EJBs estiverem sendo executados no IBM WebSphere, ejb_jndi_name deverá estar no seguinte formato:

    cell/nodes/node_name/servers/server_name/ejb_jndi_name

    ou

    cell/clusters/cluster_name/ejb_jndi_name

    conforme descrito na documentação do IBM WebSphere, no endereço http://publib.boulder.ibm.com/infocenter/wasinfo/v6r1/index.jsp?topic=/com.ibm.websphere.nd.iseries.doc/info/iseriesnd/ae/rnam_example_prop3.html.

  • jms - jms://host:port/factoryJndiName/destJndiName

    • Para direcionar um destino JMS para vários servidores, use o seguinte formato:

    jms://host1:port,host2:port/QueueConnectionFactory/DestName

    Observe que, ao criar um serviço de proxy, você pode configurar um URI de ponto final JMS, mesmo que o servidor nesse ponto final não esteja disponível. No entanto, no caso do JMS, quando você ativa a sessão, o ponto final deve estar disponível. Para saber mais, consulte Ativando Sessões.

    Observação: Embora o Oracle WebLogic Server permita o uso de barras em nomes JNDI, como "myqueues/myqueue", os nomes JNDI com barras interferem no formato de URI exigido pelo Oracle Service Bus e você não pode usá-los. Para resolver esse problema, defina e referencie um servidor externo JMS no URI. Para obter mais informações, consulte "Configure foreign servers" em Oracle Fusion Middleware Oracle WebLogic Server Administration Console Online Help em http://www.oracle.com/pls/as111170/lookup?id=WLACH.

    Em um cluster: os nomes de host no URI JMS devem corresponder exatamente aos nomes de host dos servidores de cluster, conforme configurados no WebLogic Server.

  • local

    • Este transporte não requer um URI de ponto final.

  • mq - mq://<local-queue-name>?conn=<mq-connection-resource-ref>

    • local-queue-name é o nome da fila MQ a partir da qual o serviço de proxy lê mensagens.

    mq-connection-resource-ref é o caminho (projeto/pasta) e o nome do recurso de conexão MQ; por exemplo, default/my_MQconnection.

    Observação: O URI do Ponto Final não pode conter espaços, portanto não crie recursos de Conexão MQ ou projetos/pastas com espaços nos nomes.

    Para disponibilizar o transporte MQ no Oracle Service Bus, consulte Conexões MQ.

URI do Ponto Final (continuação)
  • sb - service_name

    • service_name é o identificador exclusivo para o serviço de proxy. Por default, esse nome será o nome do serviço de proxy.

    service_name só deve conter caracteres permitidos em URIs (conforme descrito em RFC2396 em http://www.ietf.org/rfc/rfc2396.txt), exceto que não pode conter caracteres de barra (/) nem de dois-pontos (:).

  • sftp - sftp://hostname:port/directory
  • tuxedo - exportname

    • O URI exportname corresponde a uma Exportação do WTC que o domínio remoto do Tuxedo identifica como um serviço do Tuxedo.

    Se mais de um URI for especificado, você deverá ter nomes de recurso exclusivos para os pontos finais. Se nenhum nome remoto for especificado, seu valor será o valor do nome do recurso. Se nenhum nome remoto for informado ou se o nome remoto e o nome do recurso forem iguais, apenas um URI será permitido. Nesse caso, o nome do recurso e o nome remoto terão o mesmo valor. Isso permite que os usuários usando Importações do WTC já definidas usem o balanceamento de carga e o failover do WTC.

    Observação: Se você configurar dois URIs idênticos, um erro indicará que o nome do serviço já existe.

  • ws - /contextPath

    • contextPath deve ser exclusivo para serviços de proxy que usam o transporte HTTP ou WS.

Obter Todos os Cabeçalhos
Selecione Sim para recuperar todos os cabeçalhos do transporte.
Selecione Não para recuperar um conjunto definido de cabeçalhos. Se você selecionar Não, informe um conjunto de cabeçalhos no campo Cabeçalho, depois clique em Adicionar. (Esta etapa não se aplica ao transporte Local.)
Observação: O Oracle Service Bus não transmite o cabeçalho de Autorização HTTP da solicitação para o pipeline, porque esta ação abre uma vulnerabilidade de segurança. Você pode inadvertidamente criar uma ação de log que grave o nome de usuário e a senha não criptografada em um arquivo de log. Se o padrão de design exigir que o cabeçalho de Autorização HTTP esteja no pipeline, faça o seguinte:

a. No comando de inicialização do Oracle Service Bus, defina a seguinte propriedade do sistema como verdadeira: com.bea.wli.sb.transports.http.GetHttpAuthorizationHeaderAllowed

b. Na Console do Oracle Service Bus Administration, na página Configuração do Transporte, selecione Obter Todos os Cabeçalhos ou Cabeçalhos Especificados pelo Usuário e especifique Autorização.
c. Reinicie o Oracle Service Bus.

O Oracle Service Bus transmitirá o cabeçalho de Autorização para o pipeline.
