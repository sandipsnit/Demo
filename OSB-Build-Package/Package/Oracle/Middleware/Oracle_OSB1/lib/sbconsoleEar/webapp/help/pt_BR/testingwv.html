Configurando Dados do Teste de Serviços de Proxy

Tabela 32-1 lista as opções de configuração para teste dos serviços de proxy. Os campos para aceitar entrada no documento de solicitação são baseados no tipo de serviço.

Tabela 32-1 - Opções de Configuração de Serviços de Proxy

Opções/Campos
Descrição
Nome

Operações associadas ao serviço de proxy.

Execute o teste.

Redefina os valores de entrada.

Configuração de Teste
Defina as opções de configuração de teste.
Chamada Direta
Envia uma mensagem para o serviço de proxy sem usar a camada de transporte do Oracle Service Bus. Os dados de entrada na Console de Teste devem ser os enviados do cliente para o serviço de proxy.
O oposto de uma chamada direta é uma chamada indireta. Pode ser chamada com a desmarcação da opção Chamada Direta. Por default, é executada para serviços de negócios. A chamada indireta envia mensagens por meio da camada de transporte. Nesse caso, os dados de entrada na Console de Teste devem ser aqueles que estão sendo enviados de um serviço de proxy para o serviço chamado.
Incluir Rastreamento
O rastreamento mostra o estado da mensagem à medida que passa pelo sistema.
Documento de Solicitação
Os campos de entrada geram a mensagem de solicitação que é enviada para o serviço de proxy. Clique em Executar para executar o teste com os valores informados. A Console de Teste exibe a mensagem de solicitação e a mensagem de resposta do serviço, além dos metadados.
A entrada na seção Documento da Solicitação é específica ao tipo de serviço. Os tipos de serviço e uma descrição da entrada necessária para cada um são listados nas seções a seguir.
qualquer XML
A entrada de solicitações de serviço sob a forma de payload. O payload é o conteúdo da mensagem que está sendo enviada. O conteúdo deve ser um documento XML. É possível navegar para um arquivo ou informar o conteúdo da mensagem na caixa de texto.
Documento de Solicitação - continuação
qualquer SOAP
A entrada de solicitações de serviço no formato de um payload. O payload é o conteúdo da mensagem sendo enviada. O conteúdo deve ser o envelope SOAP. É possível navegar para um arquivo ou informar o conteúdo da mensagem na caixa de texto.
Mensagens
Os serviços de mensagens definem 4 tipos de entrada possíveis: nenhum, XML, Binário ou Texto. O serviço solicita uma única entrada, baseada em arquivo ou em texto, exceto o tipo nenhum, que não requer nenhuma entrada.
A Oracle recomenda informar entrada binária de um arquivo. Os dados informados na área de texto são convertidos em entrada binária usando a codificação do sistema.
Os dados informados de arquivos para serviços de texto devem ser convertidos em texto. O campo de entrada de codificação é usado para especificar a codificação a ser aplicada durante a conversão. A codificação do sistema é usada se esse campo não for configurado. Por default, o campo de codificação é inicializado com o valor de codificação configurado no ponto final do serviço de proxy.
XML
A entrada de solicitações de serviço sob a forma de payload. O payload é o conteúdo da mensagem que está sendo enviada. O conteúdo deve ser um documento XML. É possível navegar para um arquivo ou informar o conteúdo da mensagem na caixa de texto.
Se o serviço de proxy for um serviço baseado no WSDL com várias operações definidas, a Console de Teste gerará e fornecerá um documento de amostra a ser usado durante o teste do serviço. É possível usar esses dados de amostra diretamente, editá-los e depois executar o teste, ou fornecer seus próprios dados de teste.
Todas as operações são listadas na parte superior da página.
Documento de Solicitação - continuação
Documento SOAP
Para um documento SOAP, o envelope SOAP geralmente é composto por zero ou mais cabeçalhos e um payload do corpo. As guias Form e XML oferecem caminhos alternativos para especificar o conteúdo.
A guia Form contém um campo Cabeçalho SOAP e um campo Corpo SOAP. O conteúdo do campo Cabeçalho SOAP deve ser uma tag de Cabeçalho SOAP (isso permite a definição de mais de um cabeçalho). O campo Corpo SOAP contém os dados que são realmente enviados como parte da mensagem. O conteúdo deve ser um documento XML. Tanto o cabeçalho quanto o corpo são usados para gerar o envelope SOAP.
Documento de Solicitação - continuação
RPC SOAP
Para o RPC SOAP, o envelope SOAP é composto por zero ou mais cabeçalhos e por zero ou mais argumentos.
As guias Form e XML oferecem caminhos alternativos para especificar o conteúdo.
A guia Form contém uma única entrada para cabeçalhos SOAP e um campo de entrada para cada argumento (o nome do campo de entrada corresponde ao nome do argumento). O conteúdo do campo Cabeçalho SOAP deve ser uma tag de Cabeçalho SOAP (isso permite a definição de mais de um cabeçalho).
O WSDL é usado para detectar o tipo de cada argumento. Um campo de entrada de linha única é usado para tipos primitivos; um campo de entrada de várias linhas é usado para tipos XML. Um documento de amostra é gerado automaticamente para facilitar os testes.
Os cabeçalhos e argumentos são usados pela Console de Teste para gerar o envelope SOAP.
A guia XML contém um único campo de entrada. O conteúdo desse arquivo deve ser o envelope SOAP sendo enviado. O payload (entrada XML) pode ser baseado em arquivo ou baseado em texto. A referência de um arquivo para entrada tem precedência sobre entrada textual. Pesquise e selecione o arquivo que você deseja usar no teste.
Segurança do Web Service
Esta seção só está disponível para serviços SOAP quando a operação selecionada tem uma política de Segurança de Web Services (WSS).
Provedor de Serviços
O serviço de teste obtém todas as credenciais do PKI (par de chaves) do lado do cliente para operações de segurança de Web Services (assinatura digital e criptografia) deste provedor de serviço. Esse campo é opcional. Determinados cenários requerem um provedor de serviços, dependendo da política de solicitação/resposta para a operação. Para obter mais informações, consulte "Testing Services with Web Service Security" no Oracle Fusion Middleware Administrator's Guide for Oracle Service Bus em http://www.oracle.com/pls/as111170/lookup?id=OSBAG384.
Nome do Usuário
O nome de usuário em tokens de nome de usuário de segurança de Web Services gerados a partir do serviço de teste. Esse campo é opcional. Esse nome de usuário só é necessário em alguns cenários em que a política de solicitação da operação especifique uma asserção de identidade.
Não confunda este campo com o campo do nome de usuário do contexto de segurança do transporte.
Deve ser um nome de usuário e senha válidos no realm de segurança local. Um nome de usuário ou senha inválido causará um erro do lado do cliente no serviço de teste.
Senha
A senha em tokens do nome do usuário de segurança do Web Service gerados com base no serviço de teste.
Transporte
A seção Transporte está recolhida por default. Os campos e valores dependem da configuração de teste.
Autenticação
Nome do Usuário
O nome de usuário para configuração do contexto de segurança usado pelo serviço de teste ao chamar o serviço de proxy.
Se o serviço de proxy encaminhar a mensagem para um serviço de negócios que espera um token SAML, essa será a identidade que será representada pelo token. Para obter mais informações, consulte "Using SAML for Authentication" no Oracle Fusion Middleware Developer's Guide for Oracle Service Bus em http://www.oracle.com/pls/as111170/lookup?id=OSBDV1647.
Não confunda este campo com o campo do nome de usuário da Segurança do Web Service (WSS). Deve ser um nome de usuário e senha válidos no realm de segurança local. Um nome de usuário ou senha inválido causará um erro do lado do cliente no serviço de teste.
Observação: Quando a Console de Teste chama um proxy com autenticação personalizada de token HTTP, a verificação de autenticação não é executada.
Senha
A senha associada. Para obter mais informações, consulte Nome de Usuário.
Modo de Chamada
Solicitação/Resposta
Esta opção só é exibida durante o teste de qualquer serviço de proxy SOAP ou de qualquer XML. Desmarque a caixa de seleção Solicitação/Resposta para chamadas de serviço unilaterais.
