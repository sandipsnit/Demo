了解用户、组、安全角色和策略

此部分包含以下主题：

  • 用户
  • 组
  • 角色
  • 访问控制策略
  • 安全配置数据和会话

用户

用户是可以进行验证的实体。用户可以是某个人或某个软件实体，如 Web 服务客户机。必须向每个用户赋予一个此安全领域中唯一的标识（名称）。

通常，您创建的用户可分为两类：

  • 可以访问您的代理服务或业务服务的客户机用户。

    • 如果创建了大量的客户机用户，请考虑将其组织为安全组。

  • 可使用 Oracle Service Bus 管理控制台创建或修改代理服务、业务服务以及其他 Oracle Service Bus 资源的管理用户。

    • Oracle Service Bus 对其管理功能使用基于角色的安全。Oracle Service Bus 仅将管理权限授予安全角色，而不是直接将访问权限授予用户。要将管理权限授予用户，应将用户置于一个默认的安全组中（该安全组在预定义的一个安全角色中）。

组

为了能够便利地管理大量用户，可将用户组织成命名组。然后，向组授予权限或标识，而不是向单独的用户授予访问权限或角色标识。

管理安全组

Oracle Service Bus 提供了默认安全组以便于向用户授予对管理功能（如创建代理服务）的访问权限。每个组都在一个已被授予管理权限的预定义 Oracle Service Bus 安全角色中。

有关详细信息，请参阅 Oracle Fusion Middleware Developer's Guide for Oracle Service Bus 中的“Configuring Administrative Security”，网址为：http://www.oracle.com/pls/as111170/lookup?id=OSBDV1539。

角色

安全角色是一个标识，可根据运行时环境中的条件将该标识授予用户或组。创建访问控制策略时，可将访问权限授予角色、组或用户。

例如，可创建两个组，MyCustomersEast 和 MyCustomersWest。您可创建一个名为 PrivilegedCustomer 的安全角色和一些条件，以使 MyCustomersWest 组从 EST 的上午 8 点到下午 8 点都在此角色中，而 MyCustomersEast 组从 EST 的下午 8 点到上午 8 点在此角色中。然后，您可为代理服务创建一个访问控制策略，该策略可给予 PrivilegedCustmoer 角色访问服务的权限。不同用户在不同时间具有访问权限，具体取决于这些用户在 MyCustomersEast 组还是在 MyCustomersWest 组。

管理安全角色

Oracle Service Bus 提供四个可授予管理权限的预定义安全角色（加上来自 WebLogic Server 的四个预定义角色）。您不能更改 Oracle Service Bus 管理安全角色的访问权限，但可以更改用户或组成为一个角色的条件。

有关这些角色和每个角色可用的权限的详细信息，请参阅 Oracle Fusion Middleware Developer's Guide for Oracle Service Bus 中的“Configuring Administrative Security”，网址为：http://www.oracle.com/pls/as111170/lookup?id=OSBDV1539。

访问控制策略

访问控制策略指定了一些条件，在这些条件下用户、组或角色可访问某个代理服务。例如，可创建一个策略，该策略始终允许 GoldCustomer 角色中的用户访问代理服务，但仅允许 SilverCustomer 角色中的用户在工作日夜晚 12:00 PM 之后访问代理服务。

对于所有代理服务，可以创建传输级别策略，该策略可在客户机尝试建立与代理服务的连接时应用安全检查。仅允许传输级别策略中列出的用户发出的请求继续下去。

消息级访问控制策略在客户机尝试调用具有消息级安全的代理服务时应用安全检查。可以在下列情况下创建消息级访问控制策略：

  • 代理服务是活动的 Web 服务安全中间者
  • 代理服务具有消息级定制验证

仅允许消息级别策略中列出的用户调用该操作。

安全配置数据和会话

用户、组和角色持久保存在安全提供程序中，而安全提供程序不受 Oracle Service Bus 会话控制。因此，在会话内或会话外时都可以创建或修改此数据。对此数据进行的任何添加或修改都会立即生效并可用于所有会话。即使放弃了在其中添加或修改了数据的会话，安全数据也不会被放弃。

访问控制策略保留在授权提供程序中。现在存在对 Oracle Service Bus 仓库中的访问控制策略的引用。

在 Oracle Service Bus 设计会话内部（非外部）对访问控制策略进行管理。由于是在会话内进行更改，因此可以像其他资源一样提交或放弃更改。

尽管可以从 Oracle Service Bus 管理控制台管理 ACL，但也可以在 Oracle Service Bus 外部更改策略。但是，在 Oracle Service Bus 外部更改策略可能会使 Oracle Service Bus 内的引用过期和无效。

因此，为便于一致管理，要么在 Oracle Service Bus 外对 ACL 进行完全管理（使用授权提供程序 MBean 或第三方授权提供程序工具），要么从 Oracle Service Bus 会话内对 ACL 进行完全管理。对这两种方法的任何组合都可能导致策略视图不一致。
