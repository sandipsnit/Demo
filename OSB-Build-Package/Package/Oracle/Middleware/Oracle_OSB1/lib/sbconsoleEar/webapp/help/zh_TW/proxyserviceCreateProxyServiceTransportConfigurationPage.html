傳輸組態頁面

「建立/編輯代理主機服務 - 傳輸組態」頁面是一系列頁面中的其中一頁，供您建立和編輯代理主機服務 (如建立和設定代理主機服務和編輯代理主機服務組態中所述)。

使用此頁面可選取代理主機服務的傳輸通訊協定，以及設定其他一般傳輸組態設定。表 19-3 說明如何使用此頁面。

注意： 內送傳輸層級安全適用於從屬端應用程式和 Oracle Service Bus 代理主機服務。外送傳輸層級安全適用於 Oracle Service Bus 代理主機服務與業務服務之間的連線。若要進一步瞭解傳輸層級安全，請參閱 Oracle Fusion Middleware Developer's Guide for Oracle Service Bus 中的 "Configuring Transport-Level Security"，網址為：http://www.oracle.com/pls/as111170/lookup?id=OSBDV1557。

表 19-3 建立/編輯代理主機服務 - 傳輸組態頁面
選項
建立或編輯...
通訊協定
從清單中選取傳輸通訊協定。可用的通訊協定會根據所建立的服務類型而不同：
  • WSDL Web 服務：http、jca、jms、local、sb、ws
  • 傳輸類型：jejb
  • 訊息傳遞服務：email、file、ftp、http、jms、local、mq (如果可用)、sftp、tuxedo
  • 任何 SOAP 服務：http、jms、local、sb
  • 任何 XML 服務：email、file、ftp、http、jms、local、mq (如果可用)、sb、sftp、tuxedo
端點 URI
請根據您在上方「通訊協定」欄位中選取之傳輸通訊協定的格式，輸入端點 URI。格式如下：
  • email - mailfrom:mail-server-hostname:mail-server-port
  • file - file:///drivename:/somename
  • ftp - ftp://hostname:port/directory
  • http - someName

    • HTTP 傳輸支援 HTTP 與 HTTPS 端點。

  • jca - jca://<resource_adapter_jndi>
  • jejb - URL 格式是 ejb_jndi_name。設定為 JEJB 代理主機服務的 URI 會變成尋找無狀態階段作業 Bean (從屬端 JAR 中遠端/業務介面的 JEJB 傳輸所產生) 的全域 JNDI 名稱。

    • 注意：對於 EJB 3.0，ejb_jndi_name 是產生之 Bean 中 @javax.ejb.Stateless 註解的 mappedName 屬性。產生之 EJB 服務的查找 JNDI 名稱後面會接著 #interface_class，這是業務介面的完整名稱。

    您可以存取 JEJB 代理主機服務為：

    EJB 2.1: protocol://host:port/ejb_jndi_name

    EJB 3.0: protocol://host:port/ejb_jndi_name#interface_class

    注意：如果您的 EJB 在 IBM WebSphere 上執行，則 ejb_jndi_name 必須為下列格式：

    cell/nodes/node_name/servers/server_name/ejb_jndi_name

    或

    cell/clusters/cluster_name/ejb_jndi_name

    如 IBM WebSphere 文件 (http://publib.boulder.ibm.com/infocenter/wasinfo/v6r1/index.jsp?topic=/com.ibm.websphere.nd.iseries.doc/info/iseriesnd/ae/rnam_example_prop3.html) 所述。

  • jms - jms://host:port/factoryJndiName/destJndiName

    • 若要設定多個伺服器做為 JMS 目標，請使用下列 URI 格式：

    jms://host1:port,host2:port/QueueConnectionFactory/DestName

    請注意，建立代理主機服務時，即使無法使用 JMS 端點上的伺服器，仍然可以設定該端點的 URI。然而，如果是 JMS，則啟動階段作業時該端點必須可供使用。若要進一步瞭解，請參閱啟動階段作業。

    注意：雖然 Oracle WebLogic Server 允許在 JNDI 名稱中使用正斜線 (例如 myqueues/myqueue)，但是含有正斜線的 JNDI 名稱會干擾 Oracle Service Bus 所需的 URI 格式，因此您無法使用那些名稱。若要解決此問題，請定義一個 JMS 外部伺服器，並在 URI 中參照該外部伺服器。如需詳細資訊，請參閱 Oracle Fusion Middleware Oracle WebLogic Server Administration Console Online Help 中的 Configure foreign servers，網址為 http://www.oracle.com/pls/as111170/lookup?id=WLACH。

    在叢集中：JMS URI 的主機名稱必須與設定在 WebLogic Server 的叢集伺服器名稱完全一致。

  • local

    • 此傳輸不需要端點 URI。

  • mq - mq://<local-queue-name>?conn=<mq-connection-resource-ref>

    • local-queue-name 是 MQ 佇列的名稱，代理主機服務從這個佇列讀取訊息。

    mq-connection-resource-ref 是 MQ 連線資源的路徑 (project/folder) 和名稱。例如，default/my_MQconnection。

    注意： 「端點 URI」不可包含空格，因此建立 MQ 連線資源或專案/資料夾時，請勿在名稱中包含空格。

    若要在 Oracle Service Bus 中使用 MQ 傳輸，請參閱 MQ 連線。

端點 URI (續)
  • sb - service_name

    • service_name 是代理主機服務的唯一識別碼。此名稱預設會是代理主機服務名稱。

    service_name 只能包含 URI 中允許的字元 (如 http://www.ietf.org/rfc/rfc2396.txt 所述的 RFC2396)，但不得包含正斜線 (/) 或冒號 (:) 字元。

  • sftp - sftp://hostname:port/directory
  • tuxedo - exportname

    • URI exportname 對應至被遠端 Tuxedo 網域識別為 Tuxedo 服務的 WTC 匯出。

    如果指定多個 URI，則端點必須有唯一的資源名稱。如果未指定遠端名稱，則它的值就是資源名稱的值。如果未輸入任何遠端名稱，或遠端名稱與資源名稱相同，則只會允許一個 URI。在此情況下，資源名稱和遠端名稱的值會相同。這可讓使用已定義之 WTC 匯入的使用者利用 WTC 負載平衡和容錯移轉。

    注意：如果設定兩個相同的 URI，則會顯示錯誤，指出您已有該服務名稱。

  • ws - /contextPath

    • 如果是使用 HTTP 或 WS 傳輸的代理主機服務，則 contextPath 必須是唯一。

取得所有標頭
選取「是」，以擷取傳輸中的所有標頭。
選取「否」，以擷取一組定義的標頭。如果選取「否」，請在「標頭」欄位中輸入一組標頭，然後按一下「新增」 (此步驟不適用於「本機」傳輸)。
注意： Oracle Service Bus 不會將要求中的「HTTP 授權」標頭傳遞給管線，因為這樣會開啟安全弱點。您可能會不小心建立記錄動作，而將使用者名稱和未加密的密碼寫入至記錄檔。如果您的設計模式需要「HTTP 授權」標頭位於管線中，則請執行下列操作：

a.在 Oracle Service Bus 的啟動命令中，將下列系統特性設定為 true：com.bea.wli.sb.transports.http.GetHttpAuthorizationHeaderAllowed

b.在「Oracle Service Bus 管理主控台」的「傳輸組態」頁面中選取取得所有標頭，或選取使用者指定的標頭，並指定「授權」。
c.重新啟動 Oracle Service Bus。

Oracle Service Bus 會將「授權」標頭傳遞給管線。

完成作業後

按「下一頁」繼續在下一個頁面上設定此服務，或按一下「最後一頁」在建立/編輯代理主機服務 - 摘要頁面上檢查並儲存這項組態。
